The `innochecksum` tool in `extra/innochecksum.cc` has two problems that need fixing.

The first issue is that `is_page_doublewritebuffer()` uses a hardcoded page number range to decide if a page is part of the doublewrite buffer. This is wrong because the actual doublewrite buffer location depends on what's recorded in the TRX_SYS page of the system tablespace, not a fixed offset. The function should be refactored to use the real doublewrite block locations read from the TRX_SYS page, and it should only apply to the system tablespace (space 0). It also no longer needs the page buffer as a parameter since it can work from stored state. When `skip_freed_pages` is enabled, `parse_page()` should skip doublewrite buffer pages entirely, and the main loop should also avoid rewriting checksums for doublewrite buffer pages.

The second issue is that innochecksum crashes on non-first files of a multi-file system tablespace (like ibdata2 or ibdata3) because it cannot determine the page size when page 0 is not the tablespace's first page. Add a `--tablespace-flags` command line option that lets users pass InnoDB tablespace flags directly, so the tool can initialize the correct page size without relying on reading flags from page 0. Invalid flag values should be rejected. When no flags are provided and the file doesn't start at page 0, the tool should report a helpful error suggesting the `--tablespace-flags` option. With `--tablespace-flags` provided, the tool should be able to process any file in the tablespace regardless of whether it starts at page 0. If the file starts at page 0 and the provided flags differ from the stored flags, the tool should warn about the mismatch. The page 0 reading and initialization logic currently inline in `main()` should be pulled into its own function to handle both cases cleanly.

The `init_page_size()` function currently reads flags from the page buffer internally. Refactor it to accept the flags as a parameter instead.

For innochecksum to access the doublewrite buffer constants from `trx0sys.h`, the `UNIV_INNOCHECKSUM` preprocessor guards in `storage/innobase/include/trx0sys.h`, `storage/innobase/include/fsp0types.h`, and `storage/innobase/include/trx0undo.h` need to be adjusted so the doublewrite definitions remain visible while the full transaction system types are excluded. Also, `parse_page()` should be made static.
