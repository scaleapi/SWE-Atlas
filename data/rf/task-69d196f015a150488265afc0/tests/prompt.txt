The redo log LSN allocation in InnoDB is a bottleneck in write-heavy workloads. The core data structures live in storage/innobase/include/log0log.h (the log_t struct) with the main implementation in storage/innobase/log/log0log.cc and the mini-transaction commit path in storage/innobase/mtr/mtr0mtr.cc.

When multiple mini-transactions commit concurrently, the log buffer state is tracked through several separate fields that all need to stay consistent, leading to an overly complex locking mechanism with platform-specific assembly code. The LSN allocation path performs too many atomic read-modify-write operations for what should be a fast, concurrent code path. On top of that, many callers across the codebase fetch the exact current LSN even though they only need the durable (flushed) LSN or a rough estimate, adding unnecessary synchronization overhead. The innodb_log_spin_wait_delay system variable and its associated spin loop infrastructure exist only to service this locking complexity and would be unnecessary with a simpler design.
