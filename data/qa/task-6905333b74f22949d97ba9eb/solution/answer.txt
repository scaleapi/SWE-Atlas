The keyboard protocol stack in kitty maintains completely independent stacks for the main and alternate screen buffers. Each buffer has its own 8-entry array of keyboard encoding flags, and switching between buffers simply changes which array is active without copying or modifying any flag values.

## Data Structure Architecture

The stack architecture is defined in screen.h at line 128:

```
$ grep -n "key_encoding_flags" kitty/screen.h | head -1
128:    uint8_t main_key_encoding_flags[8], alt_key_encoding_flags[8], *key_encoding_flags;
```

This declares:
- `main_key_encoding_flags[8]`: 8-entry stack for the main screen buffer
- `alt_key_encoding_flags[8]`: 8-entry stack for the alternate screen buffer  
- `*key_encoding_flags`: pointer that references whichever array is currently active

Each array element is `uint8_t`, where the high bit (0x80) marks active entries and the lower 7 bits (0x7f) store the actual flag values (0-31).

## Buffer Switching Behavior

When toggling between screen buffers, the terminal only changes the pointer—it never copies or modifies the flag arrays. This is implemented in `screen_toggle_screen_buffer` starting at line 1067 in screen.c:

```
$ sed -n '1067,1095p' kitty/screen.c
void
screen_toggle_screen_buffer(Screen *self, bool save_cursor, bool clear_alt_screen) {
    bool to_alt = self->linebuf == self->main_linebuf;
    self->active_hyperlink_id = 0;
    if (to_alt) {
        if (clear_alt_screen) {
            linebuf_clear(self->alt_linebuf, BLANK_CHAR);
            grman_clear(self->alt_grman, true, self->cell_size);
        }
        if (save_cursor) screen_save_cursor(self);
        self->linebuf = self->alt_linebuf;
        self->tabstops = self->alt_tabstops;
        self->key_encoding_flags = self->alt_key_encoding_flags;
        self->grman = self->alt_grman;
        screen_cursor_position(self, 1, 1);
        cursor_reset(self->cursor);
    } else {
        self->linebuf = self->main_linebuf;
        self->tabstops = self->main_tabstops;
        self->key_encoding_flags = self->main_key_encoding_flags;
        if (save_cursor) screen_restore_cursor(self);
        self->grman = self->main_grman;
    }
    screen_history_scroll(self, SCROLL_FULL, false);
    self->is_dirty = true;
    self->grman->layers_dirty = true;
    clear_selection(&self->selections);
    global_state.check_for_active_animated_images = true;
}
```

The key lines are the pointer assignments:
- Line 1079: `self->key_encoding_flags = self->alt_key_encoding_flags;` (switching to alternate)
- Line 1086: `self->key_encoding_flags = self->main_key_encoding_flags;` (switching to main)

This proves that flags are NEVER copied between buffers—the stacks are completely isolated.

## Stack Query Implementation

The function `screen_current_key_encoding_flags` (lines 1203-1209) retrieves the current active flags by scanning backward through the array for the first entry with the 0x80 marker bit set:

```
$ sed -n '1203,1209p' kitty/screen.c
uint8_t
screen_current_key_encoding_flags(Screen *self) {
    for (unsigned i = arraysz(self->main_key_encoding_flags); i-- > 0; ) {
        if (self->key_encoding_flags[i] & 0x80) return self->key_encoding_flags[i] & 0x7f;
    }
    return 0;
}
```

If no entries have the marker bit set, the function returns 0 (no flags active).

## Stack Push and Exhaustion Behavior

The push operation is implemented in `screen_push_key_encoding_flags` (lines 1233-1245):

```
$ sed -n '1233,1245p' kitty/screen.c
void
screen_push_key_encoding_flags(Screen *self, uint32_t val) {
    uint8_t q = val & 0x7f;
    const unsigned sz = arraysz(self->main_key_encoding_flags);
    unsigned current_idx = 0;
    for (unsigned i = arraysz(self->main_key_encoding_flags); i-- > 0; ) {
        if (self->key_encoding_flags[i] & 0x80) { current_idx = i; break; }
    }
    if (current_idx == sz - 1) memmove(self->key_encoding_flags, self->key_encoding_flags + 1, (sz - 1) * sizeof(self->main_key_encoding_flags[0]));
    else self->key_encoding_flags[current_idx++] |= 0x80;
    self->key_encoding_flags[current_idx] = 0x80 | q;
    debug_input("\x1b[35mPushed key encoding flags to: %u\x1b[39m\n", screen_current_key_encoding_flags(self));
}
```

**Stack exhaustion behavior:**
- Maximum stack depth: 8 entries (indices 0-7)
- When the stack is full (`current_idx == sz - 1`, i.e., index 7), the `memmove` shifts all entries left by one position
- This silently discards the oldest entry at index 0
- The new value is placed at the end (index 7)
- No error is raised—the operation always succeeds

**Critical point:** Stack exhaustion on one buffer has absolutely NO effect on the other buffer's stack since they use completely separate memory arrays.

## Progressive Enhancement Flags

The keyboard protocol uses bit flags to control encoding behavior. According to the protocol documentation (docs/keyboard-protocol.rst lines 276-282):

| Bit | Value | Name | Description |
|-----|-------|------|-------------|
| 0b1 | 1 | disambiguate | Use CSI u encoding for ambiguous keys |
| 0b10 | 2 | report_events | Report key repeat and release events |
| 0b100 | 4 | report_alternates | Report shifted/alternate key values |
| 0b1000 | 8 | report_all_keys | Report all keys as escape codes (including text-producing keys) |
| 0b10000 | 16 | report_text | Embed associated text in escape codes |

These are applied in key_encoding.c at lines 414-424:

```
$ sed -n '414,424p' kitty/key_encoding.c
encode_glfw_key_event(const GLFWkeyevent *e, const bool cursor_key_mode, const unsigned key_encoding_flags, char *output) {
    KeyEvent ev = {
        .key = e->key, .shifted_key = e->shifted_key, .alternate_key = e->alternate_key,
        .text = e->text,
        .cursor_key_mode = cursor_key_mode,
        .disambiguate = key_encoding_flags & 1,
        .report_all_event_types = key_encoding_flags & 2,
        .report_alternate_key = key_encoding_flags & 4,
        .report_text = key_encoding_flags & 8,
        .embed_text = key_encoding_flags & 16
    };
```

Note: The C code uses slightly different field names than the protocol documentation (e.g., `report_text` in code vs `report_all_keys` in docs for bit 8), but the behavior matches the specification.

## Analysis of Results

**Escape sequence format for ctrl+shift+a:**
- Format: `ESC [ keycode ; modifiers u`
- Keycode: 97 (ASCII for 'a')
- Modifier bits: shift=1, alt=2, ctrl=4. For ctrl+shift: (1 + 4) + 1 = 6
- Result: `\x1b[97;6u` = bytes `1b 5b 39 37 3b 36 75`

**Key observations proving stack independence:**

1. **Scenario 1→2**: Pushing disambiguate (0b1) on main doesn't change ctrl+shift+a output (already uses CSI u), but would affect ambiguous keys like Escape.

2. **Scenario 2→3**: Switching to alternate buffer and pushing report-all-keys (0b1000) causes plain 'a' to generate `\x1b[97u` instead of just `a`. This proves the alternate buffer has its own independent flags.

3. **Scenario 3→4**: Returning to main buffer restores the previous flags (0b1). Plain 'a' would again output just `a` (not shown, but confirmed by the encoding). The main buffer's stack was completely unaffected by operations on the alternate buffer.

**Modifier key behavior:**
- Without report-all-keys: modifier keys like Shift generate empty output (not reported)
- With report-all-keys (0b1000): Left Shift generates `\x1b[57441u` (57441 is the Private Use Area codepoint for this key)

**Special keys behavior:**
- Enter: `\r` (legacy) → `\x1b[13u` (with report-all-keys)
- Escape: `\x1b` (legacy, ambiguous!) → `\x1b[27u` (with disambiguate)

## Simulated Buffer Switching Scenario

Here's the exact sequence requested, demonstrating the round-trip behavior:

```
Step 1: Start on MAIN buffer, empty stack
        - current_key_encoding_flags() returns 0
        - ctrl+shift+a produces: \x1b[97;6u
        - plain a produces: a

Step 2: Push flags=1 (disambiguate) on MAIN
        - main_key_encoding_flags[0] = 0x81
        - current_key_encoding_flags() returns 1
        - ctrl+shift+a produces: \x1b[97;6u
        - Escape now produces: \x1b[27u (disambiguated)

Step 3: Switch to ALTERNATE buffer
        - key_encoding_flags pointer changes to alt_key_encoding_flags
        - alt_key_encoding_flags is all zeros (empty stack)
        - current_key_encoding_flags() returns 0
        - main_key_encoding_flags[0] still equals 0x81 (UNCHANGED)

Step 4: Push flags=8 (report-all-keys) on ALTERNATE  
        - alt_key_encoding_flags[0] = 0x88
        - current_key_encoding_flags() returns 8
        - ctrl+shift+a produces: \x1b[97;6u
        - plain a produces: \x1b[97u
        - Left Shift produces: \x1b[57441u

Step 5: Switch back to MAIN buffer
        - key_encoding_flags pointer changes to main_key_encoding_flags
        - current_key_encoding_flags() returns 1 (PRESERVED from Step 2!)
        - alt_key_encoding_flags[0] still equals 0x88 (UNCHANGED)
        - ctrl+shift+a produces: \x1b[97;6u
        - plain a produces: a (not \x1b[97u because flags=1, not 8)
```

**This proves:**
1. Each buffer maintains its own independent stack
2. Flags are NOT copied when switching buffers
3. The stack pushed on main survives the round-trip completely intact
4. Operations on one buffer never affect the other buffer's stack

## Screen Reset Behavior

A full screen reset clears BOTH arrays (lines 173-174 in screen.c):

```
$ sed -n '173,174p' kitty/screen.c
    memset(self->main_key_encoding_flags, 0, sizeof(self->main_key_encoding_flags));
    memset(self->alt_key_encoding_flags, 0, sizeof(self->alt_key_encoding_flags));
```

This is the ONLY operation that affects both stacks simultaneously.

## Edge Cases and Potential Issues

**State leakage scenarios (none found):**
- Rapidly switching buffers while manipulating keyboard mode: No leakage. Each switch simply alternates which array the pointer references.
- Pushing/popping across buffer switches: Each operation only affects the currently active buffer's array.

**Conditions where isolation could APPEAR to break (application-side issues):**

1. **Application doesn't track buffer state**: If an application pushes flags, switches buffers, and expects those flags to still be active, it will see unexpected behavior. This is by design—applications using alternate screen (vim, less, TUI apps) should manage their own keyboard mode.

2. **Application queries flags after switch**: The query returns the current buffer's flags, not the previous buffer's. An unaware application might misinterpret this.

3. **Crash without cleanup**: If an application crashes while on alternate screen with custom flags, switching back to main reveals the main screen's original flags (good) but the alternate screen retains the crashed app's flags until reset.

**Terminal-side isolation is perfect**—there is no mechanism by which operations on one buffer's stack can affect the other buffer's stack, other than a full screen reset.
