The SQL shows `ORDER BY "documents_document"."created" DESC` with `LIMIT` and `OFFSET` but no secondary sort key. When multiple rows have identical `created` values, the database makes no guarantee about their relative ordering. This is fundamental SQL behavior: ORDER BY on non-unique values produces nondeterministic results for tied rows.

The instability is amplified when tag filters are applied. I examined the `TagsFilter` class in `src/documents/filters.py`:

```
grep -A 25 "class TagsFilter" /app/src/documents/filters.py
```

Output:
```
class TagsFilter(Filter):
    def __init__(self, exclude=False, in_list=False):
        super(TagsFilter, self).__init__()
        self.exclude = exclude
        self.in_list = in_list

    def filter(self, qs, value):
        if not value:
            return qs

        try:
            tag_ids = [int(x) for x in value.split(",")]
        except ValueError:
            return qs

        if self.in_list:
            qs = qs.filter(tags__id__in=tag_ids).distinct()
        else:
            for tag_id in tag_ids:
                if self.exclude:
                    qs = qs.exclude(tags__id=tag_id)
                else:
                    qs = qs.filter(tags__id=tag_id)

        return qs
```

The `in_list=True` path uses `qs.filter(tags__id__in=tag_ids).distinct()` which creates a single JOIN. But the `else` branch (used for `tags__id__all` semantics) calls `filter(tags__id=tag_id)` in a loop, creating multiple JOINs:


The multiple JOINs to the many-to-many junction table create row multiplication before DISTINCT collapses them. The order in which the database processes and eliminates these duplicates is implementation-dependent, adding another source of nondeterminism.

To directly answer the prompt's questions about duplicates and pagination order:

Is the backend producing duplicates that get collapsed somewhere later? No. The backend is not producing duplicate document records. The DISTINCT clause removes duplicate rows that arise from JOIN operations (when a document has multiple matching tags), but this is not the source of the pagination instability. The "duplicates" users see across pages are the same document appearing in different positions due to nondeterministic ordering, not actual duplicate records being collapsed.

Is pagination happening before any de-duplication? No. In SQL execution order, DISTINCT is applied before LIMIT/OFFSET. The query first builds the joined result set, applies WHERE filters, projects columns, removes duplicates with DISTINCT, sorts with ORDER BY, and finally applies LIMIT/OFFSET for pagination. The instability occurs because ORDER BY on a non-unique column doesn't guarantee stable row positions, so the OFFSET window can capture different rows between queries even though de-duplication has already occurred.

Is the ordering quietly unstable when multiple rows tie on the primary sort key? Yes. This is the root cause. When multiple documents share the same `created` timestamp, the database makes no guarantee about their relative ordering, causing rows to shift positions between queries.

The observation that the glitch seems to depend on what the user is allowed to see is a consequence of the same underlying issue. Permission filtering reduces the result set to documents the user can access, but within that filtered subset, the ORDER BY instability persists. Different users encounter different boundary conditions at page edges because they see different collections of documents, each with its own pattern of tied timestamps. The permission system does not introduce or amplify the instability; it merely shapes which documents fall into the unstable ordering window for each user.

The exact condition that destabilizes the list is: OFFSET-based pagination applied to a queryset ordered by a non-unique field (`created`) without a secondary sort key, when multiple records share the same value in the ORDER BY column. The instability manifests as:

1. Documents appearing on multiple pages when they shift forward between requests
2. Documents disappearing from results when they shift backward between requests  
3. Apparent gaps where expected documents do not appear on any page during a browsing session

The fix would be to add a secondary sort key (like `id`) to guarantee deterministic ordering: `ordering = ("-created", "-id")`. This ensures that even when `created` timestamps tie, the `id` field provides a stable tiebreaker.
