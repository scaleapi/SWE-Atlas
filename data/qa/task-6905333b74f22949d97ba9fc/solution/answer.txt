I began investigating how kitty moves data between its core C structures and Python kittens by examining the handler architecture and running verification scripts to observe runtime behavior.

Data transfer between kitty's core C structures and Python kittens involves crossing the boundary between native code and the Python runtime. The kittens system uses a handler architecture defined in kittens/tui/handler.py where Python code receives events through callback methods. The Handler class serves as the base for all kitten implementations and manages the lifecycle of data transfer.

The handler initialization establishes the connection between the terminal core and Python. The _initialize method receives references to screen size, terminal manager, and scheduling functions that bridge the gap between C and Python.

```python
# From kittens/tui/handler.py
def _initialize(
    self,
    screen_size: ScreenSize,
    term_manager: TermManagerType,
    schedule_write: Callable[[bytes], None],
    tui_loop: LoopType,
    debug: Debug,
    image_manager: Optional[ImageManagerType] = None
) -> None:
    from .operations import commander
    self.screen_size = screen_size
    self._term_manager = term_manager
    self._tui_loop = tui_loop
    self._schedule_write = schedule_write
    self.debug = debug
    self.cmd = commander(self)
    self._image_manager = image_manager
```

The mouse event handling includes timestamp tracking for click detection. The ButtonEvent named tuple captures both the mouse event and timing information needed to distinguish clicks from other actions:

```python
# From kittens/tui/handler.py
class ButtonEvent(NamedTuple):
    mouse_event: MouseEvent
    timestamp: float

def is_click(a: ButtonEvent, b: ButtonEvent) -> bool:
    from .loop import EventType
    if a.mouse_event.type is not EventType.PRESS or b.mouse_event.type is not EventType.RELEASE:
        return False
    x = a.mouse_event.cell_x - b.mouse_event.cell_x
    y = a.mouse_event.cell_y - b.mouse_event.cell_y
    return x*x + y*y <= 4
```

The do_parse function in kitty/child-monitor.c shows how data flows from the IO thread to Python callbacks. The screen's dump_callback receives parsing events when tracing is enabled, demonstrating the event path:

```c
// From kitty/child-monitor.c lines 438-448
static bool
do_parse(ChildMonitor *self, Screen *screen, monotonic_t now, bool flush) {
    ParseData pd = {.dump_callback = self->dump_callback, .now = now};
    self->parse_func(screen, &pd, flush);
    if (pd.input_read) {
        if (pd.write_space_created) wakeup_io_loop(self, false);
        if (screen->paused_rendering.expires_at) {
            set_maximum_wait(MAX(0, screen->paused_rendering.expires_at - now));
        } else set_maximum_wait(OPT(input_delay) - pd.time_since_new_input);
    }
    return pd.input_read;
}
```

The message passing between threads in kitty/child-monitor.c shows the pattern for safe concurrent access. Messages are queued under mutex protection and processed in the main thread without holding locks:

```c
// From kitty/child-monitor.c
talk_mutex(lock);
if (UNLIKELY(self->messages_count)) {
    msgs = malloc(sizeof(Message) * self->messages_count);
    if (msgs) {
        memcpy(msgs, self->messages, sizeof(Message) * self->messages_count);
        msgs_count = self->messages_count;
    }
    memset(self->messages, 0, sizeof(Message) * self->messages_capacity);
    self->messages_count = 0;
}
talk_mutex(unlock);
```

Event delivery follows a pull model where Python callbacks are invoked from the main loop rather than being triggered directly from IO threads. This avoids complex locking scenarios but introduces latency bounded by the poll timeout.

The handler's atomic_update decorator wraps methods that need to batch terminal operations, preventing partial updates from being visible:

```python
# From kittens/tui/handler.py
@classmethod
def atomic_update(cls, func: DecoratedFunc) -> DecoratedFunc:
    from functools import wraps
    @wraps(func)
    def f(*a: Any, **kw: Any) -> Any:
        with pending_update(a[0].write):
            return func(*a, **kw)
    return cast(DecoratedFunc, f)
```

Concurrency and object ownership behave predictably under normal operation because the architecture separates concerns between threads and uses appropriate synchronization. Subtle races can emerge only when combining multiple concurrent operations that each appear correct in isolation.
