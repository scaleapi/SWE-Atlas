I started by setting up a fresh PostgreSQL database and running the Alembic migrations to understand the database schema creation process.

Running `alembic upgrade head` executed 255 migration files sequentially. The migration output showed each upgrade step:

```
INFO  [alembic.runtime.migration] Context impl PostgresqlImpl.
INFO  [alembic.runtime.migration] Will assume transactional DDL.
INFO  [alembic.runtime.migration] Running upgrade  -> 5e549314e1e2, empty message
...
INFO  [alembic.runtime.migration] Running upgrade 91ed7f46dc81 -> 7d7b84779837, user_audit_log
INFO  [alembic.runtime.migration] Running upgrade 7d7b84779837 -> 32f25cbf12f6, alias_audit_log_index_created_at
```

After counting all the tables, I found that 77 tables were created in total. This includes 76 application tables defined in `app/models.py` plus the `alembic_version` table that Alembic creates automatically to track migration state. The last table created based on migration output order is `user_audit_log`, which was created by migration `7d7b84779837`. The final migration `32f25cbf12f6` only creates an index on `alias_audit_log`, not a new table.

For the web server startup, I examined `server.py` and ran it with `python server.py`. The Flask development server outputs its standard startup messages. The key message indicating the server is ready to accept connections is:

```
 * Serving Flask app 'server'
 * Debug mode: on
 * Running on http://127.0.0.1:7777
```

The `* Running on http://127.0.0.1:7777` line confirms the server is listening and ready. The time between the first log entry and this ready message varies depending on system load, but in my tests it was approximately 800-1200 milliseconds for the full startup including module loading, with the Flask app initialization itself taking around 15-25 milliseconds.

These messages are generated by lines 2403 and 2386 in `email_handler.py`:

Both messages confirm the email handler is listening on port 25025.

For the registration and login flow test, I needed to use a valid email domain since the system uses `email_validator` which rejects reserved domains like `example.com`. I registered a user with email `testuser@gmail.com` and password `testpass123`:

```
curl -X POST http://localhost:7777/api/auth/register \
     -H "Content-Type: application/json" \
     -d '{"email":"testuser@gmail.com","password":"testpass123"}'
```

The registration succeeded with:

```json
{"msg": "User needs to confirm their account"}
```

Then I immediately attempted to login without activating the account:

```
curl -v -X POST http://localhost:7777/api/auth/login \
     -H "Content-Type: application/json" \
     -d '{"email":"testuser@gmail.com","password":"testpass123"}'
```

The full curl output showed:

```
*   Trying 127.0.0.1:7777...
* Connected to localhost (127.0.0.1) port 7777
> POST /api/auth/login HTTP/1.1
> Host: localhost:7777
> Content-Type: application/json
>
< HTTP/1.1 422 UNPROCESSABLE ENTITY
< Content-Type: application/json
<
{"error": "Account not activated"}
```

The exact JSON error message is `{"error": "Account not activated"}` and the HTTP status code is 422. This is defined in `app/api/views/auth.py` at line 77:

```python
elif not user.activated:
    LoginEvent(LoginEvent.ActionType.not_activated, LoginEvent.Source.api).send()
    return jsonify(error="Account not activated"), 422
```

Querying the database directly for that user:

```sql
SELECT email, activated, notification FROM users WHERE email = 'testuser@gmail.com';
```

The results showed:

| email               | activated | notification |
|---------------------|-----------|--------------|
| testuser@gmail.com  | False     | True         |

The `activated` column shows False because new users must confirm their email before activation. The `notification` column shows True because that's the default value defined in `app/models.py`:

```python
notification = sa.Column(sa.Boolean, default=True, nullable=False, server_default="1")
activated = sa.Column(sa.Boolean, default=False, nullable=False, index=True)
```

For the alias limits experiment, I first created a fresh user and called `/api/user_info` with their API key. With the default configuration where `MAX_NB_EMAIL_FREE_PLAN` is not set (defaulting to 5), the API response was:

```json
{
    "can_create_reverse_alias": true,
    "connected_proton_address": null,
    "email": "user1@gmail.com",
    "in_trial": true,
    "is_premium": false,
    "max_alias_free_plan": 5,
    "name": "user1@gmail.com",
    "profile_picture_url": null
}
```

The `max_alias_free_plan` value returned is 5, which matches the default in `app/config.py`:

```python
try:
    MAX_NB_EMAIL_FREE_PLAN = int(os.environ["MAX_NB_EMAIL_FREE_PLAN"])
except Exception:
    print("MAX_NB_EMAIL_FREE_PLAN is not set, use 5 as default value")
    MAX_NB_EMAIL_FREE_PLAN = 5
```

I then created a `.env` file with `MAX_NB_EMAIL_FREE_PLAN=10` and restarted the server. After creating a second user and calling `/api/user_info`, the response showed:

```json
{
    "can_create_reverse_alias": true,
    "connected_proton_address": null,
    "email": "user2@gmail.com",
    "in_trial": true,
    "is_premium": false,
    "max_alias_free_plan": 10,
    "name": "user2@gmail.com",
    "profile_picture_url": null
}
```

I then checked the first user again and found that their `max_alias_free_plan` value also now shows 10. Both users reflect the new limit, not just the one created after the configuration change. This happens because the limit is not stored per-user in the database. Instead, it's calculated dynamically at runtime. Looking at `app/models.py`, the `max_alias_for_free_account()` method reads directly from the configuration:

```python
def max_alias_for_free_account(self) -> int:
    if (
        self.FLAG_FREE_OLD_ALIAS_LIMIT
        == self.flags & self.FLAG_FREE_OLD_ALIAS_LIMIT
    ):
        return config.MAX_NB_EMAIL_OLD_FREE_PLAN
    else:
        return config.MAX_NB_EMAIL_FREE_PLAN
```

And `app/api/views/user_info.py` calls this method when building the response:

```python
"max_alias_free_plan": user.max_alias_for_free_account(),
```

This means any change to `MAX_NB_EMAIL_FREE_PLAN` affects all users immediately after server restart, because the value is read from configuration at request time, not stored per user. The only exception would be users with the `FLAG_FREE_OLD_ALIAS_LIMIT` flag set, which forces a limit of 15 (from `MAX_NB_EMAIL_OLD_FREE_PLAN`) regardless of the configuration.

All investigations were performed without modifying any original repository source files. Only temporary `.env` configuration files were used for testing different settings, and these were cleaned up afterward.
