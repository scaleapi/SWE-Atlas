SFTPGo Server Startup Observation - Reference Answer

I started by building the SFTPGo binary and setting up a test environment to observe the actual runtime behavior. My goal was to capture what really happens when the server starts, how it handles missing components, and what appears in the logs during various scenarios.

Observing Startup Without a Configuration File

When I ran sftpgo serve without providing a configuration file, the first thing that appeared in the logs was the service announcement:

```json
{"level":"info","sender":"service","message":"starting SFTPGo 0.9.5-dev, config dir: ., config file: sftpgo, log max size: 10 log max backups: 5 log max age: 28 log verbose: true, log compress: false"}
```

The server then searched for a configuration file in the following paths: the current working directory, $HOME/.config/sftpgo on Linux, and /etc/sftpgo. When no config file was found, it logged a warning that revealed all the default configuration values it would use:

```json
{"level":"warn","sender":"config","message":"error loading configuration file: Unsupported Config Type \"\". Default configuration will be used: {SFTPD:{Banner:SFTPGo_0.9.5-dev BindPort:2022 BindAddress: IdleTimeout:15 MaxAuthTries:0 Umask:0022 UploadMode:0 ...} ProviderConf:{Driver:sqlite Name:sftpgo.db ...} HTTPDConfig:{BindPort:8080 BindAddress:127.0.0.1 ...}}"}
```

From this output, I extracted the key default values that the server applies at runtime:

- SFTP server binds to port 2022 on all interfaces (empty BindAddress means [::])
- HTTP server binds to port 8080 on localhost only (127.0.0.1)
- Idle timeout is set to 15 minutes
- MaxAuthTries is 0, which according to the code comments means it defaults to SSH's standard of 6 attempts
- Umask is set to 0022
- The data provider defaults to SQLite with database file sftpgo.db
- Enabled SSH commands are [md5sum, sha1sum, cd, pwd]
- The banner is SFTPGo_0.9.5-dev

Database Missing Behavior

When the SQLite database file doesn't exist, the server logs a warning followed by an error and then terminates:

```json
{"level":"warn","sender":"sqlite","message":"sqlite database file does not exists, please be sure to create and initialize a database before starting sftpgo"}
{"level":"error","sender":"service","message":"error initializing data provider: stat sftpgo.db: no such file or directory"}
```

The server cannot proceed without a valid data provider - it exits immediately at this point. This is a hard requirement, not something it can recover from.

Successful Startup Sequence

After creating the database using the SQL schema files from /app/sql/sqlite/, I observed the complete startup sequence. With a valid database, the logs showed:

```json
{"level":"debug","sender":"sqlite","message":"sqlite database handle created, connection string: \"file:sftpgo.db?cache=shared\""}
```

The HTTP server initialization appeared next:

```json
{"level":"debug","sender":"httpd","message":"initializing HTTP server with config {BindPort:8080 BindAddress:127.0.0.1 TemplatesPath:templates StaticFilesPath:static BackupsPath:backups}"}
```

Then the SFTP server initialization began:

```json
{"level":"debug","sender":"service","message":"initializing SFTP server with config {Banner:SFTPGo_0.9.5-dev BindPort:2022 BindAddress: IdleTimeout:15 MaxAuthTries:0 Umask:0022 ...}"}
{"level":"debug","sender":"utils","message":"set umask to 0022 (18)"}
```

Since no host keys were configured and the default id_rsa file didn't exist, the server auto-generated a new RSA private key:

```json
{"level":"info","sender":"sftpd","message":"No host keys configured and \"id_rsa\" does not exist; creating new private key for server"}
{"level":"info","sender":"sftpd","message":"Loading private key: id_rsa"}
```

Finally, the SFTP server registered its listener, signaling it was ready to accept connections:

```json
{"level":"info","sender":"sftpd","message":"server listener registered address: [::]:2022"}
```

At this point, both servers were running: SFTP on port 2022 (all interfaces) and HTTP on port 8080 (localhost only).

## Authentication Failure with Non-Existent User

To observe what happens when an SFTP client tries to connect with a username that doesn't exist, I attempted to connect using SSH with a fake username. The server logs revealed:

```json
{"level":"debug","sender":"connection_failed","client_ip":"127.0.0.1","username":"nonexistent_user","login_type":"password","error":"Credentials cannot be null or empty"}
```

Followed by:

```json
{"level":"warn","sender":"sftpd","message":"failed to accept an incoming connection: [ssh: no auth passed yet, Authentication error: could not validate password credentials: Credentials cannot be null or empty]"}
```

The connection_failed sender is specifically designed for Fail2ban integration - it includes the client IP, username, login type, and error message in a structured format that can be easily parsed by security tools.

The error message "Credentials cannot be null or empty" occurs because the SSH client's initial authentication attempt included an empty password field. The data provider validates the password before looking up the user - if the password is empty, it returns this error immediately. For a non-existent user with a non-empty password, you would see a "record not found" error instead (e.g., "Not found: sql: no rows in result set").

Web Admin Interface Behavior

Testing the HTTP endpoints revealed the redirect behavior:

The root endpoint (/) returns:
```
HTTP/1.1 301 Moved Permanently
Location: /web/users
```

The /web endpoint also returns:
```
HTTP/1.1 301 Moved Permanently
Location: /web/users
```

Both endpoints redirect with HTTP 301 (Moved Permanently) to `/web/users`, which is the users management page.

The API endpoints I tested:
- /api/v1/version returns version information: {"version":"0.9.5-dev","build_date":"","commit_hash":""}
- /api/v1/providerstatus returns {"error":"","message":"Alive","status":200} when the database is healthy
- /api/v1/user returns [] with an empty database
- /api/v1/connection returns [] when there are no active SFTP connections

Startup Flow Summary

The complete startup flow follows this sequence:

```
main.go → cmd.Execute() → service.Start()
                              ↓
                    logger.InitLogger()
                              ↓
                    config.LoadConfig() → [searches paths, logs defaults if not found]
                              ↓
                    dataprovider.Initialize() → [FATAL if db missing]
                              ↓
           ┌─────────────────┴─────────────────┐
           ↓                                   ↓
    sftpdConf.Initialize()              httpdConf.Initialize()
    [goroutine]                         [goroutine, if BindPort > 0]
           ↓                                   ↓
    checkHostKeys()                     loadTemplates()
    [auto-generate if needed]           initializeRouter()
           ↓                                   ↓
    net.Listen(":2022")                 http.ListenAndServe(":8080")
           ↓                                   ↓
    "server listener registered"        [ready for requests]
```

The SFTP and HTTP servers start in separate goroutines, running in parallel. The data provider initialization is synchronous and must complete successfully before either server can start.

All temporary files created during this observation were cleaned up, and the repository remains unchanged.
