How to tell the difference after processing finishes: with skip mode, an archive PDF is still created (has_archive_version is true). With skip_noarchive mode and existing text, no archive is created (has_archive_version is false) and the log shows "Document has text, skipping OCRmyPDF entirely."

To compare final API responses for both cases, I examined the metadata endpoint:

```
cd /app

```

Output:

```
    def metadata(self, request, pk=None):
        try:
            doc = Document.objects.get(pk=pk)
        except Document.DoesNotExist:
            raise Http404()

        meta = {
            "original_checksum": doc.checksum,
            "original_size": self.get_filesize(doc.source_path),
            "original_mime_type": doc.mime_type,
            "media_filename": doc.filename,
            "has_archive_version": doc.has_archive_version,
            "original_metadata": self.get_metadata(doc.source_path, doc.mime_type),
            "archive_checksum": doc.archive_checksum,
            "archive_media_filename": doc.archive_filename,
        }

        if doc.has_archive_version:
            meta["archive_size"] = self.get_filesize(doc.archive_path)
            meta["archive_metadata"] = self.get_metadata(
                doc.archive_path,
                "application/pdf",
            )
        else:
            meta["archive_size"] = None
            meta["archive_metadata"] = None

        return Response(meta)
```

The fields that show OCR-generated text versus existing text: original_mime_type indicates the source type (image/* means all content came from OCR). has_archive_version indicates whether OCRmyPDF created an archive PDF. archive_metadata contains CreatorTool showing ocrmypdf and Tesseract versions when OCR was applied. For an image upload, original_mime_type will be image/png or similar, has_archive_version will be true, and archive_metadata will contain OCR tool information. For a PDF with existing text processed with skip_noarchive mode, has_archive_version will be false and archive_metadata will be null.

OCR start is observable via WebSocket status messages transitioning from STARTING to WORKING with parsing_document message, and via log entries showing "Calling OCRmyPDF with args". Background workers are blocked during OCR execution since consume_file is a synchronous task. The OCR_MODE setting controls whether OCR is skipped for text-containing documents. With skip mode the pipeline is still touched but existing text is preserved via skip_text=True and an archive is created. With skip_noarchive mode OCRmyPDF is bypassed entirely when original_has_text is true and no archive is created. Weak or incomplete OCR results do not prevent document creation and the document is saved with empty content. The determination of OCR-generated versus existing text must be inferred from original_mime_type, has_archive_version, and archive_metadata fields in the API response.
