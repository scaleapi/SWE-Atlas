The place where k6 decides what “the real options” are is the config consolidation pipeline in the "cmd" package, not in the scheduler itself. The loaded test path in "cmd/test_load.go" explicitly logs “Consolidating config layers…” and calls "getConsolidatedConfig()", and the result then flows through derivation and validation before being embedded into the "lib.TestRunState" that the execution scheduler uses. That means the precedence decision is made once during test loading, and the run uses a single effective options snapshot rather than re-merging per VU.

The call site that does this is in "cmd/test_load.go" in the function that consolidates, derives, and validates configuration. It makes the consolidation step explicit, and it also shows that the script-provided options are read from the initialized runner before consolidation happens.

```go
// cmd/test_load.go
gs.Logger.Debug("Consolidating config layers...")
consolidatedConfig, err := getConsolidatedConfig(gs, cliConfig, lt.initRunner.GetOptions())
if err != nil {
    return nil, err
}

derivedConfig, err := deriveAndValidateConfig(consolidatedConfig, lt.initRunner.IsExecutable, gs.Logger)
if err != nil {
    return nil, err
}
```

The consolidation order is spelled out directly in "cmd/config.go" in "getConsolidatedConfig()". The comments there match the actual code: it starts from the CLI configuration (so defaults exist in a structured form), applies the disk config, then applies runner-provided options (which includes the script’s exported options object), then environment variables, and finally reapplies the user-supplied CLI flags on top so they have the highest priority.

```go
// cmd/config.go
func getConsolidatedConfig(gs *state.GlobalState, cliConf Config, runnerOpts lib.Options) (conf Config, err error) {
    fileConf, err := readDiskConfig(gs)
    if err != nil {
        return conf, errext.WithExitCodeIfNone(err, exitcodes.InvalidConfig)
    }
    envConf, err := readEnvConfig(gs.Env)
    if err != nil {
        return conf, errext.WithExitCodeIfNone(err, exitcodes.InvalidConfig)
    }

    conf = cliConf.Apply(fileConf)
    conf = conf.Apply(Config{Options: runnerOpts})
    conf = conf.Apply(envConf).Apply(cliConf)
    conf = applyDefault(conf)

    if _, err = metrics.GetResolversForTrendColumns(conf.SummaryTrendStats); err != nil {
        return conf, err
    }

    return conf, nil
}
```

The next important piece for understanding “why did my scenarios disappear when I passed --vus/--duration” is not in "cmd/config.go" but in "lib/options.go". The "Options.Apply()" method treats execution-related fields as a mutually-exclusive group across tiers. If a higher tier specifies any of duration, iterations, stages, or scenarios, it clears the previously set execution settings from lower tiers before applying the new ones. This is the mechanism that makes “CLI shortcut options overwrite script scenarios” behave the way it does.

```go
// lib/options.go
func (o Options) Apply(opts Options) Options {
    // ...
    if opts.Duration.Valid || opts.Iterations.Valid || opts.Stages != nil || opts.Scenarios != nil {
        o.Duration = types.NewNullDuration(0, false)
        o.Iterations = null.NewInt(0, false)
        o.Stages = nil
        o.Scenarios = nil
    }

    if opts.Duration.Valid {
        o.Duration = opts.Duration
    }
    if opts.Iterations.Valid {
        o.Iterations = opts.Iterations
    }
    if opts.Stages != nil {
        // ...
    }
    if opts.Scenarios != nil {
        o.Scenarios = opts.Scenarios
    }
    // ...
    return o
}
```

The point where this consolidation becomes “final” for a run is before the scheduler starts executing VUs. After consolidation, k6 derives and validates the config and then builds a "lib.TestRunState" that contains the options it will “always run with”. In "cmd/test_load.go", "buildTestRunState()" reinjects options into the runner and puts derivedConfig.Options into the returned TestRunState, which is what downstream execution uses.

```go
// cmd/test_load.go
func (lct *loadedAndConfiguredTest) buildTestRunState(configToReinject lib.Options) (*lib.TestRunState, error) {
    if err := lct.initRunner.SetOptions(configToReinject); err != nil {
        return nil, err
    }

    return &lib.TestRunState{
        TestPreInitState: lct.preInitState,
        Runner:           lct.initRunner,
        Options:          lct.derivedConfig.Options,
        RunTags:          lct.preInitState.Registry.RootTagSet().WithTagsFromMap(configToReinject.RunTags),
        GroupSummary:     lib.NewGroupSummary(lct.preInitState.Logger),
    }, nil
}
```

Finally, the multi-VU runs above show that once the run starts, the effective options are consistent across VUs. In the CLI override run, k6 reports vus: 3 min=3 max=3 and the console logs show VU IDs 1, 2, and 3 executing the same script body, which is consistent with the idea that option consolidation produces a single effective configuration snapshot used for the whole run, rather than per-VU drift.
