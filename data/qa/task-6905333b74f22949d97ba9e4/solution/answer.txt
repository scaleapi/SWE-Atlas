The SSH kitten's secure credential transfer relies on POSIX shared memory, which provides a way to pass sensitive data between processes without sending it over the network. When the kitten initiates an SSH session, it creates a shared memory segment in /dev/shm/ with a unique name generated from the pattern "kssh-" followed by the process ID and random characters. In my test, this produced a name like kssh-test-5V5EZYBRP3CUI. The clever part is that this shared memory only exists on the local machine where kitty runs, so the actual credentials never travel over the SSH connection itself.

The data written to shared memory uses a size-prefixed format where the first 4 bytes contain the payload length in big-endian order. For a 40-byte payload, the header bytes are [0 0 0 40], and the actual data follows immediately after. This allows the reader to know exactly how many bytes to consume without any delimiter parsing issues.

The shared memory security is enforced through multiple layers of protection. First, the SHM file is created with mode 0600 (owner read/write only), preventing other users on the system from accessing it. Second, before reading the data, the code explicitly verifies that the UID and GID of the SHM file match the current process - if they don't match, an "Incorrect owner on SHM file" error is raised. Third, the permissions are verified to be exactly 0600 - if not, an "Incorrect permissions on SHM file" error is raised. Finally, the `ReadWithSizeAndUnlink` function automatically unlinks (deletes) the shared memory segment immediately after reading, ensuring one-time access and preventing any replay attacks.

```go
mmap, err := shm.CreateTemp("kssh-test-", uint64(len(testData)+8))
shm.WriteWithSize(mmap, testData, 0)
```

The shared memory stores a JSON structure that bundles everything needed for the remote bootstrap process. This includes the base64-encoded tarfile containing shell integration scripts, a cryptographic password used to verify authenticity, and connection metadata. Running my test showed the actual structure:

```json
{
  "hostname": "example.com",
  "pw": "a1b2c3d4e5f6",
  "tarfile": "bW9ja190YXJmaWxlX2NvbnRlbnRfaGVyZQ==",
  "username": "testuser"
}
```

For a 117-byte JSON payload, the shared memory region allocates 125 bytes total to accommodate the 4-byte size header plus alignment padding, and the size header reads [0 0 0 117].

The connection_data struct serves as the central orchestration point that coordinates all aspects of establishing an SSH session. It accumulates information throughout the connection process including the remote arguments to execute, host-specific configuration options loaded from ssh.conf files, the username and hostname for configuration matching, whether to use shell or Python for the bootstrap script, a unique request ID for correlating the DCS communication, and the path to the shared memory segment. This struct gets passed through the various stages of connection setup, each adding or consuming the information it needs.

The data.sh file at the top is particularly important because it contains the environment variable setup script that runs before anything else on the remote side. This ensures the remote shell has proper TERM settings, colorterm support, and kitty-specific variables configured.

The environment serialization process converts environment variable instructions into shell commands. Each variable can be exported with a value, deleted on the remote, or copied from the local environment. For shell mode, the output looks like:

```sh
export 'TERM'="xterm-kitty"
export 'COLORTERM'="truecolor"
export 'KITTY_WINDOW_ID'="1"
export 'KITTY_SHELL_INTEGRATION'="enabled"
unset 'SOME_VAR'
```

For Python mode, the format uses JSON arrays where the third element indicates whether to quote the value literally. This distinction matters because some values need shell expansion while others must remain exactly as specified.

```python
export ["TERM", "xterm-kitty", true]
export ["COLORTERM", "truecolor", true]
unset ["SOME_VAR"]
```

Connection sharing through SSH ControlMaster is one of the kitten's most useful features because it dramatically reduces latency for subsequent connections to the same host. Instead of doing full SSH handshake and authentication for every connection, the first connection establishes a persistent control socket that subsequent connections multiplex through. I tested the SSH argument parsing with sample arguments and observed how the kitten separates SSH options from the destination and remote command:

```
Input: ["-o", "StrictHostKeyChecking=no", "-p", "22", "user@host", "ls", "-la"]

Output:
SSH args: [-o StrictHostKeyChecking=no -p 22]
Server args: [user@host ls -la]
Passthrough: false
Extra args: []
```

Certain SSH flags indicate the user wants direct SSH behavior without kitten processing. The passthrough detection identifies these flags:

```
Passthrough flags: map[-G:true -N:true -T:true -V:true -f:true -n:true]
```

When the kitten sees any of these flags, it immediately execs ssh directly, bypassing all the bootstrap and shell integration logic. This is important for users who need raw SSH functionality.

The kitten interrogates the local SSH version because certain features require minimum versions.

The key version threshold is 8.4, which introduced SSH_ASKPASS_REQUIRE=force support. When running on SSH 8.4+, the kitten can reliably force askpass usage even when a terminal is present, which enables more seamless password prompting through kitty's interface rather than the terminal.

The control path for connection sharing uses a template that incorporates the kitty process ID and an SSH placeholder. The format becomes something like kitty-ssh-12345-%C where %C is an SSH token that expands to a hash of the connection parameters (local host, remote host, port, username). This ensures different destinations get different control sockets while connections to the same destination share one. The generated SSH options include ControlMaster=auto (create master if none exists, otherwise use existing), ControlPersist=yes (keep master alive after initial connection closes), ServerAliveInterval=60 (ping every 60 seconds), ServerAliveCountMax=5 (allow 5 missed pings), and TCPKeepAlive=no (disable TCP keepalives since SSH-level keepalives are more reliable).

When reusing connections, the kitten first checks if a working control master already exists by running ssh with the -O check option. This sends a status query through the control socket. If the check succeeds, the existing master is functional and the kitten routes the new connection through it. If the check fails (no master exists or it died), the kitten starts a new background ssh process with -N (no remote command) and -f (go to background) flags to establish a fresh control socket.

Bootstrap script encoding solves a tricky problem: the remote shell might be bash, zsh, fish, tcsh, or something else entirely, each with different quoting rules. The kitten cannot know which shell the remote user has, so it must encode the bootstrap script in a way that survives being passed through any POSIX-compatible shell. The solution is clever character substitution that replaces problematic characters with control characters that have no special meaning to shells:

```
Encoding scheme:
  ' (single quote, 0x27) -> \v (vertical tab, 0x0B)
  \ (backslash, 0x5C)    -> \f (form feed, 0x0C)
  \n (newline, 0x0A)     -> \r (carriage return, 0x0D)
  ! (exclamation, 0x21)  -> \b (backspace, 0x08)
```

The remote side reverses this transformation using tr before evaluating the script:

```sh
eval "$(echo "$0" | tr \v\f\r\b \047\134\n\041)"
```

This approach means the encoded script can be safely passed as a single argument to any shell. For Python-based bootstrap (used when the remote has Python available), the entire script gets base64 encoded instead, and the unwrap command is:

```python
import base64, sys; eval(compile(base64.standard_b64decode(sys.argv[-1]), 'bootstrap.py', 'exec'))
```

The DCS (Device Control String) escape sequence mechanism enables bidirectional communication between the kitty terminal and the remote shell. The format is ESC P @ kitty-TYPE | BASE64_DATA ESC \ where TYPE identifies the message purpose (like "ssh" for session data or "print" for debug output) and the payload is base64 encoded to avoid any escape sequence injection issues. The bootstrap script's dcs_to_kitty function implements this:

```sh
dcs_to_kitty() { printf "\033P@kitty-$1|%s\033\134" "$(printf "%s" "$2" | base64_encode)" > /dev/tty; }
```

When the remote bootstrap needs to retrieve the tarfile data, it sends a DCS sequence with type "ssh" containing the request ID, the shared memory path (password filename), and the password.

```
Payload: id=123-456:pwfile=/dev/shm/kssh-test:pw=abc123
Base64:  aWQ9MTIzLTQ1Njpwd2ZpbGU9L2Rldi9zaG0va3NzaC10ZXN0OnB3PWFiYzEyMw==
DCS sequence length: 79 bytes
```

Kitty receives this through the terminal connection, reads the data from the referenced shared memory, verifies the password, and sends back the tarfile contents wrapped between KITTY_DATA_START and KITTY_DATA_END markers. The bootstrap script reads this data, base64 decodes it, extracts the tarfile, and applies the shell integration.

Signal handling ensures clean behavior when users interrupt SSH sessions. The kitten sets up a goroutine that receives SIGINT (Ctrl+C) and SIGTERM signals. Since these signals typically get delivered to both the kitten and the SSH child process, the kitten's strategy is to ignore them and let SSH handle interruption naturally. This prevents confusing double-handling where both processes try to clean up. After the SSH process exits, the cleanup function restores terminal settings like echo mode and any modified terminal private modes. A separate drain function reads any residual data from the terminal to prevent leftover escape sequences from corrupting the user's shell prompt after the session ends.
