I started by locating where metadata is actually pulled during import. The API path calls get_metadata() in documents/views.py, which chooses a parser based on MIME type and calls extract_metadata(). For PDFs, that lands in RasterisedDocumentParser.extract_metadata() in paperless_tesseract/parsers.py, which converts PDF metadata values into Python strings and builds a list of dicts.

Here is the relevant code path I’m measuring, with the exact file path:

```
# /app/src/paperless_tesseract/parsers.py
def extract_metadata(self, document_path, mime_type):
    result = []
    if mime_type == 'application/pdf':
        import pikepdf
        namespace_pattern = re.compile(r"\{(.*)\}(.*)")
        pdf = pikepdf.open(document_path)
        meta = pdf.open_metadata()
        for key, value in meta.items():
            if isinstance(value, list):
                value = " ".join([str(e) for e in value])
            value = str(value)
            try:
                m = namespace_pattern.match(key)
                result.append({
                    "namespace": m.group(1),
                    "prefix": meta.REVERSE_NS[m.group(1)],
                    "key": m.group(2),
                    "value": value
                })
```

- Creates a PDF with large XMP metadata (5 MB description + 2 MB keywords).
- Calls RasterisedDocumentParser.extract_metadata() and TextDocumentParser for comparison.
- Captures RSS and line-by-line tracemalloc allocations.

This shows the core pattern:

- Text documents don’t allocate anything significant for metadata (empty list).
- PDF metadata extraction allocates inside pikepdf.models.metadata and then builds a list of Python dicts/strings in paperless_tesseract/parsers.py.
- The spike is triggered by large metadata values (XMP/keywords), not by file size itself. The big metadata PDF produces multi‑MB allocations on the pikepdf lines shown above.
- Batch behavior: keeping metadata for multiple documents causes RSS to climb (125,196 → 148,848 KB). Clearing results and GC returns most memory, but RSS doesn’t always immediately drop to the exact baseline, which matches Python’s allocator behavior.

About delayed release: after the BIG metadata run, tracemalloc_current_bytes drops to 6,140 while RSS stays at 125,196 KB, which is consistent with CPython’s pymalloc arena behavior—freed objects go back to Python’s arenas and pools, but arenas are not returned to the OS until they are fully empty. That explains why memory “doesn’t get released back to the system” even when the Python allocations are gone.

Putting this together:

- The spike is triggered when metadata extraction for PDFs encounters large XMP/metadata values. pikepdf pulls those values into Python, then RasterisedDocumentParser.extract_metadata() converts them into new Python strings and dicts. That creates a disproportionate spike for “small” PDFs that happen to carry large metadata blobs.
- Text documents don’t trigger the spike because TextDocumentParser doesn’t extract metadata (metadata list empty).
- Batch imports amplify the effect because metadata lists stay referenced across multiple documents until the batch or response is complete. The batch run shows RSS growing with each retained result and then dropping after clearing the list.
- .all() versus .iterator() matters in the same way: .all() populates an in‑memory list and caches it, while .iterator() streams rows without caching; the measured RSS difference confirms this.
- Memory that appears “not released” is consistent with CPython arena behavior: tracemalloc shows low current allocations after GC, yet RSS remains higher because arenas are retained for reuse rather than returned to the OS.
