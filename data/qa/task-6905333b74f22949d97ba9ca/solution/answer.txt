This reveals how ISO-TP encodes frame types in the first byte (the N_PCI byte). The upper nibble indicates the frame type: 0x0X for Single Frame, 0x1X for First Frame, 0x2X for Consecutive Frame, and 0x3X for Flow Control. The lower nibble carries either the payload length (for SF), part of the message length (for FF), or the sequence number (for CF). Standard CAN frames have a maximum data length of 8 bytes, which is why segmentation is needed for larger diagnostic payloads.

Now I tested the segmentation threshold to determine the maximum payload that fits in a single frame.

```bash
cd /app && python3 -c "
import sys
sys.path.insert(0, '/app')
from scapy.contrib.isotp import ISOTP

payload_7 = b'\x11\x22\x33\x44\x55\x66\x77'
frames_7 = ISOTP(data=payload_7, rx_id=0x641).fragment()
print(f'7-byte payload: {len(frames_7)} frame (Single Frame)')
print(f'first byte: 0x{bytes(frames_7[0].data)[0]:02X}')

payload_8 = b'\x11\x22\x33\x44\x55\x66\x77\x88'
frames_8 = ISOTP(data=payload_8, rx_id=0x641).fragment()
print(f'8-byte payload: {len(frames_8)} frames (First Frame + Consecutive Frames)')
"
```

Output:
```text
7-byte payload: 1 frame (Single Frame)
first byte: 0x07
8-byte payload: 2 frames (First Frame + Consecutive Frames)
```

The maximum payload size before segmentation kicks in is 7 bytes. This makes sense because a CAN frame has 8 bytes of data, and a Single Frame needs 1 byte for the N_PCI header (which encodes type 0x0 in the upper nibble and the length 0x07 in the lower nibble). So 8 - 1 = 7 bytes of payload fit in a single frame. For the 7-byte payload, the first byte is 0x07, meaning Single Frame type (0x0) with length 7.

When the payload exceeds 7 bytes, ISO-TP must use multi-frame transmission with a First Frame followed by Consecutive Frames.

Next, I tested fragmenting a 20-byte diagnostic payload to see exactly what frames get generated.

```bash
cd /app && python3 -c "
import sys
sys.path.insert(0, '/app')
from scapy.contrib.isotp import ISOTP

payload_20 = bytes(range(1, 21))
print(f'payload (20 bytes): {payload_20.hex()}')
print(f'payload as list: {list(payload_20)}')

isotp_msg = ISOTP(data=payload_20, rx_id=0x641)
frames = isotp_msg.fragment()

print(f'number of CAN frames generated: {len(frames)}')

for i, frame in enumerate(frames):
    frame_data = bytes(frame.data)
    first_byte = frame_data[0]
    frame_type = (first_byte & 0xF0) >> 4
    
    type_names = {0: 'Single Frame (SF)', 1: 'First Frame (FF)', 
                  2: 'Consecutive Frame (CF)', 3: 'Flow Control (FC)'}
    
    print(f'frame {i+1}:')
    print(f'  CAN ID: 0x{frame.identifier:03X}')
    print(f'  raw data: {frame_data.hex()}')
    print(f'  raw data (bytes): {list(frame_data)}')
    print(f'  first byte: 0x{first_byte:02X} (binary: {first_byte:08b})')
    print(f'  frame type (upper nibble): {frame_type} = {type_names.get(frame_type)}')
    
    if frame_type == 1:
        length = ((first_byte & 0x0F) << 8) | frame_data[1]
        print(f'  message length (12-bit): {length} bytes')
        print(f'  first frame data bytes: {frame_data[2:].hex()} ({len(frame_data[2:])} bytes)')
    elif frame_type == 2:
        seq_num = first_byte & 0x0F
        print(f'  sequence number (lower nibble): {seq_num}')
        print(f'  CF data bytes: {frame_data[1:].hex()} ({len(frame_data[1:])} bytes)')
"
```

```text
  sequence number (lower nibble): 2
  CF data bytes: 0e0f1011121314 (7 bytes)
```

For a 20-byte payload, ISO-TP generates 3 CAN frames: 1 First Frame and 2 Consecutive Frames.

Looking at the first bytes of each frame:
- Frame 1: 0x10 - This is a First Frame (upper nibble 0x1). The second byte 0x14 (decimal 20) combined with the lower nibble of the first byte gives the total message length. The FF header uses 2 bytes, leaving 6 bytes for data in this frame.
- Frame 2: 0x21 - This is a Consecutive Frame (upper nibble 0x2) with sequence number 1 (lower nibble). CF headers use only 1 byte, so 7 bytes of payload fit.
- Frame 3: 0x22 - Another Consecutive Frame with sequence number 2.

The math works out: 6 bytes (FF) + 7 bytes (CF1) + 7 bytes (CF2) = 20 bytes total.

Now I tested what happens with a very large payload of 5000 bytes to see if it causes an error or uses extended encoding.

```bash
cd /app && python3 -c "
import sys
sys.path.insert(0, '/app')
from scapy.contrib.isotp import ISOTP
from scapy.contrib.isotp.isotp_packet import ISOTP_MAX_DLEN, ISOTP_MAX_DLEN_2015

print(f'ISOTP_MAX_DLEN (12-bit standard): {ISOTP_MAX_DLEN} bytes (0x{ISOTP_MAX_DLEN:X})')
print(f'ISOTP_MAX_DLEN_2015 (32-bit extended): {ISOTP_MAX_DLEN_2015} bytes (0x{ISOTP_MAX_DLEN_2015:X})')

payload_5000 = bytes(range(256)) * 20
payload_5000 = payload_5000[:5000]
print(f'testing 5000-byte payload...')
print(f'5000 bytes is > ISOTP_MAX_DLEN ({ISOTP_MAX_DLEN})')

isotp_5000 = ISOTP(data=payload_5000, rx_id=0x641)
frames_5000 = isotp_5000.fragment()
print(f'fragmented into {len(frames_5000)} CAN frames')

ff_data = bytes(frames_5000[0].data)
print(f'first frame data: {ff_data[:8].hex()}')

if ff_data[0] == 0x10 and ff_data[1] == 0x00:
    length_32bit = (ff_data[2] << 24) | (ff_data[3] << 16) | (ff_data[4] << 8) | ff_data[5]
    print(f'uses 32-bit length encoding (ISO 15765-2:2015)')
    print(f'encoded length: {length_32bit} bytes')
"
```

Output:
```text
ISOTP_MAX_DLEN (12-bit standard): 4095 bytes (0xFFF)
ISOTP_MAX_DLEN_2015 (32-bit extended): 4294967295 bytes (0xFFFFFFFF)
testing 5000-byte payload...
5000 bytes is > ISOTP_MAX_DLEN (4095)
fragmented into 715 CAN frames
first frame data: 1000000013880001
uses 32-bit length encoding (ISO 15765-2:2015)
encoded length: 5000 bytes
```

The 5000-byte payload works without error. Since 5000 exceeds the original 12-bit maximum of 4095 bytes, Scapy automatically uses the ISO 15765-2:2015 extended format. In this format, the first two bytes are 0x10 0x00 (indicating FF type with zero in the 12-bit length field as an escape sequence), followed by a 4-byte (32-bit) length field. The value 0x00001388 is 5000 in hexadecimal, confirming the encoding.

The result is 715 CAN frames: 1 First Frame plus 714 Consecutive Frames, which is expected since each CF carries 7 bytes and (5000 - 2) / 7 ≈ 714 additional frames are needed.

Next, I investigated the timeout behavior for receiving segmented messages. I checked the source code for the timeout configuration.

```bash
cd /app && grep -n "timeout = " scapy/contrib/isotp/isotp_soft_socket.py
```

Output:
```text
496:        self.fc_timeout = 1
497:        self.cf_timeout = 1
```

The timeout values are both set to 1 second. `fc_timeout` is the timeout for waiting for a Flow Control frame after sending a First Frame, and `cf_timeout` is the timeout for waiting for Consecutive Frames during reception. If a middle frame never arrives during reception, the cf_timeout of 1 second applies.

I then checked what happens when the timeout fires by examining the timeout handler:

```bash
cd /app && sed -n '619,630p' scapy/contrib/isotp/isotp_soft_socket.py
```

Output:
```text
    def _rx_timer_handler(self):
        # type: () -> None
        """Method called every time the rx_timer times out, due to the peer not
        sending a consecutive frame within the expected time window"""

        if self.rx_state == ISOTP_WAIT_DATA:
            # we did not get new data frames in time.
            # reset rx state
            self.rx_state = ISOTP_IDLE
            if conf.verb > 2:
                log_isotp.warning("RX state was reset due to timeout")
```

When a timeout occurs because a consecutive frame doesn't arrive within the expected window, the `_rx_timer_handler` method resets the receive state to `ISOTP_IDLE` and logs the warning message "RX state was reset due to timeout". This effectively abandons the incomplete message. No exception is raised to the application - the partial message is simply discarded and the socket returns to an idle state ready for new messages.

Finally, I tested what happens when a middle frame is missing during message reassembly:

```bash
cd /app && python3 -c "
import sys
sys.path.insert(0, '/app')
from scapy.contrib.isotp.isotp_utils import ISOTPMessageBuilder
from scapy.layers.can import CAN

ff_data = b'\x10\x14' + bytes(range(1, 7))
cf1_data = b'\x21' + bytes(range(7, 14))
cf2_data = b'\x22' + bytes(range(14, 21))

print(f'FF:  {ff_data.hex()}')
print(f'CF1: {cf1_data.hex()} (this one will be missing)')
print(f'CF2: {cf2_data.hex()}')

ff = CAN(identifier=0x641, data=ff_data)
cf2 = CAN(identifier=0x641, data=cf2_data)

builder = ISOTPMessageBuilder(use_ext_address=False)

builder.feed(ff)
print(f'after FF - ready messages: {builder.count}, pending buckets: {len(builder.buckets)}')

builder.feed(cf2)
print(f'after CF2 (skipped CF1) - ready messages: {builder.count}, pending buckets: {len(builder.buckets)}')
print()
print('CF2 is ignored because builder expects seq=1 but got seq=2')
print('bucket stays waiting for seq=1 that never comes')
print('in real socket, after 1 second timeout fires and logs:')
print('  WARNING: RX state was reset due to timeout')
"
```

Output:
```text
FF:  1014010203040506
CF1: 210708090a0b0c0d (this one will be missing)
CF2: 220e0f1011121314
after FF - ready messages: 0, pending buckets: 1
after CF2 (skipped CF1) - ready messages: 0, pending buckets: 1

CF2 is ignored because builder expects seq=1 but got seq=2
bucket stays waiting for seq=1 that never comes
in real socket, after 1 second timeout fires and logs:
  WARNING: RX state was reset due to timeout
```

This demonstrates that ISO-TP enforces strict sequence ordering. When CF1 (sequence number 1) is missing and CF2 (sequence number 2) arrives instead, CF2 is ignored because the receiver expects frames in order. The pending bucket stays open waiting for the expected sequence number. In a real socket scenario, after 1 second the cf_timeout fires and the warning "RX state was reset due to timeout" is logged.

In summary:
- The maximum payload that fits in a single CAN frame is 7 bytes (using Single Frame format)
- Payloads of 8+ bytes require segmentation using First Frame + Consecutive Frames
- A 20-byte payload generates 3 frames with first bytes 0x10, 0x21, 0x22
- Payloads up to 4095 bytes use the standard 12-bit length encoding in the FF header
- Payloads larger than 4095 bytes (like 5000 bytes) use the ISO 15765-2:2015 extended 32-bit length encoding
- Payloads exceeding 4,294,967,295 bytes raise "Too much data in ISOTP message"
- When receiving segmented messages, if a middle frame is missing, the cf_timeout of 1 second applies
- After timeout, the warning "RX state was reset due to timeout" is logged and the incomplete message is discarded
