I started with tracing the back navigation code path to understand why the behavior feels deterministic rather than random. The erratic back navigation behavior in multi-step onboarding flows stems from a fallback mechanism in the Redux connect function that maps external back targets into every step, not just the flow's entry point. The `back_to` query parameter is designed to return users to their origin (such as the Goals screen) when they exit the first step of a flow, but a nullish coalescing operator treats it as a universal default that silently overrides the expected previous-step navigation for all subsequent steps.

Inspecting the connect function in `client/signup/step-wrapper/index.jsx` reveals the destination decision logic:

```javascript
export default connect( ( state, ownProps ) => {
	const backToParam = getCurrentQueryArguments( state )?.back_to?.toString();
	const backTo = backToParam?.startsWith( '/' ) ? backToParam : undefined;

	const backUrl = ownProps.backUrl ?? backTo;

	return {
		backUrl,
		userLoggedIn: isUserLoggedIn( state ),
	};
} )( localize( StepWrapper ) );
```

The critical line is `const backUrl = ownProps.backUrl ?? backTo`. The nullish coalescing operator (`??`) returns the right-hand operand whenever the left-hand operand is `null` or `undefined`. This means any step component that does not explicitly pass a `backUrl` prop will inherit the `back_to` query parameter as its back destination, regardless of where it sits in the flow sequence.

The flow configuration in `client/signup/config/flows-pure.js` shows how DIFM flows receive the `back_to` parameter:

```javascript
{
	name: DIFM_FLOW,
	steps: [
		userSocialStep,
		'new-or-existing-site',
		'difm-site-picker',
		'difm-options',
		'social-profiles',
		'difm-design-setup-site',
		'difm-page-picker',
	],
	destination: getDIFMSignupDestination,
	providesDependenciesInQuery: [ 'coupon', 'back_to' ],
	optionalDependenciesInQuery: [ 'coupon', 'back_to' ],
}
```

The `back_to` parameter persists in the URL throughout the flow and remains accessible to the connect function at every step.

The `NavigationLink` component in `client/signup/navigation-link/index.jsx` determines the actual navigation behavior. The `getBackUrl` method checks for an explicit `backUrl` prop first:

```javascript
getBackUrl() {
	if ( this.props.direction !== 'back' ) {
		return;
	}

	if ( this.props.backUrl ) {
		return this.props.backUrl;
	}

	const fallbackQueryParams = window.location.search
		? Object.fromEntries( new URLSearchParams( window.location.search ).entries() )
		: undefined;

	const {
		flowName,
		signupProgress,
		stepName,
		userLoggedIn,
		queryParams = fallbackQueryParams,
	} = this.props;
	const previousStep = this.getPreviousStep( flowName, signupProgress, stepName );

	const stepSectionName = get(
		this.props.signupProgress,
		[ previousStep.stepName, 'stepSectionName' ],
		''
	);

	const locale = ! userLoggedIn ? getLocaleSlug() : '';

	return getStepUrl(
		previousStep.lastKnownFlow || this.props.flowName,
		previousStep.stepName,
		stepSectionName,
		locale,
		queryParams
	);
}
```

When `this.props.backUrl` is truthy, the component returns it directly without computing the previous step. The override occurs because the connect function in `StepWrapper` populates `backUrl` from the query parameter fallback, causing NavigationLink to short-circuit its normal previous-step calculation.

The output confirms the behavior across all steps. The first step (`user-social` at position 0) correctly computes its `backUrl` as `/setup/goals`, which is the intended behavior for an entry point that should return users to their origin. However, every subsequent step—`new-or-existing-site` at position 1, `difm-site-picker` at position 2, `difm-options` at position 3, `social-profiles` at position 4, `difm-design-setup-site` at position 5, and `difm-page-picker` at position 6—all compute their `backUrl` as `/setup/goals` rather than their respective previous steps. Each of these intermediate steps falls back to the query parameter instead of computing the previous step in the flow sequence.

Every step beyond the first receives `/setup/goals` as its back destination instead of the previous step in the flow. Users clicking Back on step 3 expect to return to step 2 but are sent directly to the Goals screen, skipping all intermediate progress.

The `positionInFlow` prop is available to the component and represents the current step's index in the flow sequence: 0 for the first step, 1 for the second, and so on. The `headerText()` and `subHeaderText()` methods in the same component already use this property to customize content for the first step:

```javascript
headerText() {
	if ( this.props.positionInFlow === 0 ) {
		if ( this.props.headerText !== undefined ) {
			return this.props.headerText;
		}
		return this.props.translate( 'Let's get started' );
	}
	if ( this.props.fallbackHeaderText !== undefined ) {
		return this.props.fallbackHeaderText;
	}
}
```

However, the connect function does not consult `positionInFlow` when deciding the `backUrl` fallback. The nullish coalescing applies unconditionally.

The code path that handles the expected step-by-step navigation—and is being bypassed—is the `getPreviousStep` method in `NavigationLink`:

```javascript
getPreviousStep( flowName, signupProgress, currentStepName ) {
	const previousStep = { stepName: null };

	if ( isFirstStepInFlow( flowName, currentStepName, this.props.userLoggedIn ) ) {
		return previousStep;
	}

	const filteredProgressedSteps = getFilteredSteps(
		flowName,
		signupProgress,
		this.props.userLoggedIn
	).filter( ( step ) => ! step.wasSkipped );

	if ( filteredProgressedSteps.length === 0 ) {
		return previousStep;
	}

	const currentStepIndexInProgress = filteredProgressedSteps.findIndex(
		( step ) => step.stepName === currentStepName
	);

	if ( currentStepIndexInProgress === -1 ) {
		return filteredProgressedSteps.pop();
	}

	return filteredProgressedSteps[ currentStepIndexInProgress - 1 ] || previousStep;
}
```

This method properly handles skipped steps, flow branching, and finds the user's actual previous step in their navigation history. This logic never executes for steps beyond the first when a `back_to` query parameter exists because the `backUrl` prop is always truthy due to the query parameter fallback.

The render method in `NavigationLink` also has a guard for first-step back buttons:

```javascript
if (
	this.props.positionInFlow === 0 &&
	this.props.direction === 'back' &&
	! this.props.stepSectionName &&
	! this.props.allowBackFirstStep
) {
	return null;
}
```

This hides the Back button entirely on the first step unless `allowBackFirstStep` is explicitly set. Steps like `new-or-existing-site` set `allowBackFirstStep={ !! backUrl }` to show the Back button only when an external back target is available:

```javascript
<StepWrapper
	backUrl={ backUrl }
	allowBackFirstStep={ !! backUrl }
	{ ...props }
/>
```

The precedence rule that lets the external override take control is the nullish coalescing in the connect function. When three potential sources for the back destination disagree—the component prop `backUrl` explicitly passed by the step component, the query parameter `back_to` representing the external entry point, and the computed previous step determined by `NavigationLink.getPreviousStep()`—the current code treats the query parameter as equivalent to the component prop when the prop is absent, meaning the computed previous step is never reached for any step where the query parameter exists. The query parameter wins by default for all steps that do not explicitly pass their own `backUrl` prop.

The specific scenario where the flow "slips out into an entirely different flow" occurs when the `back_to` parameter points to a different flow's entry point. If a user enters the DIFM flow from the Goals screen (`/setup/goals`) and navigates to step 4, clicking Back returns them to the Goals screen instead of step 3. The Goals screen may then redirect into a completely different onboarding flow based on the user's current state.

When step 0 has no explicit `backUrl` prop but the query contains `back_to=/setup/goals`, the query parameter wins and the computed `backUrl` becomes `/setup/goals`—this is correct for the first step. When step 1 or step 2 or any later step have no explicit `backUrl` prop under the same query conditions, the query parameter still wins and overrides what would otherwise be the previous-step computation, resulting in `/setup/goals` instead of the preceding step. However, when any step passes an explicit `backUrl` prop (such as `/explicit/path`), that explicit prop takes highest precedence regardless of position in the flow, and the query parameter is ignored. The explicit component prop always wins when present; the query parameter wins when the prop is absent; and the computed previous step never gets a chance to determine the destination because it sits behind a truthy `backUrl` check that the query parameter fallback satisfies.

The condition that causes the override is structural: the nullish coalescing operator does not distinguish between "absent" and "intentionally unset," nor does it consider `positionInFlow`. The query parameter fallback applies to all steps equally, which is why back navigation "has a mind of its own" and snaps to the external target regardless of the current step's position in the flow.
