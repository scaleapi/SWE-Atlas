I started investigating TruffleHog's detection behavior by examining the AWS detector implementation and running entropy calculations to understand the detection thresholds.

The AWS detector in pkg/detectors/aws/common.go defines two entropy thresholds that control detection.

```go
const (
    RequiredIdEntropy     = 3.0
    RequiredSecretEntropy = 4.25
)
```

These thresholds use Shannon entropy to filter out low-randomness credentials that are likely placeholders or test values. I wrote a test script to calculate entropy values for various credential patterns.

The entropy calculation in pkg/detectors/falsepositives.go implements Shannon entropy using character frequency analysis.

```go
func StringShannonEntropy(input string) float64 {
    chars := make(map[rune]float64)
    inverseTotal := 1 / float64(len(input))

    for _, char := range input {
        chars[char]++
    }

    entropy := 0.0
    for _, count := range chars {
        probability := count * inverseTotal
        entropy += probability * math.Log2(probability)
    }

    return -entropy
}
```

The detection process in pkg/detectors/aws/access_keys/accesskey.go applies these thresholds during credential matching. When the scanner finds a potential AWS access key ID, it immediately checks the entropy before proceeding.

```go
for idMatch := range idMatches {
    if detectors.StringShannonEntropy(idMatch) < aws.RequiredIdEntropy {
        continue
    }
    // ... process further only if entropy passes
    for secretMatch := range secretMatches {
        if detectors.StringShannonEntropy(secretMatch) < aws.RequiredSecretEntropy {
            continue
        }
        // credential pair passes entropy check
    }
}
```

This explains why credentials in test fixtures often escape detection. A key ID like AKIAAAAAAAAAAAAAAAAA has entropy of only 0.57, far below the 3.0 threshold. Even a slightly patterned ID like AKIAAABBBBCCCCDDDDEE with entropy 2.62 fails to meet the threshold.

The second filter mechanism involves false positive wordlists. TruffleHog maintains several embedded wordlists that filter credentials containing common programming terms.

```go
var (
    filter *ahocorasick.Trie

    //go:embed "fp_badlist.txt"
    badList []byte
    //go:embed "fp_words.txt"
    wordList []byte
    //go:embed "fp_programmingbooks.txt"
    programmingBookWords []byte
    //go:embed "fp_uuids.txt"
    uuidList []byte
)
```

The fp_badlist.txt contains common programming terms including the word "example" which appears on line 288. This is why AWS's official example credentials AKIAIOSFODNN7EXAMPLE are never detected despite having entropy of 3.68 which passes the threshold. The false positive check catches it.

```go
func IsKnownFalsePositive(match string, falsePositives map[FalsePositive]struct{}, wordCheck bool) (bool, string) {
    if !utf8.ValidString(match) {
        return true, "invalid utf8"
    }
    lower := strings.ToLower(match)

    for fp := range falsePositives {
        fps := string(fp)
        if strings.Contains(lower, fps) {
            return true, "contains term: " + fps
        }
    }

    if wordCheck {
        if m := filter.MatchFirstString(lower); m != nil {
            return true, "matches wordlist: " + m.MatchString()
        }
    }

    return false, ""
}
```

The encoding bypass behavior stems from TruffleHog's decoder architecture in pkg/decoders/decoders.go. The scanner applies a fixed set of decoders to each chunk of data before running detection.

```go
func DefaultDecoders() []Decoder {
    return []Decoder{
        // UTF8 must be first for duplicate detection
        &UTF8{},
        &Base64{},
        &UTF16{},
        &EscapedUnicode{},
    }
}
```

This decoder list reveals a critical limitation. TruffleHog decodes Base64, UTF16, and escaped unicode, but there is no hex decoder. Credentials encoded in hexadecimal like 414b4941494f53464f444e4e374558414d504c45 for AKIAIOSFODNN7EXAMPLE will not be detected because the scanner cannot decode them.

The Base64 decoder in pkg/decoders/base64.go processes potential base64 strings and replaces them with decoded content.

```go
func (d *Base64) FromChunk(chunk *sources.Chunk) *DecodableChunk {
    decodableChunk := &DecodableChunk{Chunk: chunk, DecoderType: d.Type()}
    encodedSubstrings := getSubstringsOfCharacterSet(chunk.Data, 20, b64CharsetMapping, b64EndChars)
    decodedSubstrings := make(map[string][]byte)

    for _, str := range encodedSubstrings {
        dec, err := base64.StdEncoding.DecodeString(str)
        if err == nil && len(dec) > 0 && isASCII(dec) {
            decodedSubstrings[str] = dec
        }
    }
    // ... replaces encoded with decoded in chunk
}
```

However, this decoder runs exactly once. If credentials are double-encoded with base64, only the outer layer gets decoded. The inner base64 string remains encoded and the actual credential is never exposed to the detector. This is why developers who apply multiple encoding layers can accidentally bypass detection.


The verification disabled for safety warning originates from the overlap detection mechanism in pkg/engine/engine.go.

```go
var errOverlap = errors.New(
    "More than one detector has found this result. For your safety, verification has been disabled." +
        "You can override this behavior by using the --allow-verification-overlap flag.",
)
```

When the same secret matches multiple detectors, TruffleHog considers this a potential security risk. A malicious detector could be designed to match legitimate secrets and intercept them during verification by sending them to an attacker-controlled endpoint. The engine protects against this by disabling verification when overlap is detected.

The likelyDuplicate function uses Levenshtein distance to detect near-duplicates across different detector types.

```go
func likelyDuplicate(ctx context.Context, val chunkSecretKey, dupes map[chunkSecretKey]struct{}) bool {
    const similarityThreshold = 0.9

    valStr := val.secret
    for dupeKey := range dupes {
        dupe := dupeKey.secret
        // If the detector type is the same, we don't need to compare
        if val.detectorKey.Type() == dupeKey.detectorKey.Type() {
            continue
        }

        if valStr == dupe {
            return true
        }

        similarity := strutil.Similarity(valStr, dupe, metrics.NewLevenshtein())
        if similarity > similarityThreshold {
            return true
        }
    }
    return false
}
```

This triggers when different detectors find secrets that are identical or more than 90% similar. For example, if the AWS detector and a generic API key detector both match the same 40-character string, verification gets disabled. The scanner still reports the finding but marks it with the overlap warning.

```go
if likelyDuplicate(ctx, key, chunkSecrets) {
    res.SetVerificationError(errOverlap)
    // process result without verification
}
```

Credentials that slip through TruffleHog detection fall into these categories: hex-encoded credentials bypass the decoder layer entirely, double base64 encoding hides credentials behind an undecoded outer layer, low entropy credentials with repeating patterns fail the Shannon entropy threshold, and credentials containing common programming words like example or test get filtered as false positives.

Credentials that get caught include plaintext high-entropy credentials, single-layer base64 encoded credentials which the decoder processes successfully, and credentials with sufficient randomness that pass both entropy thresholds without containing false positive terms.

The boundary behavior at exactly the threshold values shows the secret key threshold is particularly tight. A secret with entropy 4.30 like ABCDEFGHabcdefgh1234ABCDEFGHabcdefgh12 just barely passes the 4.25 threshold, while slight reductions in character variety would push it below. The ID threshold at 3.0 is more forgiving, allowing moderately patterned strings as long as they maintain some character diversity.

Detection is deterministic because all these checks are based on the content itself. The same file with the same credentials will always produce the same detection result. The inconsistency across different files with apparently similar credentials stems from subtle differences in entropy values or the presence of false positive trigger words that may not be immediately obvious.
