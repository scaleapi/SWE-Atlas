I started by setting up a virtual framebuffer to enable running Kitty in a headless environment, then built and launched Kitty with debug input logging to observe the input handling pipeline at runtime.

Setting up the environment and building Kitty:

$ Xvfb :99 -screen 0 1024x768x24 &
$ export DISPLAY=:99
$ cd /app && python3 setup.py build --verbose 2>&1 | tail -5

Output:

/usr/local/go/bin/go build -v -ldflags '-X kitty.VCSRevision=815df1e210e0a9ab4622f5c7f2d6891d7dbeddf1 -s -w' -o kitty/launcher/kitten /app/tools/cmd

The build compiled both the C extensions (using gcc) and the Go-based kitten tool. With the build complete, I launched Kitty with debug input logging enabled:

$ ./kitty/launcher/kitty --debug-input 2>&1 &
KITTY_PID=$!

Output:

The startup logs show that GLFW initialized the XKB keymap first, then the window received focus. This tells us GLFW is the first component to see window-level events.

To create overlapping input activity, I sent rapid keystrokes while Kitty was running:

$ xdotool key a b c Return

The debug output showed the complete input flow:

This output reveals the complete input decision process. The Press event shows GLFW receiving the X11 keycode (0x26), translating it via XKB to the symbol 'a', and invoking the C-level on_key_input handler. The handler determines this is a text key and sends it to the child process. The Release event is explicitly ignored because the terminal's keyboard mode doesn't require release events to be encoded.

The Python stack shows the main thread running _run_app in kitty/main.py, which is the event loop. However, this only shows the Python portion. To see the complete picture including C code, I used strace with multi-thread tracing:

$ strace -f -p $KITTY_PID -e trace=write -s 10 -t 2>&1

While sending a keypress:
$ xdotool key y

The strace output showed the complete end-to-end correlation from keypress to pty:

[pid  8353] 04:52:12 write(2, "on_key_input"..., 99) = -1 EPIPE
[pid  8353] 04:52:12 write(8, "\1\0\0\0\0\0\0\0", 8) = 8
[pid  8353] 04:52:12 write(2, "sent key as text to child: y\n", 29) = -1 EPIPE
[pid  8353] 04:52:12 write(6, "\1\0\0\0\0\0\0\0", 8) = 8
[pid  8389] 04:52:12 write(10, "y", 1)  = 1
[pid  8389] 04:52:12 write(6, "\1\0\0\0\0\0\0\0", 8) = 8

This trace is critical for understanding the architecture. Thread 8353 (the main/UI thread) processes the key in on_key_input, then writes to fd 8 to signal the child monitor thread. Thread 8389 (the ChildMonitor thread) then writes the actual character "y" to fd 10, which is the pty master (/dev/pts/ptmx). I verified this by checking the file descriptors:

$ ls -la /proc/$KITTY_PID/fd/10
lrwx------ 1 root root 64 Jan 21 04:50 10 -> /dev/pts/ptmx

This confirms the complete path: GLFW callback (C) → on_key_input in keys.c (C) → schedule_write_to_child queues data → ChildMonitor thread writes to pty.

Examining the source code confirms this architecture. In keys.c, the on_key_input function is called from the GLFW key callback:

// From kitty/keys.c (lines 166-182)
void
on_key_input(GLFWkeyevent *ev) {
    Window *w = active_window();
    const int action = ev->action, mods = ev->mods;
    const uint32_t key = ev->key, native_key = ev->native_key;
    const char *text = ev->text ? ev->text : "";
    ...
    if (!w) { debug("no active window, ignoring\n"); return; }
    ...
}

The active_window() function in keys.c determines which window receives input:

// From kitty/keys.c (lines 106-111)
static Window*
active_window(void) {
    Tab *t = global_state.callback_os_window->tabs + global_state.callback_os_window->active_tab;
    Window *w = t->windows + t->active_window;
    if (w->render_data.screen) return w;
    return NULL;
}

This shows the focus hierarchy: the global_state tracks the current OS window, which contains tabs, and each tab has an active_window index. The input is routed to tabs[active_tab].windows[active_window].

Focus changes are propagated through GLFW's window_focus_callback in glfw.c:

// From kitty/glfw.c (lines 514-517)
static void
window_focus_callback(GLFWwindow *w, int focused) {
    if (!set_callback_window(w)) return;
    debug_input("\x1b[35mon_focus_change\x1b[m: window id: 0x%llu focused: %d\n", ...);

When a window loses focus or is closed, any pending input is simply discarded. The on_key_input function checks active_window() at line 182, and if it returns NULL (no active window or window closed), the input is ignored with the debug message "no active window, ignoring". I observed this behavior in the debug logs - there is no queuing of input for unfocused windows; it's immediately dropped.

From the runtime artifacts, I can infer the Python/C/GLFW boundaries:

1. GLFW (external library): Handles X11/Wayland communication, receives raw key events via XKB, manages window focus. This is evident from the "xkb_keycode" in debug output and the poll() calls on fd 5 (the X11 connection).

2. C layer (kitty/keys.c, kitty/child-monitor.c): The on_key_input function processes keys, encodes them, and schedules writes via schedule_write_to_child. The ChildMonitor runs in a separate thread and performs actual writes to the pty.

3. Python layer (kitty/main.py, kitty/boss.py): The main event loop runs in Python (_run_app), and shortcut handling goes through Python (dispatch_possible_special_key is called via PyObject_CallMethod in keys.c).

I can rule out two plausible but incorrect interpretations:

First incorrect interpretation: "Python handles the primary input dispatch." The strace timing evidence disproves this. Looking at the timestamps:
- 04:52:12 on_key_input is called
- 04:52:12 (same millisecond) write(8) signals ChildMonitor
- 04:52:12 (same millisecond) write(10, "y") goes to pty

If Python were in the primary dispatch path for regular text input, there would be GIL acquisition overhead and Python bytecode execution visible in the timing. Instead, the C code in on_key_input directly calls schedule_write_to_child without going through Python for normal text keys. Python is only invoked for potential shortcuts via dispatch_possible_special_key, and only if that returns "not consumed" does the key proceed to the child.

Second incorrect interpretation: "GLFW directly writes to the child pty." The multi-thread strace proves this wrong. GLFW runs in the main thread (pid 8353), but the actual pty write happens in thread 8389 (ChildMonitor). GLFW only delivers the key event; it has no knowledge of child processes or ptys. The C code in kitty acts as the bridge, queuing data and signaling the ChildMonitor thread.

Regarding correctness-versus-responsiveness tradeoffs, I observed mutex locking in the write path. Looking at child-monitor.c:

// From kitty/child-monitor.c (lines 74-79, 336-365)
#define screen_mutex(op, which) \
    pthread_mutex_##op(&screen->which##_buf_lock);

// In schedule_write_to_child:
screen_mutex(lock, write);
... memcpy to write_buf ...
if (screen->write_buf_used) wakeup_io_loop(self, false);
screen_mutex(unlock, write);

This locking mechanism is a synchronous correctness guarantee at the cost of responsiveness. The main thread acquires the write_buf_lock mutex before adding data to the buffer, then releases it. The ChildMonitor thread must acquire the same lock before reading the buffer. 

The timing evidence supports this: in the strace, the write(8) to signal the ChildMonitor and the subsequent write(10) to the pty both occur within the same millisecond (04:52:12). This low latency (sub-millisecond) suggests the locking overhead is minimal, but it's still a synchronous lock that could cause the main thread to block if the ChildMonitor is holding the lock during a slow pty write. An alternative lock-free design using a lock-free queue would trade correctness guarantees (like memory ordering) for potentially better responsiveness under contention, but Kitty chose the mutex approach for predictable behavior.

Additionally, there's a buffer size tradeoff visible in the code: the write buffer can grow up to 100MB before input is discarded ("Too much data being sent to child...ignoring it"). This queuing behavior prioritizes eventual delivery over immediate backpressure, which improves responsiveness for burst input but could delay feedback if the child is slow.
