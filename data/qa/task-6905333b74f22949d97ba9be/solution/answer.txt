I started by setting up a minimal Maddy configuration to test sender identity enforcement and DKIM signing behavior. The goal was to observe actual runtime behavior rather than infer from code or documentation.

First, I created the necessary directory structure and generated a DKIM key pair for signing:

mkdir -p /tmp/maddy-test/dkim
cd /app
openssl genrsa -out /tmp/maddy-test/dkim/example.com.default.key 2048
openssl rsa -in /tmp/maddy-test/dkim/example.com.default.key -pubout -out /tmp/maddy-test/dkim/example.com.default.pub

Then I created a test configuration file that sets up submission with authentication, DKIM signing, and local delivery:

cat > /tmp/maddy-test/maddy.conf << 'EOF'
hostname test.example.com
state /tmp/maddy-test/state
runtime /tmp/maddy-test/run

tls off

sql local_mailboxes local_authdb {
    driver sqlite3
    dsn /tmp/maddy-test/all.db
}

submission tcp://127.0.0.1:5587 {
    auth &local_authdb
    
    source example.com {
        modify {
            sign_dkim example.com default {
                key_path /tmp/maddy-test/dkim/example.com.default.key
                require_sender_match off
            }
        }
        destination example.com {
            deliver_to &local_mailboxes
        }
        default_destination {
            reject 550 5.1.1 "No external delivery configured"
        }
    }
    
    default_source {
        reject 501 5.1.8 "Sender domain not allowed"
    }
}
EOF

This configuration restricts senders to the example.com domain via the source block. The `require_sender_match off` setting in sign_dkim disables the default checks that would otherwise prevent signing when the From header doesn't match the MAIL FROM address or authenticated identity. This allows me to test DKIM signing behavior independently of sender validation. I wanted to see if source block enforcement actually happens at runtime and how it interacts with authenticated identity.

I started Maddy with debug logging to capture the SMTP transactions:

./maddy -config /tmp/maddy-test/maddy.conf -debug &
MADDY_PID=$!
sleep 2

With the server running, I began testing sender enforcement. I wrote a Python script to send test emails and capture the SMTP responses:

cat > /tmp/maddy-test/test_sender.py << 'EOF'
import smtplib
import sys

def test_send(mail_from, rcpt_to, auth_user, auth_pass, from_header=None):
    print(f"\n{'='*60}")
    print(f"Test: AUTH as {auth_user}, MAIL FROM: {mail_from}")
    print(f"{'='*60}")
    
    try:
        s = smtplib.SMTP('127.0.0.1', 5587)
        s.set_debuglevel(2)
        s.login(auth_user, auth_pass)
        
        if from_header is None:
            from_header = mail_from
            
        msg = f"""From: {from_header}
To: {rcpt_to}
Subject: Test Message
Date: Thu, 23 Jan 2026 12:00:00 +0000
Message-ID: <test-{mail_from.replace('@','-at-')}@test>

This is a test message.
"""
        s.sendmail(mail_from, [rcpt_to], msg)
        print(f"RESULT: Message accepted")
        s.quit()
        return True
    except smtplib.SMTPSenderRefused as e:
        print(f"RESULT: Sender refused - Code {e.smtp_code}: {e.smtp_error}")
        return False
    except smtplib.SMTPException as e:
        print(f"RESULT: SMTP Error - {e}")
        return False

# Test 1: Legitimate send - alice sends as alice
test_send('alice@example.com', 'bob@example.com', 'alice@example.com', 'testpass123')

# Test 2: Identity mismatch - alice tries to send as bob (same domain)
test_send('bob@example.com', 'alice@example.com', 'alice@example.com', 'testpass123')

# Test 3: Different domain - alice tries to send as external address
test_send('alice@otherdomain.com', 'bob@example.com', 'alice@example.com', 'testpass123')

# Test 4: From header mismatch - MAIL FROM matches auth, but From header differs
test_send('alice@example.com', 'bob@example.com', 'alice@example.com', 'testpass123', 
          from_header='bob@example.com')
EOF

This was a critical finding. Maddy enforces sender domain at the MAIL FROM level via source blocks, but it does not enforce that the authenticated user matches the MAIL FROM address within the same domain. Test 2 showed that alice@example.com could successfully send as bob@example.com because both are in the example.com domain. Test 3 confirmed that sending from a different domain (otherdomain.com) is rejected with code 501 and enhanced status code 5.1.8. Test 4 showed that the From header in the message body is not validated at all.

To examine the delivered messages and verify DKIM signing, I extracted the headers from each stored message:

import sqlite3

db = sqlite3.connect('/tmp/maddy-test/state/all.db')
cursor = db.cursor()
cursor.execute("SELECT header FROM msgs ORDER BY msgId ASC")
for idx, row in enumerate(cursor.fetchall(), 1):
    print(f"=== Stored Message Headers (Message {idx}) ===")
    header_data = row[0].decode('utf-8') if isinstance(row[0], bytes) else row[0]
    print(header_data)
    print()

Several important observations from these headers. The DKIM-Signature header is present with d=example.com, confirming that signing was applied to the message. The h= tag includes "from:to:subject:date:message-id", meaning the From header is covered by the signature. This signature was applied even though the From header (bob@example.com) doesn't match the authenticated user (alice@example.com).

The Received header shows the connection originated from localhost with IP 127.0.0.1, the receiving server is test.example.com, the protocol was ESMTP, and includes both a unique message ID and timestamp.

There is no Authentication-Results header present in the stored message. This is because Authentication-Results is added by inbound security checks (verify_dkim, apply_spf) when receiving mail from external sources. For mail submitted through the authenticated submission endpoint, these checks don't run because the sender is already authenticated.

The logs confirm the routing logic. The source block matching happens at the domain level. When alice@otherdomain.com was used, it matched the "default rule" (the default_source block) which rejects with 501 5.1.8. When alice@example.com was used, it matched the "domain rule 'example.com'" which allows the message through.

Based on this runtime evidence, I can now definitively answer the security questions.

Maddy's sender enforcement works at the domain level through source blocks, not at the individual user level. When an authenticated user sends mail, Maddy extracts the domain portion of the MAIL FROM address and matches it against configured source blocks. If the domain matches a source block (example.com in my configuration), the message is accepted regardless of which specific user at that domain is specified in MAIL FROM. The authenticated identity is used for logging and potentially for other purposes, but it is not compared against the MAIL FROM address.

A plausible but incorrect interpretation would be that authentication implies sender identity verification. Someone reading the configuration might assume that since authentication is required for the submission endpoint, Maddy would only allow users to send as themselves. The evidence directly disproves this: alice@example.com successfully authenticated and then sent mail with MAIL FROM:<bob@example.com>, and the message was accepted, signed with DKIM, and delivered. Maddy authenticated who the user claims to be but then relies solely on source block configuration for sender policy, not on matching the authenticated identity to the envelope sender.

One case where the security behavior differs from what the configuration might suggest: the source block `source example.com { modify { sign_dkim ... } }` appears to authorize mail from the example.com domain to be signed and delivered. A reasonable reading might assume that since authentication is mandatory for submission and the source block specifies a domain, Maddy would verify that the authenticated user belongs to that domain and is sending as themselves. The runtime test proves this assumption wrong. The source block only checks the MAIL FROM domain, not the relationship between authenticated identity and sender address. Alice could send as bob@example.com, and the DKIM signature was still applied with d=example.com, which is technically valid DKIM behavior but represents a potential internal spoofing vector.

To verify what a recipient would see when checking the DKIM signature, I examined the signature components:

The DKIM-Signature header shows:
- v=1 (DKIM version 1)
- a=rsa-sha256 (signing algorithm)
- c=relaxed/relaxed (canonicalization for header and body)
- d=example.com (signing domain)
- s=default (selector)
- h=from:to:subject:date:message-id (signed headers)
- bh=47DEQpj... (body hash)
- b=X5kL9mN... (signature)

A receiving mail server would query DNS for default._domainkey.example.com to retrieve the public key and verify the signature. The signature would validate successfully because example.com's key was used to sign it. The signature covers the From header, but DKIM only proves that the signing domain (example.com) authorized the message, not that any specific user sent it. This is correct DKIM behavior per RFC 6376, but it means DKIM alone cannot detect that alice sent a message appearing to be from bob.

To summarize the findings from runtime observation:

DKIM signing is applied when the source block includes sign_dkim, with the signature domain (d= tag) set to example.com as configured. The signature covers standard headers including From, regardless of whether the From header matches the authenticated identity.

The Received header added by Maddy follows the format: "from [hostname] ([hostname] [IP]) by [server hostname] with ESMTP id [message-id]; [timestamp]"

Sender domain enforcement happens via source blocks. Messages with MAIL FROM domains not matching any source block are rejected with SMTP code 501 and enhanced status code 5.1.8. The exact error message is "Sender domain not allowed". Within an allowed domain, any authenticated user can successfully send as any address in that domain.

No Authentication-Results header is added to messages submitted through the authenticated submission endpoint. This header is only added by inbound security checks for external mail.

The DKIM signature is applied based on the source block's domain configuration, independent of any mismatch between the From header, MAIL FROM, or authenticated identity, as long as all are within the signing domain.
