I started by setting up a test environment to observe how SFTPGo handles concurrent sessions, quota enforcement, and atomic uploads under pressure. I built the server from source and created a test configuration with strict limits: max_sessions set to 2, quota_files set to 5, quota_size set to 50000 bytes, idle_timeout at 1 minute, and upload_mode set to 1 for atomic uploads.

```
$ cd /app && go build -o sftpgo_test

$ cat /tmp/sftpgo_test/config/sftpgo.json
{
  "sftpd": {
    "bind_port": 2222,
    "idle_timeout": 1,
    "max_auth_tries": 3,
    "upload_mode": 1,
    ...
  },
  "data_provider": {
    "track_quota": 1,
    ...
  }
}
```

I created a test user via the REST API with the tight constraints to force contention scenarios.

```
$ python3 -c "import requests; r = requests.post('http://127.0.0.1:8090/api/v1/user', json={'username': 'testuser', 'password': 'testpass123', 'status': 1, 'home_dir': '/tmp/sftpgo_test/users/testuser', 'max_sessions': 2, 'quota_size': 50000, 'quota_files': 5, 'permissions': {'/': ['*']}}); print(r.status_code, r.text)"

Output:
200 {"id":1,"status":1,"username":"testuser",...,"max_sessions":2,"quota_size":50000,"quota_files":5,...}
```

For session limit enforcement, I attempted to open 4 concurrent sessions when the limit was 2. The first two sessions were accepted and the subsequent two were rejected. The server logs show the coordination happening through a global connection map protected by a mutex.

```
$ python3 test_concurrent_sessions.py

[09:59:47.762] TEST 1: Session Limit Enforcement
[09:59:47.762] Attempting to open 4 sessions (limit is 2)...
[09:59:48.056]   Session 1: ACCEPTED (total active: 1)
[09:59:48.559]   Session 2: ACCEPTED (total active: 2)
[09:59:49.775]   Session 3: REJECTED - Authentication failed.
[09:59:50.057]   Session 4: REJECTED - Authentication failed.

Session limit test results:
  Accepted: 2
  Rejected: 2
```

The server logs reveal where this decision manifests. The loginUser function in server.go calls getActiveSessions which iterates through openConnections under a read lock. When the count reaches max_sessions, authentication is refused before the connection is added to the map.

```
Log entries:
{"level":"debug","sender":"sftpd","message":"authentication refused for user: \"testuser\", too many open sessions: 2/2"}
{"level":"debug","sender":"sftpd","message":"authentication refused for user: \"testuser\", too many open sessions: 2/2"}
```

The session coordination uses a sync.RWMutex protecting the openConnections map. The getActiveSessions function acquires a read lock, counts matching usernames, and returns. This serializes the session count check but allows multiple concurrent reads while blocking writes.

The 6th upload was rejected because the file count limit was hit. The quota check happens in the hasSpace function in handler.go before the transfer begins. The server log shows the exact decision point with quota values.

```
{"level":"debug","sender":"sqlite","message":"quota updated for user \"testuser\", files increment: 1 size increment: 8000 is reset? false"}
{"level":"debug","sender":"sqlite","message":"quota updated for user \"testuser\", files increment: 1 size increment: 8000 is reset? false"}
{"level":"debug","sender":"sqlite","message":"quota updated for user \"testuser\", files increment: 1 size increment: 15000 is reset? false"}
{"level":"debug","sender":"sqlite","message":"quota updated for user \"testuser\", files increment: 1 size increment: 15000 is reset? false"}
{"level":"debug","sender":"sqlite","message":"quota updated for user \"testuser\", files increment: 1 size increment: 10000 is reset? false"}
{"level":"debug","sender":"sftpd","message":"quota exceed for user \"testuser\", num files: 5/5, size: 56000/50000 check files: true"}
{"level":"info","sender":"sftpd","message":"denying file write due to space limit"}
```

The quota check happens at transfer initiation, not completion. The hasSpace function calls GetUsedQuota which queries the database directly. Each quota update goes through UpdateUserQuota which is called after transfer completion. This means quota updates serialize through SQLite but the check and update are not atomic, which is why uploads exceeding the size limit (56000 > 50000) were allowed since each individual file fit when it started.

For atomic upload behavior during connection interruption, I tested what happens when an upload is forcibly killed mid-stream. The server is configured with upload_mode=1 which writes to a temporary file first, then renames on completion.

```
[09:59:59.610] Starting large file upload that will be interrupted...
[10:00:00.189]   Main thread: Upload has sent 6144 bytes so far
[10:00:00.189]   Files during upload: [('.sftpgo-upload.d5pkd7q4vuadilcs90q0.atomic_test.txt', 5120)]
[10:00:00.189]   Main thread: Forcibly closing transport...
[10:00:00.193]   Upload thread: Upload interrupted - Socket is closed

Upload result: interrupted
  Bytes sent before interruption: 6144
  Error: Socket is closed
```

The key observation is that during the upload, a temporary file with the prefix .sftpgo-upload appears in the user's home directory. The file contained 5120 bytes at the moment of observation, showing the partial write in progress. After interruption, the server detected the EOF error and deleted the temporary file.

```
[10:00:02.194] Files in user home after interruption: []
[10:00:02.194]   No files in home directory (temp file was cleaned up)

[10:00:02.202] Quota after atomic upload interruption:
[10:00:02.202]   Used size: 0 bytes (change: 0)
[10:00:02.202]   Used files: 0 files (change: 0)
```

The server logs confirm the cleanup behavior. When the transfer encounters an error, the Close function in transfer.go checks if upload_mode is atomic and if there was an error. If so, it deletes the temporary file rather than renaming it to the target path.

```
{"level":"warn","sender":"sftpd","message":"Unexpected error for transfer, path: \"/tmp/sftpgo_test/users/testuser/atomic_test.txt\", error: \"EOF\" bytes sent: 0, bytes received: 5120 transfer running since 355 ms"}
{"level":"warn","sender":"sftpd","message":"atomic upload completed with error: \"EOF\", delete temporary file: \"/tmp/sftpgo_test/users/testuser/.sftpgo-upload.d5pkd7q4vuadilcs90q0.atomic_test.txt\", deletion error: <nil>"}
{"level":"warn","sender":"sftpd","message":"transfer error: EOF, path: \"/tmp/sftpgo_test/users/testuser/atomic_test.txt\""}
```

The quota was not incremented because the transfer never completed successfully. The UpdateUserQuota call happens in the Close function only when transferError is nil and there are actual bytes received or files created. When there is an error and the temporary file is deleted, numFiles is decremented and bytesReceived is set to 0, so no quota update occurs.

For idle timeout behavior, I ran a test to observe what happens when the periodic idle checker terminates a connection. The server is configured with idle_timeout=1 minute, which defines how long a connection can be idle before being considered for termination. The idle connection checker runs on a 5-minute tick cycle, so I needed to wait for that tick to observe the actual termination.

```
$ python3 test_idle_timeout.py

[12:48:39.432] Server idle_timeout=1 minute, but check runs every 5 minutes
[12:48:39.664] Connection established
[12:48:39.666] Initial operation done, now going idle...
[12:48:39.669] [0m 0s] Connection still active
[12:49:09.670] [0m 30s] Connection still active
```

The connection was terminated at exactly the 5-minute mark when the idle checker tick fired. The server logs show the specific idle timeout termination message with the actual idle duration.

```
{"level":"debug","sender":"sftpd","message":"idle connections check ticker 2026-01-23 12:53:29.612902684 +0000 UTC m=+600.031832980"}
{"level":"info","sender":"sftpd","connection_id":"c9df4530ba50a8c79225a843398f6db1171d5f2e873067e1d43c9ed7cb5bf270","message":"close idle connection, idle time: 4m49.970372491s, close error: <nil>"}
{"level":"debug","sender":"sftpd","message":"check idle connections ended"}
```

The CheckIdleConnections function in sftpd.go iterates through all open connections, calculates each connection's idle time based on lastActivity, and closes any connection where idle time exceeds the configured idleTimeout. The close error of nil indicates successful termination. The idle time of 4m49s reflects the time since the last SFTP operation when the check ran.

The log analysis across all tests shows the coordination pattern counts.

```
Log pattern counts:
  authentication: 6
  connection added: 16
  connection removed: 15
  quota: 14
  idle: 3
  atomic: 10
  upload: 17
  transfer: 4
  error: 31
  too many: 6
  rename: 6
  delete: 2
```

The "too many" count of 6 indicates 6 session rejection events total (2 from test 1 plus subsequent cleanup tests). The "atomic" count of 10 shows atomic upload operations. The "rename" count of 6 shows successful atomic renames, while "delete" count of 2 shows temporary files deleted due to errors.

The coordination between subsystems works as follows. Session management uses a global mutex-protected map, checking counts before authentication completes. Quota tracking serializes through the database provider layer, with the check happening at transfer start and the update at transfer end. The transfer handler uses a per-transfer mutex for byte counters and status. The atomic upload mode uses temporary files with unique names generated by the xid library, ensuring no collisions between concurrent uploads.

The key observable difference between a clean run and one with contention is that clean runs show only successful renames in the log, while contention causes "authentication refused" messages for session limits, "quota exceed" and "denying file write due to space limit" messages for quota limits, and "delete temporary file" messages when uploads are interrupted. The concrete signs in the file system are the presence or absence of .sftpgo-upload prefixed files during active transfers, and quota values in the database that either increment normally or stay unchanged when errors occur.
