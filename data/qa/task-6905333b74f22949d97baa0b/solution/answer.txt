I’ll start with the request to investigate how the secret detection pipeline behaves when the scan is launched via the CLI versus when it is launched via a minimal Go program using the API, and to explain why the two can return different counts at runtime.

I first ran both the CLI and a minimal API program against the same directory to see whether the pipelines themselves behave differently. The directory used was `/app/test_scan_dir`, which is part of the repository. For the CLI path, I ran the filesystem command using the project’s main entrypoint so that the exact same code path users rely on is exercised.

The command I ran in the repository was:

```bash
cd /app
go run trufflehog.go filesystem /app/test_scan_dir --json --no-update --log-level=2
```

At the end of the run, the CLI printed the standard summary produced in `trufflehog.go`:

```text
chunks=5 bytes=164 verified_secrets=0 unverified_secrets=0
```

I then ran the minimal API program that lives in the same repo, `minimal_scan.go`. This program constructs an `engine.Config`, initializes a `SourceManager`, and calls `ScanFileSystem` on the same directory.

The relevant code that builds and runs the scan is:

```go
cfg := engine.Config{
    Concurrency: 1,
    Detectors:   defaults.DefaultDetectors(),
    Verify:      true,
    Dispatcher:  engine.NewPrinterDispatcher(new(output.JSONPrinter)),
}

const defaultOutputBufferSize = 64
opts := []func(*sources.SourceManager){
    sources.WithConcurrentSources(cfg.Concurrency),
    sources.WithConcurrentUnits(cfg.Concurrency),
    sources.WithSourceUnits(),
    sources.WithBufferedOutput(defaultOutputBufferSize),
}
cfg.SourceManager = sources.NewManager(opts...)

filesystemCfg := sources.FilesystemConfig{
    Paths: []string{"/app/test_scan_dir"},
}
ref, err := eng.ScanFileSystem(ctx, filesystemCfg)
```

When this program is run and allowed to finish normally, and when results are not drained manually from the engine’s internal channel, the metrics reported by `eng.GetMetrics()` after `Finish()` match the CLI exactly: 5 chunks, 164 bytes, and zero secrets found. This tells me that the underlying detection pipeline is the same in both cases, and that there is no separate “CLI engine” versus “API engine”.

Next, I identified the component that determines which detectors participate in a scan and how their configuration is derived.

I traced this by following engine initialization rather than worker or scanner code. The single place where the detector set is finalized is `engine.NewEngine` in `pkg/engine/engine.go`. That function takes an `engine.Config` and applies defaults and filters before any scanning starts.

The configuration struct that controls detector participation is defined here:

```go
type Config struct {
    Concurrency int

    Decoders  []decoders.Decoder
    Detectors []detectors.Detector

    DetectorVerificationOverrides map[config.DetectorID]bool
    IncludeDetectors              string
    ExcludeDetectors              string
    CustomVerifiersOnly           bool
    VerifierEndpoints             map[string]string

    Verify            bool
    Results           map[string]struct{}
    FilterEntropy     float64
    FilterUnverified  bool
    ShouldScanEntireChunk bool

    Dispatcher    ResultsDispatcher
    SourceManager *sources.SourceManager
    ...
}
```

Inside `engine.NewEngine`, the engine copies the detector slice from `cfg.Detectors`, applies defaults with `setDefaults`, then builds include and exclude sets using `buildDetectorSets(cfg)`. Those include and exclude strings are parsed by `config.ParseDetectors`, and the resulting sets are applied via `engine.applyFilters`.

The base detector list that both CLI and API start from comes from:

```go
func DefaultDetectors() []detectors.Detector {
    detectorList := buildDetectorList()
    ...
    return detectorList
}
```

So, at runtime, the detector set is always:

defaults.DefaultDetectors(), plus any detectors explicitly added to `Config.Detectors`, filtered by `IncludeDetectors` and `ExcludeDetectors` inside `engine.NewEngine`.

The next request in the prompt is to explain what conditions, configuration values, environment variables, or defaults lead to the CLI and the API producing different detector sets or behavior at runtime.

I looked at how the CLI populates `engine.Config` in `trufflehog.go`, specifically in the `runSingleScan` path. The CLI builds the config like this:

```go
engConf := engine.Config{
    Concurrency: *concurrency,
    Detectors:   append(defaults.DefaultDetectors(), conf.Detectors...),
    Verify:      !*noVerification,
    IncludeDetectors: *includeDetectors,
    ExcludeDetectors: *excludeDetectors,
    FilterUnverified: *filterUnverified,
    FilterEntropy:    *filterEntropy,
    Results:          parsedResults,
    Dispatcher:       engine.NewPrinterDispatcher(printer),
}
```

In contrast, the minimal API program constructs a much smaller config:

```go
cfg := engine.Config{
    Concurrency: 1,
    Detectors:   defaults.DefaultDetectors(),
    Verify:      true,
    Dispatcher:  engine.NewPrinterDispatcher(new(output.JSONPrinter)),
}
```

From this comparison, I can see three concrete configuration differences that directly affect behavior and counts:

First, the CLI may append extra detectors from a YAML config file (`conf.Detectors`), while the API program never does. That alone can cause the CLI to run more detectors.

Second, the CLI always honors `IncludeDetectors` and `ExcludeDetectors` flags, which are parsed and applied during engine initialization. If those flags are used, the API program must set the same fields explicitly or it will run a different detector subset.

Third, the CLI applies result filtering and verification-related options (`Results`, `FilterUnverified`, `FilterEntropy`) that decide which matches are counted as findings. The API program leaves these at zero values, which changes what ultimately gets counted.

Finally, the prompt asks why the CLI filesystem scan reports more findings than an API-based scan at runtime. In the minimal program in this repo, the biggest concrete cause is how results are consumed.

The engine exposes an internal channel via `ResultsChan()`, but that channel is intended to be consumed only by the engine’s notifier workers. Those workers deduplicate results, update `VerifiedSecretsFound` and `UnverifiedSecretsFound`, and dispatch output.

In `minimal_scan.go`, the program does this instead:

```go
for range eng.ResultsChan() {
    foundSecretsCount++
}
```

By doing this, the API program races with the notifier workers and steals results off the channel before the engine can process them normally. That means some results never update metrics and never reach the dispatcher, which makes counts diverge from the CLI. The CLI never reads from `ResultsChan` directly; it relies entirely on the notifier pipeline.

When I remove that loop and instead read counts from `eng.GetMetrics()` after calling `Finish()`, the API and CLI produce the same results for the same directory on this commit.

That is the concrete, runtime reason the CLI and API can report different numbers of findings, even though they share the same detection engine.
