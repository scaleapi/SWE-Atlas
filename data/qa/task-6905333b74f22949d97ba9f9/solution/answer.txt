I started my investigation by attempting to run the main entry point directly to see what would break. Running python3 __main__.py immediately failed with this traceback:

Traceback (most recent call last):
  File "/app/__main__.py", line 7, in <module>
    main()
  File "/app/kitty/entry_points.py", line 194, in main
    from kitty.main import main as kitty_main
  File "/app/kitty/main.py", line 11, in <module>
    from .borders import load_borders_program
  File "/app/kitty/borders.py", line 7, in <module>
    from .fast_data_types import BORDERS_PROGRAM, add_borders_rect, get_options, init_borders_program, os_window_has_background_image
ModuleNotFoundError: No module named 'kitty.fast_data_types'

This traceback reveals the dependency chain: the entry point calls into kitty.main, which imports borders, which immediately tries to load from fast_data_types. The failure happens at the very first import that touches the native layer, showing how deeply wired Python is into the C extension.

The fast_data_types module is a compiled C extension that must be built before kitty can function. Without it, the Python layer is essentially inert.

To understand how deep this dependency runs, I tested which Python modules could survive without the native bridge:

```python
# test_imports.py
modules = ['kitty.types', 'kitty.constants', 'kitty.rgb', 'kitty.cli', 'kitty.utils']
for mod in modules:
    try:
        __import__(mod)
        print(f"OK: {mod}")
    except:
        print(f"FAILS: {mod}")
```

```bash
$ python3 test_imports.py
```

Output:
OK: kitty.types
OK: kitty.constants
FAILS: kitty.rgb
FAILS: kitty.cli
FAILS: kitty.utils

Only 2 modules can be imported without the compiled C extension - kitty.types and kitty.constants. Everything else depends on fast_data_types, revealing that Python cannot stand alone in this architecture.

To answer which language does the heavy lifting, I analyzed the codebase composition by counting files and lines:

```bash
$ find kitty/ -name "*.c" | wc -l
```

Output: 51

```bash
$ wc -l kitty/screen.c
```

Output: 4932 kitty/screen.c

The kitty directory contains 51 C source files, including 2 in the kitty/launcher/ subdirectory. The largest is screen.c at 4,932 lines, which handles the terminal's screen buffer, cursor management, and VT escape sequence parsing. This is where the actual terminal emulation happens - the performance-critical hot path runs entirely in C.

The GLSL shader files revealed why kitty calls itself GPU accelerated:

```bash
$ ls kitty/*.glsl | wc -l
```

Output: 13

```bash
$ wc -l kitty/cell_vertex.glsl kitty/cell_fragment.glsl
```

Output:
233 kitty/cell_vertex.glsl
204 kitty/cell_fragment.glsl
437 total

There are 13 GLSL shader files. The core rendering happens in cell_vertex.glsl (233 lines) and cell_fragment.glsl (204 lines). The vertex shader positions every cell on screen, resolves colors from a 256-entry color table, and handles cursor position and selection state. The fragment shader performs multi-pass compositing with gamma-correct alpha blending.

The rendering pipeline works in passes: first backgrounds, then images below text, special cells like cursor, foreground text, and finally images above text. This is why kitty can have transparent backgrounds with images behind text - it's a genuine GPU compositing pipeline, not marketing fluff.

For the kittens directory, I examined what actually lives there:

```bash
$ ls -d kittens/*/ | wc -l
```

Output: 20

There are 20 kitten directories total. To understand if they're truly independent, I checked which ones have both Python and Go implementations:

```bash
$ for k in kittens/*/; do
    test -f "$k/main.py" && test -f "$k/main.go" && basename $k
done | wc -l
```

Output: 14

14 of the 20 kitten directories are hybrid, having both Python main.py and Go main.go files. The remaining directories include the tui framework and pure-Python kittens like broadcast and panel. The Go code compiles into a standalone kitten binary:

```bash
$ find tools/ -name "*.go" | wc -l
```

Output: 193

```bash
$ find kittens/ -name "*.go" | wc -l
```

Output: 64

There are 193 Go files in tools/ and 64 in kittens/. When you run kitten icat from a built installation, you get the Go version which communicates via terminal escape sequences. The Python files in kittens are mostly CLI option definitions and documentation stubs.

Attempting to run a kitten directly through Python:

```bash
$ python3 -m kittens.icat
```

Output: /usr/bin/python3: No module named kittens.icat.__main__; 'kittens.icat' is a package and cannot be directly executed

This confirms the kittens aren't designed to run as standalone Python modules. They either need the fast_data_types bridge when invoked through kitty, or they run as compiled Go binaries that communicate via escape sequences.

The architecture breaks down into four layers:

Python sits at the top handling orchestration - configuration parsing, window and tab management, the CLI interface, and remote control protocol. It never touches the hot path.

The fast_data_types C extension forms the critical bridge. Those 51 C source files implement terminal emulation in screen.c, font rendering via FreeType, OpenGL shader compilation, child process management, and GLFW window bindings. Without this compiled module, Python has nothing to orchestrate.

Below that, 13 GLSL shaders run on the GPU. The 233-line cell_vertex.glsl and 204-line cell_fragment.glsl handle all text rendering with proper color space handling and gamma correction. Every character you see passes through these shaders.

The Go layer provides standalone tooling - 193 files in tools/ compile into the kitten binary for SSH integration, image display, file transfer, and other utilities that can work independently of the Python runtime.

The immediate crash when running directly reveals the tight coupling: this is not a system where Python happens to call some C for speed. The C extension is the foundation, Python is the glue, and the GPU does all the actual drawing. The modular-looking kittens are either compiled Go (truly independent) or Python code that needs fast_data_types just like everything else.
