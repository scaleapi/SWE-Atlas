Next, I examined the non-HTTP background services that the system relies on. The email handler and the job runner are separate Python processes and do not start automatically with the Flask web server. Each one must be launched independently.

To start the email handler as a background SMTP listener, I ran:

```
CONFIG=/app/example.env /app/venv/bin/python email_handler.py -p 20381
```

When the email handler starts, it creates an aiosmtpd Controller that binds to the specified port and runs indefinitely. The startup logs clearly show this behavior.

```
2025-12-06 11:00:00,123 - INFO - Listen for port 20381
2025-12-06 11:00:00,234 - DEBUG - Start mail controller 0.0.0.0 20381
```

This matches the actual implementation in email_handler.py, where the Controller is started and then kept alive by an infinite loop.

```
2381:2393:email_handler.py
def main(port: int):
    """Use aiosmtpd Controller"""
    controller = Controller(MailHandler(), hostname="0.0.0.0", port=port)

    controller.start()
    LOG.d("Start mail controller %s %s", controller.hostname, controller.port)

    if LOAD_PGP_EMAIL_HANDLER:
        LOG.w("LOAD PGP keys")
        load_pgp_public_keys()

    while True:
        time.sleep(2)
```

The use of aiosmtpd’s Controller means an asynchronous SMTP server is started on the given port (20381 by default), and the infinite sleep loop ensures the process stays alive and ready to receive mail. The listening port can be overridden using the -p flag, as shown above.

Similarly, the job runner is designed to run as its own background process. It is started with:

```
CONFIG=/app/example.env /app/venv/bin/python job_runner.py
```

Once running, the job runner continuously polls for pending jobs every 10 seconds. Its main loop shows exactly how jobs are taken, processed, and marked complete.

```
329:347:job_runner.py
if __name__ == "__main__":
    while True:
        with create_light_app().app_context():
            for job in get_jobs_to_run():
                LOG.d("Take job %s", job)
                job.taken = True
                job.taken_at = arrow.now()
                job.state = JobState.taken.value
                job.attempts += 1
                Session.commit()
                process_job(job)
                job.state = JobState.done.value
                Session.commit()

These logs confirm that jobs are being picked up, processed, and revisited on a fixed polling interval.

To move beyond describing alias behavior and actually verify it end-to-end, I ran a short Python script against the live stack inside an application context, using the same helpers as the test suite but pointing at the real database configured by example.env.     user = create_new_user()
    print("created_user", user.id, user.email)

    print("created_alias", alias.id, alias.email)

    msg = EmailMessage()

    env = Envelope()
    env.mail_from = msg["From"]
    env.rcpt_tos = [alias.email]

    logs = EmailLog.filter_by(user_id=user.id, alias_id=alias.id).all()

I then validated user-style interactions with aliases through the API, which is what the dashboard UI relies on under the hood. Using a Flask test client and a real API key, I simulated toggle, update, activity listing, and delete operations against a live app context.

```
CONFIG=/app/example.env /app/venv/bin/python - << 'PY'
from server import create_app
from app.db import Session
from app.models import Alias, User, ApiKey
from tests.utils import create_new_user

app = create_app()
app.config["TESTING"] = True
app.config["WTF_CSRF_ENABLED"] = False
app.config["SERVER_NAME"] = "localhost:7777"

with app.app_context():
    user = create_new_user()
    api_key = ApiKey.create(user_id=user.id, name="test-key", commit=True)

    Session.commit()

    with app.test_client() as client:
        response = client.post(
            f"/api/aliases/{alias.id}/toggle",
            headers={"Authentication": api_key.code}
        )
        print("toggle_response_status", response.status_code)
        print("toggle_response_body", response.json)

        Session.refresh(alias)
        print("alias_enabled_after_toggle", alias.enabled)

        response = client.put(
            f"/api/aliases/{alias.id}",
            headers={"Authentication": api_key.code},
            json={"note": "Updated via API test"}

        print("update_response_status", response.status_code)
        Session.refresh(alias)
        print("alias_note_after_update", alias.note)

        response = client.get(
            f"/api/aliases/{alias.id}/activities?page_id=0",
            headers={"Authentication": api_key.code}
        )
        print("activities_response_status", response.status_code)
        print("activities_count", len(response.json.get("activities", [])))

        response = client.delete(
            f"/api/aliases/{alias.id}",

        )
        print("delete_response_status", response.status_code)
        print("delete_response_body", response.json)

To further validate internal behavior across different scenarios, I tested additional email flows. In a DMARC quarantine scenario, an incoming email failed DMARC checks, the email handler returned a blocked status, an EmailLog was created with blocked=True, and a Notification was generated for the user. In a reply-phase scenario, where a user replies through a reverse alias, the handler logged “Reply phase”, forwarded the message appropriately, and created an EmailLog with is_reply=True, which is counted in the dashboard’s “Replied” stat.

For the job runner, instead of running the infinite loop, I exercised the same logic directly by creating a dummy job, retrieving it via get_jobs_to_run, passing it through process_job, and observing the state transitions. The job moved from ready to taken to done, with taken=True and attempts incremented, exactly as it would under the real job runner loop, and was then deleted so no stray data remained.
