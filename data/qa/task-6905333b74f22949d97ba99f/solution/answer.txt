I started by tracing the like button click to understand where the pending intent gets stored when a logged-out user triggers a requires-login action. The Reader logged-out like action loss is a manifestation of a fundamental state persistence gap: when a pending user action is stored only in Redux memory state without any persistence mechanism, that state is inevitably cleared when the page reloads after authentication completes. The action intent is captured before authentication starts, but the page reload that finalizes the login transition clears the in-memory state before any handler can replay it.

This shows that `withPersistence` is imported and used only on line 19 for `lastPath`. Inspecting the Redux reducer configuration in `client/state/reader-ui/reducer.js` reveals where state persistence is selectively applied. The `lastPath` reducer uses `withPersistence` (lines 19-28):

```javascript
// client/state/reader-ui/reducer.js (lines 7, 19-28)
import { combineReducers, withPersistence } from 'calypso/state/utils';

/**
 * Keep the last reader stream path selected by the user, for the purpose of autoselecting it
 * when user navigates back to Reader
 */
export const lastPath = withPersistence( ( state = null, action ) => {
	switch ( action.type ) {
		case READER_VIEW_STREAM:
			if ( action.path && action.path.startsWith( '/reader' ) ) {
				return action.path;
			}
			break;
	}
	return state;
} );
```

However, the `lastActionRequiresLogin` reducer in the same file (lines 45-54) has no persistence wrapper. I verified this by checking the reducer definition:

```bash
sed -n '45,54p' client/state/reader-ui/reducer.js
```

```
export const lastActionRequiresLogin = ( state = null, action ) => {
	switch ( action.type ) {
		case READER_REGISTER_LAST_ACTION_REQUIRES_LOGIN:
			return action.lastAction;
		case READER_CLEAR_LAST_ACTION_REQUIRES_LOGIN:
			return null;
		default:
			return state;
	}
};
```

The `lastPath` reducer is wrapped with `withPersistence`, which enables the state to survive page reloads through Calypso's persistence infrastructure. However, `lastActionRequiresLogin` is a plain reducer with no persistence wrapper. This means the pending action exists only in JavaScript memory and is lost the moment the page reloads.

```javascript
/**
 * Get reader last path selected
 * @param state redux state
 * @returns string|null {lastPath} last feed path visited in the reader

export function getLastPath( state ) {
	return state.readerUi.lastPath;

/**
 * Get last reader action that requires user to be logged in
 * @param state redux state
 * @returns string|null {loggedInAction} logged in action clicked in the reader
 */
export function getLastActionRequiresLogin( state ) {
	// Check if lastActionRequiresLogin is defined, if not return null
	if ( ! state.readerUi?.lastActionRequiresLogin ) {
		return null;
	}
	return state.readerUi?.lastActionRequiresLogin;
}
```

This selector uses optional chaining to safely access the nested state property and returns `null` if the pending action does not exist or if the `readerUi` slice is undefined.

The verification confirms the `loggedInAction` object has no `redirectTo` property and falls through to the else branch that calls `window.location.reload()`. This is the critical failure point.

The `window.location.reload()` call completely reinitializes the application. Redux state is rebuilt from scratch. Before the reload, the state contains the pending like action, but after the reload completes the state becomes `null` because the reducer initializes with `state = null` and there is no persistence mechanism to restore the previous value.

The full sequence of the failure proceeds as follows:

1. User clicks like on a post while logged out
2. `LikeButtonContainer.handleLikeToggle` detects `!isLoggedIn` and dispatches `registerLastActionRequiresLogin({ type: 'like', siteId, postId })`
3. The Redux reducer `lastActionRequiresLogin` stores this object in memory at `state.readerUi.lastActionRequiresLogin`
4. The `LayoutLoggedOut` component sees `loggedInAction` is truthy and renders `ReaderJoinConversationDialog`
5. User clicks "Create account" or "Log in", and a popup window opens to wordpress.com authentication
6. User completes signup or login in the popup, which sends a `postMessage` with `{ service: 'wordpress' }` to the parent window
7. The `useLoginWindow` hook receives the message and calls `onLoginSuccess()`
8. In `logged-out.jsx`, `onLoginSuccess` executes `window.location.reload()` (line 311)
9. The page reloads, Redux store initializes fresh, and `lastActionRequiresLogin` defaults to `null`
10. No handler exists to check localStorage or any other persistence layer for pending actions
11. The like action is permanently lost

The exact condition that causes the replay path to be skipped is the absence of two required mechanisms: first, a persistence layer that bridges the authentication boundary by storing the pending action in a medium that survives page reloads (such as localStorage); second, a handler component that runs after authentication completes, retrieves the persisted action, validates it, and dispatches the appropriate Redux action to execute the user's original intent.

The `withPersistence` wrapper used by `lastPath` would not be sufficient even if applied to `lastActionRequiresLogin`, because Calypso's persistence infrastructure uses IndexedDB and is scoped to the current user context. The transition from logged-out to logged-in may involve a different persistence namespace, making the stored state inaccessible. An explicit localStorage implementation with a timestamp-based expiration would be required to reliably bridge this authentication boundary, along with a component mounted in the authenticated Reader context that checks for and replays any pending action shortly after the page loads.
