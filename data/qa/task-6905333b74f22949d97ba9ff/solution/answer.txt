To understand kitty's initialization flow, I built the terminal from source and launched it with debug flags using a virtual X11 display. This revealed how the window system, GPU context, and font rendering subsystems connect during the critical early startup phase.

The repository was already compiled, so I set up a virtual display and ran kitty with debug output enabled:

```bash
Xvfb :99 -screen 0 1280x1024x24 &
export DISPLAY=:99
export XDG_RUNTIME_DIR=/tmp/runtime-$$
mkdir -p $XDG_RUNTIME_DIR && chmod 700 $XDG_RUNTIME_DIR

timeout 10 ./kitty/launcher/kitty --debug-rendering --debug-font-fallback -e sh -c "echo 'Testing'; sleep 3" 2>&1
```

Running this produced timestamped debug output showing the initialization sequence:


Kitty selects its GLFW module based on platform and environment. I traced this programmatically:

```python
from kitty.constants import glfw_path
from kitty import fast_data_types as f

glfw_module = 'x11'  # Selected because WAYLAND_DISPLAY not set
path = glfw_path(glfw_module)
print(f"GLFW library: {path}")

result = f.glfw_init(path, edge_spacing, False, True, True)
print(f"GLFW init: {'SUCCESS' if result else 'FAILED'}")
```

This confirmed the backend selection:

```
GLFW library: /app/kitty/glfw-x11.so
GLFW init: SUCCESS
```

The selection logic in `kitty/main.py` chooses `cocoa` for macOS, `wayland` if `WAYLAND_DISPLAY` or `WAYLAND_SOCKET` is set, and falls back to `x11` otherwise. The detected OpenGL version was 4.5 Core Profile, exceeding the minimum requirement of 3.1 on Linux (3.3 on macOS) with GLSL version 140.

Crucially, the debug output identifies the renderer as llvmpipe (Mesa), which confirms that due to the Xvfb virtual environment, the terminal is relying on software rasterization (CPU) rather than hardware GPU acceleration

After GLFW initialization, kitty queries the display:

```python
physical_dpi = f.glfw_get_physical_dpi()
content_scale = f.glfw_primary_monitor_content_scale()
monitor_size = f.glfw_primary_monitor_size()
```

The display queries returned:

```
Physical DPI X: 96.19
Physical DPI Y: 96.33
Content scale X: 1.0
Content scale Y: 1.0
Monitor width: 1280px
Monitor height: 1024px
```

The logical DPI used for font rendering is computed as physical DPI multiplied by content scale, yielding 96.19 × 96.33 in this case.

Font loading depends on the DPI values. Looking at `glfw.c` line 1202:

```c
FONTS_DATA_HANDLE fonts_data = load_fonts_data(OPT(font_size), xdpi, ydpi);
PyObject *ret = PyObject_CallFunction(get_window_size, "IIddff", 
    fonts_data->cell_width, fonts_data->cell_height, 
    fonts_data->logical_dpi_x, fonts_data->logical_dpi_y, xscale, yscale);
```

The `load_fonts_data` function creates a `FontGroup` keyed by font size and DPI, loads four font faces via fontconfig, and calculates cell metrics from the primary font. These metrics, cell width, cell height, baseline, underline position are extracted from the font's internal tables scaled by DPI. The window size is then computed from the desired terminal dimensions multiplied by cell size.

Window creation happens after font metrics are known. In `create_os_window`, kitty sets OpenGL hints before creating the window:

```c
glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, OPENGL_REQUIRED_VERSION_MAJOR);
glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, OPENGL_REQUIRED_VERSION_MINOR);
glfwWindowHint(GLFW_OPENGL_FORWARD_COMPAT, true);
glfwWindowHint(GLFW_DEPTH_BITS, 0);
glfwWindowHint(GLFW_STENCIL_BITS, 0);
```

After making the context current, `gl_init()` in `gl.c` loads OpenGL functions via GLAD, verifies the version, and checks for required extensions:

```c
global_state.gl_version = gladLoadGL(glfwGetProcAddress);
if (global_state.debug_rendering) 
    printf("[%.3f] GL version string: %s\n", monotonic_t_to_s_double(monotonic()), gl_version_string());
```

Kitty then enables `GL_FRAMEBUFFER_SRGB` for automatic gamma correction and compiles shader programs for cell rendering, graphics, background images, and tinting.

The glyph cache uses a 2D array texture sized according to OpenGL limits. From `shaders.c`:

```c
glGetIntegerv(GL_MAX_TEXTURE_SIZE, &(max_texture_size));
glGetIntegerv(GL_MAX_ARRAY_TEXTURE_LAYERS, &(max_array_texture_layers));
```

With typical values of 8192 for texture size and 512 for array layers, the sprite tracker divides each layer into a grid of cell-sized slots. Glyphs are rasterized by FreeType, uploaded to sprite positions, and tracked in a hash table. The debug output confirmed DejaVuSansMono was selected with normal, bold, italic, and bold-italic variants loaded.

The subsystem dependencies form a strict chain:

1. Platform detection → selects GLFW module
2. GLFW init → connects to display server, queries default DPI
3. Display configuration → physical DPI × content scale = logical DPI
4. Font loading → uses DPI to calculate cell metrics
5. Window creation → uses cell metrics to compute pixel dimensions
6. GL context → loads OpenGL, queries texture limits
7. Shader compilation → prepares rendering programs
8. Sprite texture allocation → sized by cell metrics and GL limits
9. Child process launch → rendering infrastructure ready

This ordering explains why DPI changes (like moving to a different monitor) trigger font reloading, the entire chain from step 4 onward depends on DPI values. The timestamps in the debug output show OS window creation at 223ms, GL initialization at 183ms, and child launch at 243ms, with the complete startup taking roughly 250ms.
