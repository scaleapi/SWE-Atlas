For erasure coding behavior with multiple drives, MinIO uses a configuration where with 4 drives it applies EC:2, meaning 2 data blocks plus 2 parity blocks. This architecture allows the cluster to survive up to 2 simultaneous drive failures for reads. The default parity for a 4-drive setup is documented in docs/distributed/SIZING.md which shows stripe_size=4 has parity=2.

When a drive becomes unavailable during a write operation, MinIO returns a specific error to the client. Examining the source code in api-errors.go reveals the error handling:

```go
ErrSlowDownWrite: {
    Code:           "SlowDownWrite",
    Description:    "Resource requested is unwritable, please reduce your request rate",
    HTTPStatusCode: http.StatusServiceUnavailable,
},
```

The error code is SlowDownWrite with HTTP status 503 Service Unavailable. The client-facing error message is "Resource requested is unwritable, please reduce your request rate". The underlying server error is more specific - examining erasure-errors.go shows:

```go
var errErasureWriteQuorum = errors.New("Write failed. Insufficient number of drives online")
```

Similarly, when read quorum fails, the error code is SlowDownRead with the same HTTP 503 status:

```go
ErrSlowDownRead: {
    Code:           "SlowDownRead",
    Description:    "Resource requested is unreadable, please reduce your request rate",
    HTTPStatusCode: http.StatusServiceUnavailable,
},
```

The underlying read quorum error message from erasure-errors.go is:

```go
var errErasureReadQuorum = errors.New("Read failed. Insufficient number of drives online")
```

For files that were already stored before a drive vanished, they can still be read successfully as long as sufficient shards remain to meet the read quorum. The write quorum is calculated in the objectQuorumFromMeta function in erasure-metadata.go:

```go
writeQuorum := dataBlocks
if dataBlocks == parityBlocks {
    writeQuorum++
}
```

For a 4-drive EC:2 configuration where dataBlocks equals parityBlocks (both are 2), write quorum is 3 drives. Read quorum is calculated as setDriveCount minus parityBlocks, which equals 2 drives for this configuration.

The metric names that track drive availability in MinIO are defined in metrics-v3-cluster-health.go:

```go
healthDrivesOfflineCount = "drives_offline_count"
healthDrivesOnlineCount  = "drives_online_count"
```

These are exposed as Prometheus metrics with names like minio_cluster_drive_offline_total and minio_cluster_drive_online_total. The metrics endpoint requires authentication by default. To enable public access, set the environment variable MINIO_PROMETHEUS_AUTH_TYPE=public before starting the server (as done in the startup command above). Then query:

```bash
curl -s http://127.0.0.1:9000/minio/v2/metrics/cluster | grep drive
```

The above shows a single-drive setup with 1 drive online and 0 offline. In a 4-drive erasure-coded cluster (EC:2), the healthy baseline would show minio_cluster_drive_online_total as 4 and minio_cluster_drive_offline_total as 0. After one drive fails, the values change to 3 online and 1 offline. After the drive is restored and healing completes, the metrics return to baseline.

For the healing mechanism, the system detects newly available drives through the monitorLocalDisksAndHeal function in background-newdisks-heal-ops.go which runs periodically:

```go
defaultMonitorNewDiskInterval = time.Second * 10
```

The specific event that triggers healing is the drive returning to an online state with either a missing format.json file, stale data compared to other drives, or a healing tracker file (.healing.bin) present from a previous incomplete healing operation:

```go
healingTrackerFilename = ".healing.bin"
```

The healing process writes specific log messages that can be observed in the server output. The key log message when healing starts is:

```
Healing drive '/path/to/disk' - 'mc admin heal alias/ --verbose' to check the current status.
```

When healing completes successfully, the message pattern is:

```
Healing of drive '/path/to/disk' is finished (healed: X, skipped: Y).
```

The criteria that determine whether a specific object needs to be healed on a particular drive are implemented in the shouldHealObjectOnDisk function in erasure-healing.go:

```go
func shouldHealObjectOnDisk(erErr error, partsErrs []int, meta FileInfo, latestMeta FileInfo) (bool, error) {
    if errors.Is(erErr, errFileNotFound) || errors.Is(erErr, errFileVersionNotFound) || errors.Is(erErr, errFileCorrupt) {
        return true, erErr
    }
    if erErr == nil {
        if meta.XLV1 {
            return true, errLegacyXLMeta
        }
        if !latestMeta.Equals(meta) {
            return true, errOutdatedXLMeta
        }
        if !meta.Deleted && !meta.IsRemote() {
            for _, partErr := range partsErrs {
                if slices.Contains([]int{checkPartFileNotFound, checkPartFileCorrupt}, partErr) {
                    return true, errPartMissingOrCorrupt
                }
            }
        }
        return false, nil
    }
    return false, erErr
}
```

The function checks several conditions to determine healing necessity. First, if the error indicates the file is not found (errFileNotFound), the version is not found (errFileVersionNotFound), or the file is corrupt (errFileCorrupt), healing is immediately needed. Second, if the file exists but uses the legacy XL format (meta.XLV1), healing is required to upgrade the format, returning errLegacyXLMeta. Third, if the metadata on this drive does not match the latest metadata from other drives, the drive has outdated data and needs healing, returning errOutdatedXLMeta. Finally, even if the xl.meta metadata file is valid, the function checks each data part file and triggers healing if any part is missing (checkPartFileNotFound) or corrupt (checkPartFileCorrupt), returning errPartMissingOrCorrupt.

The heal tracking structure in background-newdisks-heal-ops.go tracks detailed progress with fields including:

```go
ItemsHealed  uint64
ItemsFailed  uint64
ItemsSkipped uint64
BytesDone    uint64
BytesFailed  uint64
BytesSkipped uint64
```
