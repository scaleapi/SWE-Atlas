I started by exploring the SimpleLogin codebase structure to trace the three multi-step runtime workflows requested.

To understand what happens when incorrect verification codes are submitted repeatedly, I first located the core verification logic in `app/mailbox_utils.py`. Running grep on the file revealed that the key constant is set to 3, with the enforcement check appearing at line 195.

```
$ grep -n "MAX_ACTIVATION_TRIES" app/mailbox_utils.py
43:MAX_ACTIVATION_TRIES = 3
195:    if activation.tries >= MAX_ACTIVATION_TRIES:
```

The system enforces exactly 3 failed attempts per activation code. Looking at the actual enforcement code (lines 185-220 in `app/mailbox_utils.py`), I traced the sequence of state changes.

```python
activation = (
    MailboxActivation.filter(MailboxActivation.mailbox_id == mailbox_id)
    .order_by(MailboxActivation.created_at.desc())
    .first()
)
```

The runtime first fetches the most recent activation record for the mailbox. Then three checks happen in order.

First, if `activation.tries >= MAX_ACTIVATION_TRIES`, the system immediately clears all activation codes via `clear_activation_codes_for_mailbox(mailbox)` and raises `CannotVerifyError("Invalid activation code. Please request another code.")`. This effectively locks the user out until they request a fresh code.

Second, there's a 15-minute expiration check: `if activation.created_at < arrow.now().shift(minutes=-15)`. Old codes are cleared and rejected the same way.

Third, if the code doesn't match (`if code != activation.code`), the tries counter increments, that is, `activation.tries = activation.tries + 1` followed by `Session.commit()`. This persists the failed attempt to the database before raising the error.

Tracing the job runner main loop (lines 329-347 in `job_runner.py`), I found that jobs are created with `state=ready` and `attempts=0`. The runner loop polls every 10 seconds, and when it picks up a job, it immediately sets:

```python
job.taken = True
job.taken_at = arrow.now()
job.state = JobState.taken.value
job.attempts += 1
Session.commit()
```

This happens before `process_job(job)` is called. If the job completes successfully, `job.state = JobState.done.value` is set. The interesting behavior is what happens on failure: there's no try/catch around `process_job()`, so if it raises an exception, the job remains in `taken` state with `attempts` already incremented.

The `get_jobs_to_run()` function (lines 307-326) shows the retry logic:

```
$ sed -n '307,326p' job_runner.py
def get_jobs_to_run() -> List[Job]:
    # Get jobs that match all conditions:
    #  - Job.state == ready OR (Job.state == taken AND Job.taken_at < now - 30 mins AND Job.attempts < 5)
    #  - Job.run_at is Null OR Job.run_at < now + 10 mins
    taken_at_earliest = arrow.now().shift(minutes=-config.JOB_TAKEN_RETRY_WAIT_MINS)
    run_at_earliest = arrow.now().shift(minutes=+10)
    query = Job.filter(
        and_(
            or_(
                Job.state == JobState.ready.value,
                and_(
                    Job.state == JobState.taken.value,
                    Job.taken_at < taken_at_earliest,
                    Job.attempts < config.JOB_MAX_ATTEMPTS,
                ),
            ),
            or_(Job.run_at.is_(None), and_(Job.run_at <= run_at_earliest)),
        )
    )
    return query.all()
```

Jobs are eligible for retry if they're in `ready` state, OR they're in `taken` state AND `taken_at < now - 30 minutes` AND `attempts < 5`. This means a failed job will be retried after 30 minutes, up to 5 total attempts.

For SyncEvent background tasks, the pattern is similar but with explicit retry counting. In `events/runner.py` (lines 20-46):

```
$ sed -n '20,46p' events/runner.py
    def __on_event(self, event: SyncEvent):
        try:
            event_created_at = event.created_at
            start_time = arrow.now()
            success = self.__sink.process(event)
            if success:
                event_id = event.id
                SyncEvent.delete(event.id, commit=True)
                LOG.info(f"Marked {event_id} as done")

                end_time = arrow.now() - start_time
                time_between_taken_and_created = start_time - event_created_at

                newrelic.agent.record_custom_metric("Custom/sync_event_processed", 1)
                newrelic.agent.record_custom_metric(
                    "Custom/sync_event_process_time", end_time.total_seconds()
                )
                newrelic.agent.record_custom_metric(
                    "Custom/sync_event_elapsed_time",
                    time_between_taken_and_created.total_seconds(),
                )
            else:
                event.retry_count = event.retry_count + 1
                Session.commit()

```

When processing fails (returns False), `retry_count` increments. The `DeadLetterEventSource` (lines 84-112 in `events/event_source.py`) picks up events older than 10 minutes with retry_count below the max threshold, giving them another chance.

The cleanup task (`tasks/cleanup_old_jobs.py`) deletes jobs that are either done, error state, or taken with attempts >= 5. This means permanently failed jobs eventually get cleaned up rather than accumulating indefinitely.

The exact format is generated in `generate_verp_email()` (lines 1438-1464 in `app/email_utils.py`):

```
$ sed -n '1438,1464p' app/email_utils.py
def generate_verp_email(
    verp_type: VerpType, object_id: int, sender_domain: Optional[str] = None
) -> str:
    """Generates an email address with the verp type, object_id and domain encoded in the address
    and signed with hmac to prevent tampering
    """
    # Encoded as a list to minimize size of email address
    # Time is in minutes granularity and start counting on 2022-01-01 to reduce bytes to represent time
    data = [
        verp_type.value,
        object_id or 0,
        int((time.time() - VERP_TIME_START) / 60),
    ]
    json_payload = json.dumps(data).encode("utf-8")
    # Signing without itsdangereous because it uses base64 that includes +/= symbols and lower and upper case letters.
    # We need to encode in base32
    payload_hmac = hmac.new(
        config.VERP_EMAIL_SECRET.encode("utf-8"), json_payload, VERP_HMAC_ALGO
    ).digest()[:8]
    encoded_payload = base64.b32encode(json_payload).rstrip(b"=").decode("utf-8")
    encoded_signature = base64.b32encode(payload_hmac).rstrip(b"=").decode("utf-8")
    return "{}.{}.{}@{}".format(
        config.VERP_PREFIX,
        encoded_payload,
        encoded_signature,
        sender_domain or config.EMAIL_DOMAIN,
    ).lower()
```

The function creates a JSON payload containing `[verp_type.value, object_id, timestamp_in_minutes]`, then base32-encodes both the payload and an HMAC signature. The final address format is `{VERP_PREFIX}.{base32_payload}.{base32_signature}@{domain}`.

When a bounce arrives, I traced how the system identifies the original email. In `email_handler.py` (lines 2056-2074), the bounce detection checks if the recipient starts with the VERP prefix or matches the bounce patterns. The `get_verp_info_from_email()` function parses the address, verifies the HMAC signature, and extracts the VerpType and object_id.

The thresholds are: more than 12 bounces in 24 hours triggers immediate disable, more than 5 bounces in 24h combined with more than 10 bounces in the prior 7 days triggers disable, bounces on at least 9 of the last 10 days triggers disable, and at the account level more than 10 bounces per day for more than 4 of the last 10 days triggers disable.

If any threshold is met, the alias is disabled via `change_alias_status(alias, enabled=False, ...)`.

The reply phase handler (lines 1595-1670) records the bounce against `contact.website_email` instead of the mailbox, creates the same RefusedEmail and Bounce records, but notably does NOT call `should_disable()`. The alias is never disabled based on reply-phase bounces, because those failures are due to the external recipient's server, not the user's mailbox configuration.

Both handlers set `email_log.bounced = True`, `email_log.refused_email_id`, and `email_log.bounced_mailbox_id` before committing, which allows the user to see the bounced email details in their dashboard.
