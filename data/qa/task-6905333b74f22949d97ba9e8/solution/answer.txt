This confirmed that SimpleLogin has a session fallback mechanism. When an API request comes in without the `Authentication` header, the system checks Flask-Login's `current_user.is_authenticated`. If you're logged in via the web interface, the request succeeds with HTTP 200 and returns the normal JSON response containing the requested data. This dual authentication approach is quite practical for developers who want to test API endpoints while logged into the dashboard, they don't need to manually set up API key headers for every request. That same test also revealed something interesting about privileged operations. When you try to access sudo-protected endpoints without sudo mode active, you get HTTP 440 with `{"error": "Need sudo"}`. That 440 status code caught my attention because it's not a standard HTTP code - it's SimpleLogin's custom status code specifically for sudo requirements. Standard HTTP doesn't define 440, so SimpleLogin chose this number to clearly distinguish sudo failures from regular authentication failures (401) or permission denials (403). This custom code allows client applications to specifically detect when sudo re-authentication is needed versus other auth issues, enabling them to prompt users to re-enter their password rather than showing a generic error.

So the raw bytes stored in Redis are pickle-serialized Python dictionaries, not JSON like I initially expected. The binary data starts with protocol bytes `[128, 4, ...]` which indicates pickle protocol 4, the most recent protocol version that Python's pickle module uses by default. This choice of pickle over JSON makes sense for Flask session storage because pickle can serialize arbitrary Python objects including datetimes and custom classes, while JSON is limited to basic types. The trade-off is that pickle data is Python-specific and not human-readable, but for server-side session storage this isn't a significant drawback.

When you deserialize an authenticated session, you find these key fields: `_user_id`, `_fresh`, `_id`, and `csrf_token`. The `_user_id` field stores the database ID of the logged-in user, which Flask-Login uses to load the user object on each request. The `_fresh` boolean indicates whether the user authenticated recently (true) or has a remembered session from a previous visit (false) - this distinction matters for security-sensitive operations. The `_id` is Flask's internal session identifier, and `csrf_token` provides cross-site request forgery protection for form submissions.

Sessions are stored with Redis keys in the format `session:{session_id}` where the session_id is a UUID v4 string. This simple prefix scheme keeps session data organized and allows for easy pattern-based operations like finding all sessions or implementing session cleanup. The session cookies are signed with `itsdangerous.Signer` using `salt='session'` and `key_derivation='hmac'`. The HMAC key derivation provides cryptographic binding between the secret key and the salt, ensuring that even if an attacker discovered signatures from a different salt context, they couldn't forge session cookies.

The session ID does NOT change during login. The same session ID that exists before you authenticate persists after successful login. The login flow just adds the `_user_id` key to the existing session data rather than creating a fresh session. This means SimpleLogin doesn't implement session fixation protection through ID regeneration.

This is a notable security consideration. In a classic session fixation attack, an attacker could set a known session ID in a victim's browser before they log in, then use that same session ID to hijack the authenticated session afterward. Best practice recommends regenerating the session ID upon authentication to invalidate any pre-authentication session tokens. Looking at the codebase, the `purge_session()` method exists in `session.py` and does regenerate the session ID, but it's only called from `logout_session()`, not during the login flow. The login files (`login.py` and `login_utils.py`) simply call `login_user()` which populates the existing session with user data without changing the session identifier. For most deployments this risk is mitigated by HTTPS and secure cookie flags, but it's worth noting that the defense-in-depth measure of session ID regeneration is not currently implemented.

Reply-To headers get replaced with a reverse alias so replies route back through SimpleLogin. This is essential for the privacy model, when someone replies to a forwarded email, their reply goes to a special reverse alias address (like `reply+uniqueid@simplelogin.co`) instead of your real email. SimpleLogin then forwards that reply to the original sender, maintaining the anonymity of your actual email address throughout the conversation.

Received headers are preserved since they're standard routing headers that document the path an email took through various mail servers. Stripping these would look suspicious to spam filters and could break email authentication checks like SPF and DKIM that rely on routing information. SimpleLogin adds its own Received header to the chain but keeps existing ones intact, maintaining a complete audit trail of the message's journey.

Custom incoming X-headers pass through unchanged, which is important for emails that carry application-specific metadata. Many automated emails use X-headers to convey tracking information, subscription IDs, or other data that recipients or their email clients might need. By preserving these, SimpleLogin ensures that forwarded emails maintain their full functionality.

The system also adds its own custom X-headers during forwarding: `X-SimpleLogin-Type` indicates the direction ("Forward" or "Reply"), `X-SimpleLogin-EmailLog-ID` provides a unique identifier for tracking and debugging, and `X-SimpleLogin-Envelope-From` records the original envelope sender. These headers help with troubleshooting delivery issues and provide transparency about how the message was processed.

The tokens expire after exactly 600 seconds, which is 10 minutes. They use `itsdangerous.TimestampSigner` with a `max_age=600` parameter. When a token is too old, you get a `SignatureExpired` exception, and the API returns HTTP 412 with "Alias creation time is expired, please retry".

The 10-minute window is a reasonable balance between security and usability. It's long enough that users can browse available domains, think about their alias prefix, and complete the creation process without rushing. But it's short enough to limit the window of opportunity if a signed suffix token were somehow leaked or intercepted. The `TimestampSigner` embeds the signing time directly into the token (the `.aXYhtA.` portion in the signed output), so the server doesn't need to store any state to verify expiration, it simply computes whether the embedded timestamp is within the allowed age range.

The signed token format follows the pattern `{original_value}.{timestamp}.{signature}`, where the timestamp is base64-encoded and the signature is an HMAC of the value plus timestamp. This design means you can't simply modify the timestamp to extend validity because that would invalidate the signature, and you can't forge a new signature without knowing the server's secret key. The HTTP 412 (Precondition Failed) status code is an appropriate choice for this scenario since it indicates that a precondition of the request (token validity) was not met.


Every time you make an API call with a key, two database fields get updated. The `last_used` field gets set to the current timestamp via `arrow.now()`, and `times` is an integer counter that increments by 1 with each call.

The `arrow` library is used instead of Python's built-in `datetime` because Arrow provides a cleaner API and better timezone handling. The timestamp includes full ISO 8601 formatting with microsecond precision and timezone offset (e.g., `2026-01-25T14:00:45.767896+00:00`), which is useful for debugging and auditing API usage across different time zones.

These tracking fields serve multiple purposes. The `last_used` timestamp helps users identify stale API keys that haven't been used in a long time and might be candidates for revocation. It also helps with security auditing, if you suspect unauthorized access, you can check when each key was last used. The `times` counter provides a simple usage metric that can help identify unusually active keys (potential abuse) or rarely-used keys (candidates for cleanup). Both fields are updated synchronously with every API request before the actual endpoint logic runs, ensuring accurate tracking even if the endpoint itself fails. The update is committed to the database immediately via `Session.commit()`, so the stats are durably recorded.
