I started by building Maddy to examine its module initialization sequence firsthand.

```
$ go build -o maddy ./cmd/maddy
```

To understand how the module registry gets populated, I traced through maddy.go and found that Go's init functions run before main, registering every module type with the global registry. The maddy.go file imports numerous packages purely for side effects.

```go

The debug log format comes from line 68:

```

The lazy initialization pattern works through GetInstance in internal/module/instances.go at line 53. When any code needs a module, it calls GetInstance which checks the Initialized map first. If the module is not yet initialized, it sets Initialized to true before calling Init, preventing circular dependency loops:

```go
// From internal/module/instances.go:53-75
func GetInstance(name string) (Module, error) {
	aliasedName := aliases[name]
	if aliasedName != "" {
		name = aliasedName
	}

	mod, ok := instances[name]
	if !ok {
		return nil, fmt.Errorf("unknown config block: %s", name)
	}

	// Break circular dependencies.
	if Initialized[name] {
		return mod.mod, nil
	}

	Initialized[name] = true
	if err := mod.mod.Init(mod.cfg); err != nil {
		return mod.mod, err
	}

	return mod.mod, nil
}
```

This explains why modules can reference each other with ampersand syntax regardless of configuration file order. The instances registry stores both the module object and its config.Map together before Init is called, so any module can be looked up and initialized on demand.

Endpoint modules follow a different path than regular modules. Looking at instancesFromConfig in maddy.go, endpoint modules are checked first via GetEndpoint and collected separately. They are never placed in the global instances registry:

```go
// From maddy.go:312-321
endpFactory := module.GetEndpoint(modName)
if endpFactory != nil {
	inst, err := endpFactory(modName, block.Args)
	if err != nil {
		return nil, err
	}

	endpoints = append(endpoints, modInfo{instance: inst, cfg: block})
	continue
}
```

After all regular modules are registered, endpoints are initialized first, which drives the lazy initialization cascade for any modules they reference. The runtime output confirms this ordering: the sql module initializes, then SMTP starts listening (triggering its reference to local_mailboxes), then IMAP starts listening (triggering its references).

This runtime evidence shows exactly how the pipeline coordinates modules during message flow. The check module require_mx_record initializes its state when the message arrives. The modifier replace_rcpt transforms the recipient address from "user+tag@test.local" to "user@test.local" by stripping the plus-address suffix. The logs show the three-level modifier chain: global modifiers leave the address unchanged, per-source modifiers apply the regex replacement, and per-rcpt modifiers pass through unchanged.

The message pipeline coordinates these modules through the checkRunner in internal/msgpipeline/check_runner.go. Check execution runs in parallel using goroutines, with results merged via sync.Once for rejection and quarantine decisions. Modifiers are tracked at three levels in msgpipelineDelivery: globalModifiersState, sourceModifiersState, and rcptModifiersState.

The complete journey from configuration text to a working module graph follows this sequence: the parser reads the configuration file into Node structures, moduleMain processes global directives, then instancesFromConfig walks each block. Endpoint modules are collected separately and initialized first. When endpoints initialize, they parse their nested pipeline configurations, triggering GetInstance calls for any ampersand references. This causes lazy initialization of those modules. The process cascades until all reachable modules are initialized.

The runtime evidence that the module graph has fully settled appears in the log output. When I see "smtp: listening on tcp://127.0.0.1:2525" and "imap: listening on tcp://127.0.0.1:1993", this confirms both endpoints have fully initialized with all their referenced modules. Any module that remains uninitialized would cause an error about unused configuration blocks.
