The Grouping to Matrix transformation produces empty strings for missing row and column intersections by default. This differs semantically from numeric zero and affects how downstream visualizations render and calculate. The transformation's `emptyValue` option defaults to `SpecialValue.Empty` which yields empty string `''`, and visualizations like bar charts convert non-numeric values to null, causing missing cells to render as absent data rather than zero-height bars.

I started by locating the default empty value configuration in the transformation:

```bash
$ grep -n "DEFAULT_EMPTY_VALUE" packages/grafana-data/src/transformations/transformers/groupingToMatrix.ts
26:const DEFAULT_EMPTY_VALUE = SpecialValue.Empty;
71:  const emptyValue = options.emptyValue || DEFAULT_EMPTY_VALUE;
```

The transformation implementation in `packages/grafana-data/src/transformations/transformers/groupingToMatrix.ts` defines the default empty value:

```typescript
export interface GroupingToMatrixTransformerOptions {
  columnField?: string;
  rowField?: string;
  valueField?: string;
  emptyValue?: SpecialValue;
}

const DEFAULT_COLUMN_FIELD = 'Time';
const DEFAULT_ROW_FIELD = 'Time';
const DEFAULT_VALUE_FIELD = 'Value';
const DEFAULT_EMPTY_VALUE = SpecialValue.Empty;
```

The `DEFAULT_EMPTY_VALUE` is set to `SpecialValue.Empty`. I checked the `SpecialValue` enum definition:

```bash
$ grep -A 6 "export enum SpecialValue" packages/grafana-data/src/types/transformations.ts
export enum SpecialValue {
  True = 'true',
  False = 'false',
  Null = 'null',
  Empty = 'empty',
}
```

The enum is defined in `packages/grafana-data/src/types/transformations.ts`:

```typescript
export enum SpecialValue {
  True = 'true',
  False = 'false',
  Null = 'null',
  Empty = 'empty',
}
```

The enum provides four options: True, False, Null, and Empty. There is no Zero option for numeric zero values.

When the transformation processes data, it builds a matrix from row, column, and value fields:

```typescript
const matrixValues: { [key: string]: { [key: string]: unknown } } = {};

for (let index = 0; index < valueField.values.length; index++) {
  const columnName = keyColumnField.values[index];
  const rowName = keyRowField.values[index];
  const value = valueField.values[index];

  if (!matrixValues[columnName]) {
    matrixValues[columnName] = {};
  }

  matrixValues[columnName][rowName] = value;
}
```

The `matrixValues` object only stores entries that exist in the input data. For row/column combinations not present in the input, the value is undefined in the object.

When building the output fields, the transformation fills in missing values using `getSpecialValue`:

```typescript
for (const columnName of columnValues) {
  let values = [];
  for (const rowName of rowValues) {
    const value = matrixValues[columnName][rowName] ?? getSpecialValue(emptyValue);
    values.push(value);
  }
  // ...
}
```

The nullish coalescing operator `??` triggers when `matrixValues[columnName][rowName]` is undefined (missing combination), and calls `getSpecialValue` with the configured `emptyValue` option:

```typescript
function getSpecialValue(specialValue: SpecialValue) {
  switch (specialValue) {
    case SpecialValue.False:
      return false;
    case SpecialValue.True:
      return true;
    case SpecialValue.Null:
      return null;
    case SpecialValue.Empty:
    default:
      return '';
  }
}
```

For the default case `SpecialValue.Empty`, this returns empty string `''`.

The test file demonstrates this behavior:

```typescript
it('generates Matrix with multiple fields', async () => {
  const cfg: DataTransformerConfig<GroupingToMatrixTransformerOptions> = {

```

The input has three data points: C1/R1=1, C1/R2=4, C2/R1=5. The combination C2/R2 is missing. The output shows `C2.values` as `[5, '']` where `''` is the empty string for the missing C2/R2 intersection.

When a bar chart visualizes this data, I traced the value conversion:

```bash
$ grep -n "Number.isFinite" public/app/plugins/panel/barchart/utils.ts
100:        values: field.values.map((v) => (Number.isFinite(v) ? v : null)),
```

The empty string is converted during data preparation in `public/app/plugins/panel/barchart/utils.ts`:

```typescript
frame.fields.forEach((field) => {
  if (field !== xField) {
    if (field.type === FieldType.number && !field.config.custom?.hideFrom?.viz) {
      const field2 = {
        ...field,
        values: field.values.map((v) => (Number.isFinite(v) ? v : null)),
        // ...
      };

      fields.push(field2);
    }
  }
});
```

The mapping `(v) => (Number.isFinite(v) ? v : null)` converts each value. For an empty string, `Number.isFinite('')` evaluates to `false` because empty string is not a finite number, so the empty string is replaced with `null`, and `null` in chart data means no data point, so the bar does not render.

This creates the semantic distinction between empty string and numeric zero. When the value is `0`, `Number.isFinite()` returns `true`, so the chart renders a bar with zero height. When the value is an empty string `''`, `Number.isFinite()` returns `false`, converting it to null, so the bar is absent. When the value is `null`, `Number.isFinite()` also returns `false`, and the bar is absent.

The user who expects missing means zero gets unexpected behavior because the transformation produces empty strings, which visualizations interpret as missing data rather than zero values.

The test with `emptyValue: SpecialValue.Null` shows the alternative:

```typescript
it('generates Matrix with empty entries', async () => {
  const cfg: DataTransformerConfig<GroupingToMatrixTransformerOptions> = {
    id: DataTransformerID.groupingToMatrix,
    options: {
      emptyValue: SpecialValue.Null,
    },
  };
  // ...
  const expected: Field[] = [
    // ...
    {
      name: '1000',
      type: FieldType.number,
      values: [1, null],
      config: {},
    },
    // ...
  ];
});
```

With `SpecialValue.Null`, missing values become `null` instead of empty string. However, this still produces absent bars in charts because `Number.isFinite(null)` is also `false`.

The user intent of missing means zero cannot be achieved with the current `SpecialValue` options because none of them produce numeric zero. Empty returns `''` which is treated as non-numeric and the bar becomes absent. Null returns `null` which is treated as non-numeric and the bar becomes absent. True returns `true` which is treated as non-numeric and the bar becomes absent. False returns `false` which is treated as non-numeric and the bar becomes absent.

To get numeric zero for missing intersections, the transformation would need a `SpecialValue.Zero` option or equivalent that returns `0`, but this option does not currently exist in the codebase.

Calculations like totals and color scales are similarly affected. When summing column values, the result is `1 + 4 = 5` (only real values) versus `1 + 4 + 0 = 5` if missing was zero. The total is numerically the same in this case, but the count of values differs. For averages, the result is `(1 + 4) / 2 = 2.5` (two values) versus `(1 + 4 + 0) / 3 = 1.67` (three values if missing counted as zero).

Color scales that map values to colors treat empty/null as no value with no color or a special missing-value color, rather than as the color for zero.

The exact condition where semantics shift from data to absence: the transformation uses the nullish coalescing operator `??` to check for undefined matrix entries and replaces them with `getSpecialValue(emptyValue)`. When `emptyValue` is `SpecialValue.Empty`, the return value `''` is placed in a numeric field. Downstream visualizations check `Number.isFinite()` and reject the empty string as non-numeric data, converting it to null and treating the cell as absent rather than as a valid zero value.

Comparing value handling in bar chart rendering showed:

```
Value: 20    -> Number.isFinite: true  -> bar renders at height 20
Value: ""    -> Number.isFinite: false -> bar renders as ABSENT
Value: null  -> Number.isFinite: false -> bar renders as ABSENT
Value: 0     -> Number.isFinite: true  -> bar renders at height 0
```

The visualization appears to say B has no Y data rather than B has zero Y value, which may misrepresent the underlying meaning if the absence of a row in the input data is semantically equivalent to a zero measurement.
