I started investigating how Paperless-NGX manages document files when metadata changes by examining the signal handlers and file handling modules.

The file movement mechanism is implemented in handlers.py through two signal receivers. Here's the relevant code from `src/documents/signals/handlers.py`:

```python
@receiver(models.signals.m2m_changed, sender=Document.tags.through)

    # ... generates new filename based on current tags ...

        validate_move(instance, old_source_path, instance.source_path)
```

Examining the actual logging behavior in the file relocation code reveals an important finding: successful file moves happen silently without debug logging. The `update_filename_and_move_files` function performs `os.rename()` operations without any logger calls for successful moves. Logging only occurs when validation fails.

From `src/documents/signals/handlers.py`, the `validate_move` function contains the only logging in this code path:

```python
def validate_move(instance, old_path, new_path):
    if not os.path.isfile(old_path):
        logger.fatal(f"Document {str(instance)}: File {old_path} has gone.")
        raise CannotMoveFilesException()

    if os.path.isfile(new_path):
        logger.warning(
            f"Document {str(instance)}: Cannot rename file "
            f"since target path {new_path} already exists.",
        )
        raise CannotMoveFilesException()
```

This means during a successful file relocation triggered by tag changes, no log messages appear at all. The file is simply moved using `os.rename()` and the database is updated. The operation completes silently.

Only when validation fails do log messages appear:

```
[FATAL] Document test_invoice: File /media/documents/originals/test_invoice.pdf has gone.
[WARNING] Document test_invoice: Cannot rename file since target path /media/documents/originals/finance,important/test_invoice.pdf already exists.
```

This design choice means successful file relocations leave no trace in the logs, while failures are clearly reported. To observe the actual file movement happening, one would need to either add custom logging or monitor the filesystem directly.

Now, what happens when something goes wrong during a move? The rollback mechanism lives in the exception handler of `update_filename_and_move_files`. From `src/documents/signals/handlers.py`:

```python
except (OSError, DatabaseError, CannotMoveFilesException):
    # This happens when either:
    #  - moving the files failed due to file system errors
    #  - saving to the database failed due to database errors
    
    try:
        if move_original and os.path.isfile(instance.source_path):
            os.rename(instance.source_path, old_source_path)
        
        if move_archive and os.path.isfile(instance.archive_path):
            os.rename(instance.archive_path, old_archive_path)
    except Exception:
        pass  # Sanity checker will catch inconsistencies
    
    instance.filename = old_filename
    instance.archive_filename = old_archive_filename
```

The validation function that can trigger rollback, from `src/documents/signals/handlers.py`:


Moving beyond file handling, I wanted to understand why the classifier sometimes finishes training instantly while other times it takes much longer. The answer lies in SHA1 hashing. From `src/documents/classifier.py`:

```python
def train(self):
    logger.debug("Gathering data from database...")
    m = hashlib.sha1()
    
    for doc in Document.objects.order_by("pk").exclude(
        tags__is_inbox_tag=True,
    ):
        preprocessed_content = preprocess_content(doc.content)
        m.update(preprocessed_content.encode("utf-8"))
        
        y = -1
        dt = doc.document_type
        if dt and dt.matching_algorithm == MatchingModel.MATCH_AUTO:
            y = dt.pk
        m.update(y.to_bytes(4, "little", signed=True))
        
        # Similar for correspondent and tags
    
    new_data_hash = m.digest()
    
    if self.data_hash and new_data_hash == self.data_hash:
        return False  # Skip training - data unchanged
```

The classifier computes a SHA1 hash of all training data (document content, types, correspondents, tags). When the hash matches the stored hash, training completes instantly by returning False. When data changes, the hash differs and full training proceeds with log messages like "Gathering data from database", "Vectorizing data", and "Training tags classifier".

Duplicate detection was another mystery I wanted to unravel. How can two files that look completely different be rejected as duplicates? The answer is MD5 checksums. From `src/documents/consumer.py`:

```python
def pre_check_duplicate(self):
    with open(self.path, "rb") as f:
        checksum = hashlib.md5(f.read()).hexdigest()
    if Document.objects.filter(
        Q(checksum=checksum) | Q(archive_checksum=checksum),
    ).exists():
        self._fail(
            MESSAGE_DOCUMENT_ALREADY_EXISTS,
            f"Not consuming {self.filename}: It is a duplicate.",
        )
```

Two files that look completely different visually can be rejected as duplicates if their binary content produces the same MD5 hash. The system checks both the original checksum and archive_checksum fields, so even uploading an archived version of an existing document will be caught.

The sanity checker intrigued me next. It compares stored checksums against actual file checksums. From `src/documents/sanity_checker.py`:

```python
with doc.source_file as f:
    checksum = hashlib.md5(f.read()).hexdigest()

if not checksum == doc.checksum:
    messages.error(
        f"Checksum mismatch of document {doc.pk}. "
        f"Stored: {doc.checksum}, actual: {checksum}."
    )

# When no issues found:
if len(self._messages) == 0:
    logger.info("Sanity checker detected no issues.")
```

Finally, I wondered about ghost files lurking in the media folder. The sanity checker detects orphans by tracking all files in the media directory. From `src/documents/sanity_checker.py`:

```python
# Collect all files in media directory
present_files = []
for root, subdirs, files in os.walk(settings.MEDIA_ROOT):
    for f in files:
        present_files.append(os.path.normpath(os.path.join(root, f)))

# After processing all documents, remaining files are orphans
for extra_file in present_files:
    messages.warning(f"Orphaned file in media dir: {extra_file}")
```

Orphaned files do linger in the media folder. They are the files that exist on disk but have no corresponding document record in the database. This can happen from failed consumption, manual file placement, database restores, or interrupted deletions. The sanity checker walks the entire media directory, tracks which files belong to documents, and reports any extras as warnings.

The entire investigation demonstrates how Paperless-NGX maintains document integrity through multiple layers: signal driven file management for immediate relocation, checksum verification for duplicate detection and sanity checking, and comprehensive orphan detection. No source files were modified during this investigation, and all temporary test files were cleaned up afterward.
