Services Disabled by Default

The startup logs reveal several services that report as "disabled" or "skipped" because they require explicit configuration:

| Service | Log Message | Default State | Configuration Required |
|---------|-------------|---------------|------------------------|
| SMTP Notifications | `SMTP not configured, notifications disabled` | Disabled | `[smtp] enabled = true` and `host` must be set |
| Remote Cache | `Using database as cache backend` | Database fallback | `[remote_cache] type = redis` or `memcached` with connection info |
| HA Alerting | `HA alerting disabled, no HA peers configured` | Disabled | `[unified_alerting] ha_peers` must list peer addresses |
| Image Renderer | `No image renderer plugin found` | Unavailable | Install `grafana-image-renderer` plugin |
| Recording Rules | (silent - disabled by default) | Disabled | `[recording_rules] enabled = true` |
| Remote Alertmanager | (silent - disabled by default) | Disabled | `[unified_alerting] remote.alertmanager.enabled = true` |

These services implement the `registry.CanBeDisabled` interface in the codebase. Their `IsDisabled()` methods check configuration values, and when the required settings are missing or set to `false`, the services skip initialization and log their disabled status.

For example, SMTP is controlled by `conf/defaults.ini`:
```ini
[smtp]
enabled = false
host =
```

Since `enabled = false` and no `host` is configured, the notification service detects this and reports itself as disabled rather than failing.

This explains why "some services report disabled while others claim success" — services with no external dependencies (like the SQLStore, plugin loader, HTTP server) initialize successfully, while services requiring external infrastructure (SMTP server, Redis, HA peers, renderer plugin) report as disabled until explicitly configured.

Verifying Authentication with Actual Login

Testing that the default credentials work:

```
$ curl -s -X POST http://localhost:3000/login \
  -H "Content-Type: application/json" \
  -d '{"user":"admin","password":"admin"}' \
  -c cookies.txt -w "\nHTTP Status: %{http_code}\n"

{"message":"Logged in","redirectUrl":"/"}
HTTP Status: 200
```

The login succeeds with HTTP 200 and returns `{"message":"Logged in"}`.

Verifying with an authenticated API call:
```
$ curl -s -b cookies.txt http://localhost:3000/api/user

{"id":1,"email":"admin@localhost","name":"","login":"admin","theme":"","orgId":1,"isGrafanaAdmin":true,...}
```

The admin user has `id:1`, email `admin@localhost`, and `isGrafanaAdmin:true`.

Testing unauthenticated access (anonymous auth disabled):
```
$ curl -s http://localhost:3000/api/user -w "\nHTTP Status: %{http_code}\n"

{"message":"Unauthorized"}
HTTP Status: 401
```

Anonymous access correctly returns HTTP 401 Unauthorized, confirming anonymous authentication is disabled by default.

1. Database already exists - no file creation
2. All migrations already recorded in `migration_log` table - all skipped
3. Admin user already exists (user count > 0) - creation logic bypassed
4. Preinstalled plugin already present - no download needed

Generated Files Required for Runtime

The Wire dependency injection framework generates `pkg/server/wire_gen.go` which is required for the server to compile and run. This file begins with:

```go
// Code generated by Wire. DO NOT EDIT.

//go:generate go run -mod=mod github.com/google/wire/cmd/wire gen -tags "oss"
//go:build !wireinject
// +build !wireinject

package server
```

Without this generated file, the dependency injection would fail and the server would not start. Running `make gen-go` regenerates this file from `pkg/server/wire.go`.

| `[security] admin_user` | `admin` | Default admin username |
| `[security] admin_password` | `admin` | Default admin password |
| `[security] admin_email` | `admin@localhost` | Default admin email |
| `[auth.anonymous] enabled` | `false` | Anonymous access disabled |
| `[users] allow_sign_up` | `false` | Self-registration disabled |

The system is NOT in permissive mode. It requires authentication with the well-known default credentials `admin/admin`.

Summary

When Grafana starts from a completely clean state:

1. Database: SQLite3 database created at `data/grafana.db`
2. Migrations: All schema migrations execute and are recorded in `migration_log` table
3. Admin User: Created automatically with username `admin`, password `admin`, email `admin@localhost`
4. Organization: Default `Main Org.` created with id `1`
5. Plugins: 
   - Core plugins available immediately (compiled in)
   - Bundled frontend plugins loaded from `public/app/plugins/`
   - `grafana-lokiexplore-app` auto-downloaded to `data/plugins/`
6. Security: Anonymous access disabled; authentication required
7. Generated Files: `pkg/server/wire_gen.go` must exist (generated by Wire)
8. Disabled Services: SMTP, remote cache (Redis/Memcached), HA alerting, and image renderer are disabled by default because they require explicit configuration of external infrastructure

On subsequent runs, admin creation and migrations are skipped because the system detects existing state, resulting in significantly faster startup.
