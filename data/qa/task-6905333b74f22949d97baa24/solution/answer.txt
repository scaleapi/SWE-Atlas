To investigate the shutdown behavior with a ramping executor, I created a test script with 5 starting VUs and ran it, then sent SIGINT after about 9 seconds.

```bash
export const options = {
```

For the gRPC server streaming test with a 30ms graceful ramp down, I started the gRPC server from the examples and created this test script. Note: This test requires running the gRPC example server first with `cd /app && go run ./lib/testutils/grpcservice/cmd/server.go`.

For the data sharing behavior investigation, I created a test to measure memory usage with increasing VU counts. If the SharedArray was working correctly, memory should remain constant regardless of VUs (just the data size plus the runtime overhead). If each VU creates its own copy, then memory will grow linearly with VU count.

First I created a CSV file for testing:
```bash
python3 -c "
import random
import string
with open('/tmp/large_data.csv', 'w') as f:
    f.write('id,name,value,description,extra1,extra2,extra3\n')
    for i in range(100000):
        desc = ''.join(random.choices(string.ascii_letters, k=50))
        f.write(f'{i},item_{i},{random.randint(1,10000)},{desc},extra_data_{i},more_data_{i},padding_{i}\n')
"
ls -lh /tmp/large_data.csv
```

Output:
```
-rw-r--r--    1 root     root       11.4M Jan 19 07:47 /tmp/large_data.csv
```

Test script:
```bash
cat > /tmp/test_csv_memory.js << 'EOF'
import { open } from 'k6/experimental/fs'
import csv from 'k6/experimental/csv'
import { sleep } from 'k6';

export const options = {
    duration: '5s',
}

const file = await open('/tmp/large_data.csv');
const csvRecords = await csv.parse(file, { delimiter: ',' })

export default function() {
    const record = csvRecords[__ITER % csvRecords.length];
    sleep(1);
}
EOF
```

Command executed to measure memory with different VU counts:
```bash
echo "=== Testing with 1 VU ===" && /usr/bin/time -v /tmp/k6 run --vus 1 --duration 5s /tmp/test_csv_memory.js 2>&1 | grep "Maximum resident"
echo "=== Testing with 5 VUs ===" && /usr/bin/time -v /tmp/k6 run --vus 5 --duration 5s /tmp/test_csv_memory.js 2>&1 | grep "Maximum resident"
echo "=== Testing with 10 VUs ===" && /usr/bin/time -v /tmp/k6 run --vus 10 --duration 5s /tmp/test_csv_memory.js 2>&1 | grep "Maximum resident"
```

Output:
```
=== Testing with 1 VU ===
        Maximum resident set size (kbytes): 175604

=== Testing with 5 VUs ===
        Maximum resident set size (kbytes): 265860

=== Testing with 10 VUs ===
        Maximum resident set size (kbytes): 416892
```

The output tells us that there is a memory leak. With an 11.4MB CSV file, the memory footprint should remain relatively constant regardless of VU count if the SharedArray was working correctly. Instead, memory grows linearly with increasing number of VUs (176MB → 266MB → 417MB), showing that each VU is creating its own copy of the entire dataset.

The root cause is in the module.go file at js/modules/k6/experimental/csv/module.go line 126:
```go
underlyingSharedArrayName := parseSharedArrayNamePrefix + strconv.Itoa(time.Now().Nanosecond())
```

The function `time.Now().Nanosecond()` returns a different value each time it's called, generating a unique SharedArray name for each VU during initialization. Since the SharedArray implementation uses the name as a key to determine if data should be shared, each VU ends up with its own separate copy of the parsed file instead of sharing a single copy.

For the Prometheus metric reporting integrity investigation, I created a test to capture and inspect Prometheus remote write output from k6. This test requires setting up a capture server to intercept the Prometheus remote write data.

Test script:
```bash
cat > /tmp/test_prometheus.js << 'EOF'
import http from 'k6/http';
import { check } from 'k6';
import exec from 'k6/execution';

export let options = {
    vus: 1,
    duration: '2s',
    iterations: 1
};

export default function () {
    exec.vu.metrics.tags.BU = 'AA';
    const res = http.get('https://httpbin.org/get');
    check(res, {
        'status is 200': (r) => r.status === 200,
    });
}
EOF
```

I wrote a capture server to intercept and decode the Prometheus remote write data (requires Go with prometheus protobuf dependencies):

```go
package main

import (
    "fmt"
    "io"
    "log"
    "net/http"

    prompb "buf.build/gen/go/prometheus/prometheus/protocolbuffers/go"
    "github.com/klauspost/compress/snappy"
    "google.golang.org/protobuf/proto"
)

func main() {
    http.HandleFunc("/api/v1/write", func(w http.ResponseWriter, r *http.Request) {
        body, _ := io.ReadAll(r.Body)
        decoded, _ := snappy.Decode(nil, body)
        var req prompb.WriteRequest
        proto.Unmarshal(decoded, &req)

        for _, ts := range req.Timeseries {
            var labels []string
            for _, label := range ts.Labels {
                labels = append(labels, fmt.Sprintf("%s=%s", label.Name, label.Value))
            }
            // Print metrics that have the BU tag
            for _, label := range ts.Labels {
                if label.Name == "BU" {
                    fmt.Printf("Labels: %v\n", labels)
                    break
                }
            }
        }
        w.WriteHeader(http.StatusOK)
    })
    log.Fatal(http.ListenAndServe(":9090", nil))
}
```

Command (requires capture server running on port 9090):
```bash
K6_PROMETHEUS_RW_SERVER_URL=http://127.0.0.1:9090/api/v1/write \
K6_PROMETHEUS_RW_TREND_STATS=min,max \
/tmp/k6 run --out experimental-prometheus-rw /tmp/test_prometheus.js
```

```
Labels: [BU=AA_max __name__=k6_http_req_sending expected_response=true method=GET ...]
Labels: [BU=AA_min __name__=k6_http_req_receiving expected_response=true method=GET ...]

Labels: [BU=AA_min __name__=k6_http_req_duration expected_response=true method=GET ...]

```

I noticed something wrong in the output: the BU tag value shows AA_min and AA_max instead of just AA. The trend stat suffixes (_min, _max) are being appended to my custom tag instead of to the __name__ label.

Expected behavior:
```
__name__=k6_http_req_receiving_min
BU=AA
```

Actual Behavior (corrupted):
```
__name__=k6_http_req_receiving
BU=AA_min
```

I investigated the Prometheus remote write code to find the root cause. In vendor/github.com/grafana/xk6-output-prometheus-remote/pkg/remotewrite/trend.go I found the CacheNameIndex() function that locates the __name__ label:
```go
func (tg *trendAsGauges) CacheNameIndex() {
    if tg.labels[0].Name == namelbl {
        return
    }
    i := sort.Search(len(tg.labels), func(i int) bool {
        return tg.labels[i].Name == namelbl  // BUG: should be >=
    })
    if i < len(tg.labels) && tg.labels[i].Name == namelbl {
        tg.ixname = uint16(i)
    }
}
```

The bug is in the sort.Search comparison function. This function requires a monotonic comparison function where f(i) being true implies f(i+1) is also true (i.e, it finds the first index where the condition becomes true). Using == produces a non monotonic pattern (false,false,false,false) which violates this contract.

When a custom tag like "BU" sorts before __name__ alphabetically (ASCII B < _) the labels array has BU at index 0 and __name__ somewhere after. The binary search with == can miss __name__ entirely because it may check indices on either side without ever landing on the correct position. When this happens, ixname defaults to 0 (the first label). The Append() function then runs:
```go
ts.Labels[tg.ixname].Value += "_" + suffix
```

This appends the suffix to whatever label is at index 0 (BU) instead of to __name__ causing the corruption.
