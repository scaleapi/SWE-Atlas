When kitty's terminal window is resized, the text reflow system must redistribute content across new dimensions while preserving logical line boundaries and cursor positions. The primary orchestration happens in screen.c where the resize entry point coordinates the rewrapping of both visible and scrollback buffers.

The screen_resize function manages three key data structures during reflow. LineBuf holds the visible screen content through main_linebuf and alt_linebuf. HistoryBuf maintains the scrollback history. The Cursor structure tracks position adjustments needed as content moves.

However, examining the linebuf_rewrap signature reveals a problem with how this information gets communicated:

```
$ grep 'void linebuf_rewrap' kitty/lineops.h
void linebuf_rewrap(LineBuf *self, LineBuf *other, index_type *, index_type *, HistoryBuf *, index_type *, index_type *, index_type *, index_type *, ANSIBuf*);
```

There is no boolean parameter to convey whether the history buffer's last line was a continuation that spans into the screen buffer. The history buffer rewrap confirms this isolation by passing NULL as the historybuf parameter:


The issue manifests when a long logical line gets split between the history buffer and the screen buffer. Before resize, the history's last line has next_char_was_wrapped set to true, and the screen's first line continues from it. During the history rewrap, that buffer processes in isolation without knowledge of the screen content. When the screen rewrap runs, the first line of the screen is not marked as a continuation because there is no previous line within the screen buffer carrying the wrap flag.

The init_src_line macro in rewrap_inner calls linebuf_init_line which sets is_continued based on the previous line in the same buffer. For line 0 of the screen, no previous line exists within that buffer, so it gets treated as a new logical line.

This causes the rewrap to treat the split portions as separate logical lines, inserting unwanted line breaks at the boundary where the line was originally split between history and screen. The fix would require checking history_buf_endswith_wrap before rewrapping the screen buffer, passing that information to linebuf_rewrap as an additional parameter, and implementing a restitch operation to merge the split portions back together when the window widens.
