I found the key indexing code in the documents module, so I read the index.py file to understand the search implementation.

```python
# src/documents/index.py (lines 52-61, 87-107, 118-125)

def open_index(recreate=False):
    try:
        if exists_in(settings.INDEX_DIR) and not recreate:
            return open_dir(settings.INDEX_DIR, schema=get_schema())
    except Exception:
        logger.exception("Error while opening the index, recreating.")

    if not os.path.isdir(settings.INDEX_DIR):
        os.makedirs(settings.INDEX_DIR, exist_ok=True)
    return create_in(settings.INDEX_DIR, get_schema())

def update_document(writer, doc):
    tags = ",".join([t.name for t in doc.tags.all()])
    tags_ids = ",".join([str(t.id) for t in doc.tags.all()])
    writer.update_document(
        id=doc.pk,
        title=doc.title,
        content=doc.content,
        correspondent=doc.correspondent.name if doc.correspondent else None,
        correspondent_id=doc.correspondent.id if doc.correspondent else None,
        has_correspondent=doc.correspondent is not None,
        tag=tags if tags else None,
        tag_id=tags_ids if tags_ids else None,
        has_tag=len(tags) > 0,
        type=doc.document_type.name if doc.document_type else None,
        type_id=doc.document_type.id if doc.document_type else None,
        has_type=doc.document_type is not None,
        created=doc.created,
        added=doc.added,
        asn=doc.archive_serial_number,
        modified=doc.modified,
    )

def add_or_update_document(document):
    with open_index_writer() as writer:
        update_document(writer, document)

def remove_document_from_index(document):
    with open_index_writer() as writer:
        remove_document(writer, document)
```

This reveals the system uses Whoosh for full-text search. The open_index function has a fail-safe mechanism - if the index is missing or corrupted, it auto-creates a new empty one. The add_or_update_document function is what gets called to sync document changes to the search index.

Then I looked at the views.py to see what happens when a document is updated via API.

```python
# src/documents/views.py (lines 212-217)

def update(self, request, *args, **kwargs):
    response = super(DocumentViewSet, self).update(request, *args, **kwargs)
    from documents import index

    index.add_or_update_document(self.get_object())
    return response
```

This is the key finding - the index.add_or_update_document call happens AFTER the database save but BEFORE returning the response. This means index updates are synchronous, not queued to a background worker.

So the index update happens synchronously right in the API handler before returning. I also checked the management command for reindexing.

```python
# src/documents/management/commands/document_index.py

class Command(BaseCommand):

    help = "Manages the document index."

    def add_arguments(self, parser):
        parser.add_argument("command", choices=["reindex", "optimize"])
        parser.add_argument(
            "--no-progress-bar",
            default=False,
            action="store_true",
            help="If set, the progress bar will not be shown",
        )

    def handle(self, *args, **options):
        with transaction.atomic():
            if options["command"] == "reindex":
                index_reindex(progress_bar_disable=options["no_progress_bar"])
            elif options["command"] == "optimize":
                index_optimize()
```

This confirms there's a management command available for forcing index reconciliation with the database. The "reindex" option will rebuild the entire index from scratch.

And the actual reindex function in tasks.py:

```python
# src/documents/tasks.py (lines 38-45)

def index_reindex(progress_bar_disable=False):
    documents = Document.objects.all()

    ix = index.open_index(recreate=True)

    with AsyncWriter(ix) as writer:
        for document in tqdm.tqdm(documents, disable=progress_bar_disable):
            index.update_document(writer, document)
```

The recreate=True parameter means the index is completely wiped and rebuilt from the database. The tqdm progress bar provides visual feedback during the rebuild. Despite being in tasks.py, this function runs synchronously when called - it's not dispatched to the task queue.
