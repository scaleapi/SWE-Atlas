I built and ran SFTPGo from this checkout and I only trusted what I could observe in stdout logs, HTTP responses, the structured access log lines, and /metrics.
I built the binary and ran it with serve while pointing at a brand new temp config directory each time.

cd /appgo build -o /tmp/sftpgo ./

I then exercised the proxy header story by sending the same GET / request plain and then again with Forwarded, X-Forwarded-For, and X-Real-IP. The HTTP response did not change between plain and proxy-ish. Both came back as the same 301 redirect with the same Location.
plain GET /HTTP/1.1 301 Moved PermanentlyLocation: /web/usersproxy-ish GET /HTTP/1.1 301 Moved PermanentlyLocation: /web/users

What changed was what got logged. Here are the two access-log lines (volatile values redacted) for those two requests. The plain request logged remote_addr as the real socket address with a port and logged uri under 127.0.0.1:8080. The proxy-ish request logged remote_addr as the first IP in X-Forwarded-For and logged uri with the Host header I set.
{"level":"info","time":"[REDACTED]","sender":"httpd","method":"GET","proto":"HTTP/1.1","remote_addr":"127.0.0.1:45324","request_id":"[REDACTED]","uri":"http://127.0.0.1:8080/","user_agent":"curl/8.12.1","resp_status":301,"resp_size":45,"elapsed_ms":0}{"level":"info","time":"[REDACTED]","sender":"httpd","method":"GET","proto":"HTTP/1.1","remote_addr":"203.0.113.9","request_id":"[REDACTED]","uri":"http://hosthdr.example/","user_agent":"curl/8.12.1","resp_status":301,"resp_size":45,"elapsed_ms":0}
That proxy-ish line also answers the “which address and scheme does it log” question. With X-Forwarded-For: 203.0.113.9, 10.0.0.1 and X-Real-IP: 198.51.100.7, the logged remote_addr matched the first X-Forwarded-For IP, not X-Real-IP and not the last forwarded-for IP. And even though I sent X-Forwarded-Proto: https and Forwarded: proto=https, the logged uri still started with http://, so those proto headers did not flip the logged scheme to https:// in this setup.

I also checked how the sftpgo_http_* metrics move when repeating the same request plain versus with forwarded headers. I took a /metrics snapshot, then did a plain GET /, then took another snapshot, then did the proxy-ish GET /, then took a third snapshot. These are the exact /metrics lines I captured each time.
/metrics snapshot before the plain GET /sftpgo_http_client_errors_total 0sftpgo_http_req_ok_total 1sftpgo_http_req_total 1sftpgo_http_server_errors_total 0/metrics snapshot after the plain GET /sftpgo_http_client_errors_total 0sftpgo_http_req_ok_total 2sftpgo_http_req_total 3sftpgo_http_server_errors_total 0/metrics snapshot after the proxy-ish GET /sftpgo_http_client_errors_total 0sftpgo_http_req_ok_total 3sftpgo_http_req_total 5sftpgo_http_server_errors_total 0
The pattern is the same after the plain and proxy-ish GET /. Each time I scrape /metrics, it returns 200 and increments both sftpgo_http_req_total and sftpgo_http_req_ok_total. Each time I hit GET /, it returns 301 and increments sftpgo_http_req_total but does not increment the 2xx “ok” counter. That is why after each “GET / then scrape” step, req_total jumps by 2 while req_ok_total jumps by 1. This shows the counters move the same way for the same request regardless of whether I include forwarded headers, and it also shows the self-scrape effect of /metrics.
