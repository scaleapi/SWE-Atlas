The test environment bootstraps with a large amount of global state that only exists when tests run. The key setup happens in test/client/setup-test-framework.js, where globals, polyfills, and mocks are installed before any test executes:

Globals from Node.js: global.TextEncoder, global.TextDecoder, global.ReadableStream, global.TransformStream, global.Worker, global.structuredClone
Polyfills: global.ResizeObserver (from resize-observer-polyfill package)
Mocks: global.CSS.supports (mocked), global.fetch (mocked), global.matchMedia (mocked)

None of these exist in the same form when the dev server runs normally. The dev server runs in a browser environment where these APIs are provided natively, but in Jest's Node.js environment they must be explicitly set up.

The test environment also sets environment variables that differ from normal runtime. Looking at package.json, the test-client script is defined as:

"test-client": "TZ=UTC jest -c=test/client/jest.config.js"

This means TZ=UTC is set only when tests run. Outside of tests, TZ is unset. This matters because tests that deal with dates and times rely on consistent timezone behavior. For example, the trials tests in client/state/sites/plans/test/trials.js use:

jest.useFakeTimers().setSystemTime( new Date( '2022-01-10T00:00:00+00:00' ) );

The test "Should return the correct number of days left before the trial expires" passes and expects exactly 31 days because the system time is frozen and TZ=UTC ensures consistent date parsing. Without TZ=UTC, these tests could produce different results depending on the local timezone of the machine running them.

Additionally, the jest.config.js defines test-only globals:

globals: {
    google: {},
    __i18n_text_domain__: 'default',
}

These global variables (google and __i18n_text_domain__) exist only in the test environment and are not set when running the development server.

What stood out immediately about network handling is that tests disable all real network access by default using nock.disableNetConnect(). Any code path that accidentally tries to hit the real WordPress API during a test will fail with error code ENETUNREACH unless it is explicitly mocked. The error message is: Nock: Disallowed net connect for "public-api.wordpress.com:443/rest/v1/me"

To trace how a mocked API response flows through the action creator back to the test assertion, I examined client/state/stats/posts/test/actions.js and the corresponding action creator in client/state/stats/posts/actions.js.

The action creator requestPostStats is a thunk that dispatches multiple actions:

export function requestPostStats( siteId, postId, fields = [] ) {
    return ( dispatch ) => {
        dispatch( { type: POST_STATS_REQUEST, postId, siteId, fields } );

        return wpcom
            .site( siteId )
            .statsPostViews( postId, { fields: fields.join() } )
            .then( ( stats ) => {
                dispatch( receivePostStats( siteId, postId, stats ) );
                dispatch( { type: POST_STATS_REQUEST_SUCCESS, siteId, postId, fields } );
            } )
            .catch( ( error ) => {
                dispatch( { type: POST_STATS_REQUEST_FAILURE, siteId, postId, fields, error } );
            } );
    };
}

The test uses the useNock helper to intercept network requests before they reach the real API:

useNock( ( nock ) => {
    nock( 'https://public-api.wordpress.com:443' )
        .persist()
        .get( '/rest/v1.1/sites/2916284/stats/post/2454?fields=views%2Cyears' )
        .reply( 200, { views: 2, years: {} } )
        .get( '/rest/v1.1/sites/2916285/stats/post/2455?fields=views' )
        .reply( 403, {
            error: 'authorization_required',
            message: 'User cannot access this private blog.',
        } );
} );

The data flow for the failure case works like this: the test calls requestPostStats(2916285, 2455, ['views'])(spy), which dispatches POST_STATS_REQUEST first, then calls wpcom.site(2916285).statsPostViews(). Nock intercepts this request and returns the mocked 403 response. The .catch() block in the action creator receives the error object containing the message "User cannot access this private blog." and dispatches POST_STATS_REQUEST_FAILURE with that error. The test asserts:

test( 'should dispatch fail action when request fails', () => {
    return requestPostStats( 2916285, 2455, [ 'views' ] )( spy ).then( () => {
        expect( spy ).toHaveBeenCalledWith( {
            type: POST_STATS_REQUEST_FAILURE,
            siteId: 2916285,
            postId: 2455,
            fields: [ 'views' ],
            error: expect.objectContaining( { message: 'User cannot access this private blog.' } ),
        } );
    } );
} );

The test passing proves the complete data flow: nock intercepts the request, returns the mocked 403 error, the action creator's .catch() block transforms it into a POST_STATS_REQUEST_FAILURE action with the error message from the mock, and dispatches it to the spy where the assertion verifies the exact shape of the action. The real API is never contacted.

This test proves a value differs between test and development contexts. The test in client/state/sites/plans/test/trials.js mocks the config module:

jest.mock( '@automattic/calypso-config', () => {
    const mock = () => '';
    mock.isEnabled = jest.fn( () => true );
    return mock;
} );

const withTrialEnabled = () =>
    config.isEnabled.mockImplementation( ( key ) => key === 'plans/migration-trial' );
const withTrialDisabled = () =>
    config.isEnabled.mockImplementation( ( key ) => key !== 'plans/migration-trial' );

The test "Should return true when the migration trial is in the purchases list" calls isSiteOnMigrationTrial() twice within the same test:

test( 'Should return true when the migration trial is in the purchases list', () => {
    const plan = getPlan( siteId, PLAN_MIGRATION_TRIAL_MONTHLY );
    const state = getState( plan, siteId );

    withTrialEnabled();
    expect( isSiteOnMigrationTrial( state, siteId ) ).toBeTruthy();

    withTrialDisabled();
    expect( isSiteOnMigrationTrial( state, siteId ) ).toBeFalsy();
} );

The test passing with 1 passed proves that the same function isSiteOnMigrationTrial() returns true when config.isEnabled('plans/migration-trial') is mocked to return true, and returns false when mocked to return false. In development, config/development.json has "plans/migration-trial": true as a static value. In tests, this value is dynamically controlled by the mock and can change within a single test execution. The test output confirms both assertions passed, proving the mock controlled the value at runtime.
