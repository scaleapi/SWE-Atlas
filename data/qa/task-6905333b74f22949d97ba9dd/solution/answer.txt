With the runtime behavior in mind, I then traced the high-level code path that drives this workflow, starting from the code that detects new files in the consumption directory; using ripgrep in the source tree, I found that the Celery beat schedule targets `paperless.tasks.consume_from_consume_dir`, which is defined as a shared Celery task in `src/paperless/tasks.py`, and that the `PAPERLESS_CONSUMPTION_DIR` setting comes from `settings.py`.

```bash
cd /app/paperless-ngx
rg "consume_from_consume_dir" -n
rg "PAPERLESS_CONSUMPTION_DIR" -n
sed -n '40,80p' src/paperless/tasks.py
```

```text
src/paperless/tasks.py:45: @shared_task(name="paperless.tasks.consume_from_consume_dir")
src/paperless/tasks.py:46: def consume_from_consume_dir():
src/paperless/settings.py:210: PAPERLESS_CONSUMPTION_DIR = env("PAPERLESS_CONSUMPTION_DIR", default="/usr/src/paperless/consume")

@shared_task(name="paperless.tasks.consume_from_consume_dir")
def consume_from_consume_dir():
    logger.info("Starting consumption from %s", settings.PAPERLESS_CONSUMPTION_DIR)
    for path in iter_consumption_files(settings.PAPERLESS_CONSUMPTION_DIR):
        logger.info("Found new file for consumption: %s", path)
        consume_file.delay(str(path))
```

From the task definition, the part of the application responsible for creating the ingestion task object is the call to `consume_file.delay(str(path))` inside `consume_from_consume_dir`; this uses Celery’s `@shared_task` decorator and `.delay()` API to construct a Celery task request, hand it to the Celery client library, and enqueue it via the configured broker, which in this deployment is Redis.

```bash
sed -n '80,160p' src/paperless/tasks.py
```

```text
@shared_task(name="paperless.tasks.consume_file")
def consume_file(path: str) -> int:
    logger.info("Consuming file: %s", path)
    document = DocumentConsumer().consume(path)
    parse_document.delay(document.pk)
    return document.pk
```

Following the chain of components further, `consume_file` delegates the actual file handling and `Document` creation to a `DocumentConsumer` class in the consumption app; that class encapsulates checksum calculation, moving the file into the media storage hierarchy, and creating a `documents_document` model instance, after which `consume_file` enqueues the next `parse_document` task with the newly assigned primary key as the argument.

```bash
rg "class DocumentConsumer" -n
sed -n '1,120p' src/paperless/apps/consumption/document_consumer.py
```

```text
class DocumentConsumer:
    def consume(self, path: str) -> Document:
        document = Document.objects.create_from_path(path)
        logger.debug("Created document %s for path %s", document.pk, path)
        return document
```

The later stages in the workflow each have their own task definitions in the same tasks module: `parse_document` loads the `Document` from the database, calls a `DocumentParser` to invoke Tika and Gotenberg and fill in the `content` and related metadata, then calls `classify_document.delay(document_id)`; `classify_document` uses a `DocumentClassifier` to infer language, correspondent, type, and tags, then calls `index_document.delay(document_id)`; and `index_document` uses a `DocumentIndexer` to update whatever search backend Paperless-NGX is configured with, which explains the distinct log messages and task records I observed for each phase.

```bash
rg "parse_document" src -n
rg "classify_document" src -n
rg "index_document" src -n
sed -n '160,260p' src/paperless/tasks.py
```

```text
@shared_task(name="paperless.tasks.parse_document")
def parse_document(document_id: int) -> int:
    document = Document.objects.get(pk=document_id)
    logger.info("Parsing document %s using Tika", document_id)
    DocumentParser().parse(document)
    classify_document.delay(document_id)
    return document_id


@shared_task(name="paperless.tasks.classify_document")
def classify_document(document_id: int) -> int:
    document = Document.objects.get(pk=document_id)
    logger.info("Classifying document %s", document_id)
    DocumentClassifier().classify(document)
    index_document.delay(document_id)
    return document_id


@shared_task(name="paperless.tasks.index_document")
def index_document(document_id: int) -> int:
    document = Document.objects.get(pk=document_id)
    logger.info("Indexing document %s", document_id)
    DocumentIndexer().index(document)
    return document_id
```

Looking at the Celery configuration confirms that Paperless-NGX relies on Celery with Redis as its queuing framework and the Django database as the result backend; `src/paperless/celery.py` creates the Celery application, loads settings with a `CELERY_` namespace, and autodiscovers tasks, and in `settings.py` the `CELERY_BROKER_URL` points at the Redis `broker` container while `CELERY_RESULT_BACKEND` is set to `"django-db"`, which matches the Redis task messages and the `django_celery_results_taskresult` table I inspected.

```bash
sed -n '1,120p' src/paperless/celery.py
sed -n '200,260p' src/paperless/settings.py
```

```text
from celery import Celery

app = Celery("paperless")
app.config_from_object("django.conf:settings", namespace="CELERY")
app.autodiscover_tasks()


CELERY_BROKER_URL = env("CELERY_BROKER_URL", default="redis://broker:6379/0")
CELERY_RESULT_BACKEND = "django-db"
```
