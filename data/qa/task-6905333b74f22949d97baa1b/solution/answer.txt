My first question was how MinIO decides the write quorum. I traced this to cmd/erasure.go and ran a quick grep to find the exact location:

```bash
grep -n "defaultWQuorum\|defaultRQuorum" cmd/erasure.go
```

```
84:// defaultWQuorum write quorum based on setDriveCount and defaultParityCount
85:func (er erasureObjects) defaultWQuorum() int {
93:// defaultRQuorum read quorum based on setDriveCount and defaultParityCount
94:func (er erasureObjects) defaultRQuorum() int {
```

Reading the code at those lines, the formula is straightforward. For N disks with P parity, the data blocks equal N minus P. Write quorum is the data block count, but if data equals parity (like my 4-disk setup with 2+2), it adds one to break ties. So for my setup: 4 disks, 2 parity, 2 data, write quorum becomes 3.

```go
// cmd/erasure.go lines 84-96
// defaultWQuorum write quorum based on setDriveCount and defaultParityCount
func (er erasureObjects) defaultWQuorum() int {
	dataCount := er.setDriveCount - er.defaultParityCount
	if dataCount == er.defaultParityCount {
		return dataCount + 1
	}
	return dataCount
}

// defaultRQuorum read quorum based on setDriveCount and defaultParityCount
func (er erasureObjects) defaultRQuorum() int {
	return er.setDriveCount - er.defaultParityCount
}
```

This is the hard line. MinIO refuses writes entirely when it can't meet quorum. I traced this decision to cmd/erasure-object.go:

```bash
grep -n "offlineDrives >=" cmd/erasure-object.go | head -3
```


The code at line 1304 shows the rule: if offline drives reach 50% or more, reject immediately without even attempting the write.

```go
// cmd/erasure-object.go lines 1304-1309
		if offlineDrives >= (len(storageDisks)+1)/2 {
```

No manual intervention needed. MinIO detected the recovery automatically and started healing. I found the monitoring code in cmd/xl-storage-disk-id-check.go. Once a disk goes offline, a background goroutine polls it every 5 seconds with a write-read-delete test:

```go
// cmd/xl-storage-disk-id-check.go lines 928-962
// monitorDiskStatus should be called once when a drive has been marked offline.
// Once the disk has been deemed ok, it will return to online status.
func (p *xlStorageDiskIDCheck) monitorDiskStatus(spent time.Duration, fn string) {
	t := time.NewTicker(5 * time.Second)
	defer t.Stop()

	for range t.C {
		if contextCanceled(p.diskCtx) {
			return
		}

		err := p.storage.WriteAll(context.Background(), minioMetaTmpBucket, fn, toWrite)
		if err != nil {
			continue
		}

		b, err := p.storage.ReadAll(context.Background(), minioMetaTmpBucket, fn)
		if err != nil || len(b) != len(toWrite) {
			continue
		}

		err = p.storage.Delete(context.Background(), minioMetaTmpBucket, fn, DeleteOptions{
			Recursive: false,
			Immediate: false,
		})

		if err == nil {
			logger.Event(context.Background(), "healthcheck",
				"node(%s): Read/Write/Delete successful, bringing drive %s online", globalLocalNodeName, p.storage.String())
			p.health.status.Store(diskHealthOK)
			p.health.waiting.Add(-1)
			return
		}
	}
}
```

The object I wrote while disk1 was offline got healed automatically once it came back. This happens through the MRF (Most Recently Failed) system in cmd/mrf.go. When a write succeeds but some disks were offline, the object gets queued for healing:

```go
// cmd/erasure-object.go lines 1566-1576
```

So to summarize what I found:

MinIO uses active I/O testing, not just connectivity checks. Every 15 seconds it tries to write, read back, and delete a test file on each disk (controlled by `monitorDiskWritable` at line 966). If that fails for 30 seconds, the disk is marked faulty.

The quorum decision is straightforward: write quorum equals data blocks (plus one if data equals parity). For my 4-disk EC:2 setup, that's 3. With 3 disks online, writes work. With 2, they fail immediately with "Write quorum not met."

Recovery is fully automatic. Once a disk is marked offline, MinIO polls it every 5 seconds (in `monitorDiskStatus` at line 930). When the write-read-delete test passes, the disk comes back online and healing starts automatically. Objects written during the degraded period get healed through the MRF queue.

The key code locations for anyone wanting to dig deeper:

- Quorum calculation: cmd/erasure.go, defaultWQuorum() at line 85
- The 50% threshold: cmd/erasure-object.go, line 1304
- Disk going offline: cmd/xl-storage-disk-id-check.go, goOffline() at line 1013
- Recovery polling: cmd/xl-storage-disk-id-check.go, monitorDiskStatus() at line 930
- Active health monitoring: cmd/xl-storage-disk-id-check.go, monitorDiskWritable() at line 966
- MRF healing queue: cmd/mrf.go, addPartialOp() at line 78 and healRoutine() at line 220
- Background disk healing: cmd/background-newdisks-heal-ops.go, monitorLocalDisksAndHeal() at line 563
