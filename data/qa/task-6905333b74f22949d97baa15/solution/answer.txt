I started by examining the source code to find the exact configuration values, then ran tests to verify my understanding.

$ cat > /tmp/verify_queue_timing.go << 'EOF'
package main

import (
    "fmt"
    "math"
    "time"
)

func main() {
    // Configuration values from queue.go
    maxParallelism := 16
    initialRetryTime := 15 * time.Minute
    retryTimeScale := 2.0
    maxTries := 8
    
    fmt.Println("=== Queue Configuration Values ===")
    fmt.Printf("maxParallelism:   %d\n", maxParallelism)
    fmt.Printf("initialRetryTime: %v\n", initialRetryTime)
    fmt.Printf("retryTimeScale:   %.1f\n", retryTimeScale)
    fmt.Printf("maxTries:         %d\n", maxTries)
    
    // Initial delivery calculation
    messages := 500
    deliveryTime := 2 * time.Second
    batches := (messages + maxParallelism - 1) / maxParallelism
    initialTime := time.Duration(batches) * deliveryTime
    
    fmt.Println("\n=== Initial Delivery Time ===")
    fmt.Printf("Messages: %d\n", messages)
    fmt.Printf("Parallel workers: %d\n", maxParallelism)
    fmt.Printf("Batches: ceil(%d/%d) = %d\n", messages, maxParallelism, batches)
    fmt.Printf("Time per batch: %v (parallel)\n", deliveryTime)
    fmt.Printf("Total initial delivery: %v\n", initialTime)
    
    // Retry delays
    fmt.Println("\n=== Retry Delay Schedule ===")
    var totalDelay time.Duration
    for attempt := 1; attempt <= maxTries; attempt++ {
        delay := initialRetryTime * time.Duration(math.Pow(retryTimeScale, float64(attempt-1)))
        totalDelay += delay
        fmt.Printf("After attempt %d: TriesCount=%d, delay = 15min × 2^%d = %v\n", 
            attempt, attempt, attempt-1, delay)
    }
    fmt.Printf("\nTotal delay sum: %v (%.2f hours)\n", totalDelay, totalDelay.Hours())
    
    // Edge case
    fmt.Println("\n=== Edge Case: TriesCount = 0 ===")
    powResult := math.Pow(retryTimeScale, float64(0-1))
    durationResult := time.Duration(powResult)
    edgeDelay := initialRetryTime * durationResult
    fmt.Printf("math.Pow(2.0, -1) = %v\n", powResult)
    fmt.Printf("time.Duration(0.5) = %v (truncated to int64)\n", durationResult)
    fmt.Printf("15min × 0 = %v\n", edgeDelay)
    fmt.Println("Note: This never occurs because TriesCount++ happens before delay calculation")
}
EOF

The verification script confirms the timing calculations. With 16 parallel workers processing 500 messages where each takes 2 seconds, the initial delivery pass completes in approximately 64 seconds. The calculation works because all 16 deliveries run simultaneously, so each batch of 16 messages completes in 2 seconds rather than 32 seconds.

The total time until bounce messages are generated is 63 hours and 45 minutes. This comes from summing all eight retry delays: 15 minutes plus 30 minutes plus 1 hour plus 2 hours plus 4 hours plus 8 hours plus 16 hours plus 32 hours. After the ninth delivery attempt, when TriesCount reaches 8, the check on line 390 triggers bounce generation.

The edge case analysis reveals an interesting quirk of Go's type system. If TriesCount were zero when the formula executes, math.Pow(2, -1) returns 0.5. However, time.Duration is an int64 representing nanoseconds, so converting 0.5 to time.Duration truncates it to zero. The resulting delay would be 15 minutes times 0, which equals 0 seconds rather than the mathematically expected 7.5 minutes.

This edge case never occurs in practice because the code structure prevents it. Looking at lines 407 and 414 of queue.go, the increment meta.TriesCount++ happens before the delay calculation. Since messages enter with TriesCount at 0, the first increment brings it to 1 before any delay is computed. The minimum exponent used is therefore 0, giving a minimum delay of exactly 15 minutes.

Users should expect approximately 64 hours from the first delivery attempt until receiving a bounce message when the destination server remains consistently unreachable, with each retry pass taking about 64 seconds to process all 500 messages in parallel.
