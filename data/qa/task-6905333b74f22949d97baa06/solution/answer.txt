I started by examining how Kitty's history buffer handles high output loads at runtime. The core implementation lives in history.c where the HistoryBuf structure manages terminal scrollback using a segmented memory architecture with circular buffer indexing. This design solves two problems simultaneously: it avoids allocating massive contiguous memory blocks upfront, and it efficiently handles the common case where old lines are continuously evicted as new output arrives.

First, I verified the SEGMENT_SIZE constant directly from the source:

```
$ grep -n "SEGMENT_SIZE" kitty/history.c | head -5
15:#define SEGMENT_SIZE 2048
23:    const size_t cpu_cells_size = self->xnum * SEGMENT_SIZE * sizeof(CPUCell);
24:    const size_t gpu_cells_size = self->xnum * SEGMENT_SIZE * sizeof(GPUCell);
25:    s->cpu_cells = calloc(1, cpu_cells_size + gpu_cells_size + SEGMENT_SIZE * sizeof(LineAttrs));
38:    index_type seg_num = y / SEGMENT_SIZE;
```

The choice of 2048 lines per segment represents a balance between memory efficiency and allocation overhead. Each segment holds CPU cells (character data and attributes), GPU cells (rendering information), and line attributes. By using fixed-size segments, Kitty can allocate memory in predictable chunks rather than resizing a single massive array as the buffer grows.

The segment allocation happens lazily through the segment_for function which I verified in the source:

```c
static index_type
segment_for(HistoryBuf *self, index_type y) {
    index_type seg_num = y / SEGMENT_SIZE;
    while (UNLIKELY(seg_num >= self->num_segments && SEGMENT_SIZE * self->num_segments < self->ynum)) add_segment(self);
    if (UNLIKELY(seg_num >= self->num_segments)) fatal("Out of bounds access to history buffer line number: %u", y);
    return seg_num;
}
```

This lazy allocation strategy means that when you create a HistoryBuf with a capacity of 10,000 lines, Kitty does not immediately allocate memory for all 10,000 lines. Instead, it allocates segments on demand as lines are accessed. The UNLIKELY macro hints to the compiler that segment allocation is rare after initial warmup, optimizing the common path where segments already exist.

This confirms the buffer correctly stores all 10,000 lines with line(0) being the most recently pushed content and line(9999) being the oldest. The performance of approximately 42 milliseconds for 10,000 lines means Kitty can handle roughly 238,000 lines per second, which is more than sufficient for even the most aggressive terminal output scenarios.

The circular buffer wrap behavior becomes visible when pushing more lines than the buffer capacity. I traced through the historybuf_push function:

```c
static index_type
historybuf_push(HistoryBuf *self, ANSIBuf *as_ansi_buf) {
    index_type idx = (self->start_of_data + self->count) % self->ynum;
    init_line(self, idx, self->line);
    if (self->count == self->ynum) {
        pagerhist_push(self, as_ansi_buf);
        self->start_of_data = (self->start_of_data + 1) % self->ynum;
    } else self->count++;
    return idx;
}
```

The key insight here is the use of modular arithmetic. The expression (start_of_data + count) % ynum calculates where the next line should be written. Before the buffer is full, count simply increments. Once count equals ynum (buffer at capacity), the buffer transitions to wrap-around mode. At this point, start_of_data advances with each push, effectively rotating a "window" through the circular buffer. The line being overwritten is first serialized to the pager history via pagerhist_push, preserving it for extended scrollback.

The count stays capped at 5 (the buffer capacity) while the oldest entries (WRAP_LINE_0, WRAP_LINE_1, WRAP_LINE_2) were evicted as new lines wrapped around. Notice how line(0) always returns the most recently pushed content, regardless of where it physically resides in the underlying array. This abstraction is handled by the index_of function which translates logical line numbers to physical buffer positions.

The transition from "filling" mode to "wrapping" mode happens seamlessly at push 5. Before this point, each push increments count. Starting at push 5, count stays at 5 but start_of_data begins advancing. This is the critical edge case in the buffer management: the first time a line must be evicted. The pagerhist_push call ensures no data is lost during this transition.

The pager history mechanism is configured through pagerhist_extend which I verified in the source:

```c
static bool
pagerhist_extend(PagerHistoryBuf *ph, size_t minsz) {
    size_t buffer_size = ringbuf_capacity(ph->ringbuf);
    if (buffer_size >= ph->maximum_size) return false;
    size_t newsz = MIN(ph->maximum_size, buffer_size + MAX(1024u * 1024u, minsz));
    ringbuf_t newbuf = ringbuf_new(newsz);
    if (!newbuf) return false;
    size_t count = ringbuf_bytes_used(ph->ringbuf);
    if (count) ringbuf_copy(newbuf, ph->ringbuf, count);
    ringbuf_free((ringbuf_t*)&ph->ringbuf);
    ph->ringbuf = newbuf;
    return true;
}
```

The initial allocation uses MIN(1024*1024, pagerhist_sz), starting at 1MB and growing in 1MB increments up to maximum_size, as confirmed in the source at line 67. This conservative growth strategy prevents the pager history from immediately consuming large amounts of memory. When the ring buffer fills, pagerhist_extend doubles the allocation (up to 1MB increments) by creating a new buffer, copying existing content, and freeing the old buffer. The maximum_size cap prevents unbounded memory growth.

The pager history stores evicted lines as ANSI escape sequences rather than the structured cell format used by HistoryBuf. This serialization trades some access speed for significantly reduced memory usage, since ANSI text is more compact than the full cell representation with all its rendering attributes. When a user scrolls far back into history, these serialized lines are re-parsed for display.

The logical line numbering (where 0 is most recent) correctly translates to physical buffer positions even after multiple wrap cycles. The index_of function uses the formula idx = (start_of_data + count - 1 - lnum) % ynum to perform this translation. When lnum is 0 (most recent line), this resolves to the position just written. When lnum is count-1 (oldest line), this resolves to start_of_data, which is the next position to be overwritten.

The memory structures remain consistent during concurrent scrollback navigation because segment allocation and ring buffer writes are sequential within the same thread context. There are no locks protecting the history buffer because all access occurs on the main thread. The segment_for function's bounds check triggers add_segment only when accessing lines beyond currently allocated segments, preventing out-of-bounds access while avoiding premature allocation.

One edge case worth noting involves the is_continued attribute for wrapped lines. When a line is too long to fit in the terminal width, it wraps to the next line with is_continued set to true. The init_line function in history.c handles this specially for line 0 by checking if the pager history ring buffer ends with a newline. If not, the line is marked as continued, ensuring that when users scroll through history, wrapped lines display correctly as a single logical unit rather than appearing fragmented.
