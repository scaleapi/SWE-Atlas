To check whether alias reply handling in SimpleLogin was actually misrouting replies or just behaving differently than expected, I reproduced the full inbound reply flow in the development environment. I did not modify any repository source files. Instead, I used a temporary tracing script (which I removed afterward) to simulate an inbound reply and capture exactly how the system resolved the alias and which user ID it ultimately routed the reply to.

First, I used a small temporary script to drive the real mail pipeline. That script did not change any library or app code it only:
 -Imported the existing SMTP mail handler
 -Created real data in the test DB (user, alias, contact)
 -Fed a synthetic inbound reply into the same code path that production uses
 -Stubbed out SMTP sending so routing could be printed instead of actually sent

In real use, the very first Python code that sees an incoming email is:
 `MailHandler.handle_DATA(...)` (called by aiosmtpd)

From there, it calls:
 -`MailHandler._handle(envelope, msg)`, which then calls
 -`handle(envelope, msg)`, which in turn calls
 -`handle_reply(...)` for reverse-alias replies

My temporary script invoked `_handle(envelope, msg)` directly, which is the same internal path used after `handle_DATA` passes control to the handler.

To actually verify the routing outcome in a realistic scenario, I simulated a user replying to an email that had previously been forwarded through an alias. Inside the temporary script, I performed these steps:
Inside the temporary script, I performed these steps:

1. Created a new user in the test database (for example, `user_id = 3`, email like `user_x@mailbox.test`).
2. Created an alias owned by that user (for example, `alias_id = 6`, email like `test_alias@sl.local`).
3. Created a `Contact` linked to that alias, with:
     -`website_email` set to some external address (for example, `contact@example.com`)
     -`reply_email` set to a reverse-alias (for example, `8c00952c-1234-5678@sl.local`).
4. Built a test inbound email message:
    -`From: user_x@mailbox.test` (the user replying)
    -`To: 8c00952c-1234-5678@sl.local` (the reverse-alias)
5. Created an SMTP-style `Envelope` object:
    -`envelope.mail_from = user_x@mailbox.test`
    -`envelope.rcpt_tos = ["8c00952c-1234-5678@sl.local"]`
6. Called the real handler:
   ```
   MailHandler()._handle(envelope, msg)
   ```
This simulated the situation where a user replies to a forwarded email using the reverse-alias address.

When the test reply was processed, the runtime logs showed the following key stages.
First, the generic handler detected this as a reply to a reverse-alias:
Example log line:
```
DEBUG Reply phase user_x@mailbox.test(user_x@mailbox.test) -> 8c00952c-1234-5678@sl.local
```
That shows:
-`mail_from` (envelope sender) is `user_x@mailbox.test`
- The system recognized the recipient `8c00952c-1234-5678@sl.local` as a reverse-alias
- It entered the reply-specific path (`handle_reply`), not the forward path

Inside `handle_reply`, the system resolved the data path as:
 -Look up the `Contact` by `reply_email = "8c00952c-1234-5678@sl.local"`
 -From that `Contact`, get:
  - `contact.alias` (the alias object)
  - `contact.user_id` (the owning user)
- From the alias, get:
  - `alias.email` (for From header)
  - `alias.user` (the same user as above in the normal, consistent case)
  - Derive the mailbox based on `mail_from` and the alias’s allowed mailboxes

The logs then recorded the creation of an `EmailLog` for this reply. A representative line looked like:
```DEBUG Create <EmailLog 2> for <Contact 3 contact@example.com 6>, <User 3 Test User user_x@mailbox.test>, <Mailbox 3 user_x@mailbox.test>
```
After running the script, I also inspected `EmailLog` in the database and saw a concrete row like:

- `id  = 2`
- `is_reply  = true`
- `user_id   = 3`
- `alias_id  = 6`
- `mailbox_id = 3`

This confirmed the following:
- The reply was attributed to the same user who owned the alias (`user_id = 3`).
- The reply was associated with the correct alias (`alias_id = 6`).
- The mailbox used matched the replying mailbox (`mailbox_id = 3`).

Finally, right before sending, the system rewrote headers and prepared the outgoing message. Example log lines:

```
DEBUG From header is test_alias@sl.local
DEBUG Replace To header, old: 8c00952c-1234-5678@sl.local, new: contact@example.com
```

The stubbed `sl_sendmail` showed the final routing decision:
- `envelope_from`: a VERP-style bounce address (for example, `sl.<token>@sl.local`)
- `to_email`: `contact@example.com`
- `From`: `test_alias@sl.local`
- `To`: `contact@example.com`

So, in this controlled run, the system forwarded the reply to the expected external recipient associated with that alias and user.
 
The most important part of the pipeline, in terms of routing correctness, is the mapping:

reverse-alias email to Contact to Alias to User

In code, the first critical lookup in reply-phase is:

- `Contact.get_by(reply_email=<reverse_alias>)`

The `EmailLog` creation then uses:

- `EmailLog.user_id = contact.user_id`
- `EmailLog.alias_id = contact.alias_id`

This is a subtle but important point:
`EmailLog.user_id` comes from `contact.user_id`, not directly from `alias.user_id`.

That means:
- If `contact.reply_email` points to the wrong `Contact` row,
- Or if, for some reason, `contact.user_id` does not match `alias.user_id`,
- Then the system will record and treat the reply as belonging to the wrong user, even though the logs might still show the “right” alias address.

In other words:
- Logs can say ""Reply phase  <correct reverse-alias>”
- But if the `Contact` row for that reverse-alias belongs to another user, the routing and `EmailLog.user_id` will follow the data in `Contact`, not your expectations.

So, based on this end-to-end trace:
- The first handler is `MailHandler.handle_DATA`, which passes the message into `_handle`, then `handle`, then `handle_reply`.
- The alias is resolved to a user via `Contact` and `Alias`.
- The final `user_id` used is taken from `contact.user_id` and written into `EmailLog`.
- The most likely place a wrong routing decision could originate is the `Contact` data:
  - Incorrect or stale `Contact.reply_email` mappings
  - Mismatched `contact.user_id` relative to `alias.user_id`

When an inbound reply arrives, the first backend component that processes it is `MailHandler.handle_DATA`, which receives the SMTP delivery from the server. That function then passes the message into `MailHandler._handle(envelope, msg)`, and from there the system enters the handle and handle_reply logic that deals specifically with reverse-alias replies.

The alias is mapped back to the owning account by using the reverse-alias email address to look up the corresponding row in the Contact table. The contact entry then leads to the correct alias via `contact.alias`, and the correct user via both contact.user_id and alias.user. This chain of relationships ensures the system knows which account the reply belongs to and which mailbox is allowed to send on behalf of the alias.

During processing, SimpleLogin creates an EmailLog entry to persist the routing outcome. In the actual trace, this row contained a real user_id, such as 3, confirming that the reply was tied to the correct alias owner. The key detail is that EmailLog.user_id comes directly from contact.user_id, and that is the value the backend uses when deciding where the reply should be delivered.

Because the reply pipeline ultimately trusts the Contact record, any incorrect routing would most likely come from stale or mismatched contact data. If the contact.reply_email points to the wrong user or if contact.user_id no longer matches the alias’s true owner, the backend will confidently route the reply according to that incorrect mapping even if the logs still show that the correct alias address was recognized.
