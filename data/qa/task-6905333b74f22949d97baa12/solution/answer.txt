I investigated how address tracking works in this mail server and found that your colleague's explanation has several key details backwards. Let me walk through what the system actually does.

The tracking mechanism lives in the message metadata structure, not in the modify or dsn packages you checked. The core data structure is defined in internal/module/msgmetadata.go:

```go
// OriginalRcpts contains the mapping from the final recipient to the
// recipient that was presented by the client.
//
// MsgPipeline will update that field when recipient modifiers
// are executed.
//
// It should be used when reporting information back to client (via DSN,
// for example) to prevent disclosing information about aliases
// which is usually unwanted.
OriginalRcpts map[string]string
```

The direction is the opposite of what your colleague described. They said you can "look up user@domain.com and find it became alias@domain.com" but the actual mapping stores `OriginalRcpts[FINAL_ADDRESS] = ORIGINAL_ADDRESS`. So if user@domain.com becomes alias@domain.com, the system stores `OriginalRcpts["alias@domain.com"] = "user@domain.com"` with the final address as the key and the original as the value.

Your colleague also mentioned the mapping is "built incrementally by each modifier as it runs, creating a chain of all intermediate transformations." This is also incorrect. Looking at internal/msgpipeline/msgpipeline.go, the tracking happens centrally after all modifiers have completed:

```go
func (dd *msgpipelineDelivery) AddRcpt(ctx context.Context, to string) error {
    originalTo := to  // Save ONLY the original

    // Run through ALL modifier chains
    to = globalModifiersState.RewriteRcpt(ctx, to)
    to = sourceModifiersState.RewriteRcpt(ctx, to)  
    to = rcptModifiersState.RewriteRcpt(ctx, to)

    // Store ONLY first→last mapping after all modifiers ran
    if originalTo != to {
        dd.msgMeta.OriginalRcpts[to] = originalTo
    }
}
```

The modifiers themselves are completely unaware of this tracking. From internal/module/modifier.go:

```go
// MsgPipeline will take of populating MsgMeta.OriginalRcpts. RewriteRcpt
// doesn't do it.
```

This means only the first (original) and last (final) addresses are stored. If a message goes through multiple transformations like rcpt1@example.com to rcpt1-alias@example.com to rcpt1-alias2@example.com, the map only contains `OriginalRcpts["rcpt1-alias2@example.com"] = "rcpt1@example.com"`. The intermediate rcpt1-alias@example.com is discarded.

The Multiple test explicitly verifies the chained transformation behavior:

```go
// Chain: rcpt1@example.com → rcpt1-alias@example.com → rcpt1-alias2@example.com
original1 := target.Messages[0].MsgMeta.OriginalRcpts["rcpt1-alias2@example.com"]
if original1 != "rcpt1@example.com" {
    t.Errorf("wrong OriginalRcpts value...")
}
```

The reason for this reverse direction becomes clear when you consider the use case. When delivery fails for alias@domain.com, the system needs to generate a bounce message. It looks up the final address in OriginalRcpts to find the original address the sender used, then reports the failure using that original address. This prevents exposing internal alias structure to external senders.

The statusCollector wrapper in msgpipeline.go handles real-time status reporting for LMTP connections using this same reverse translation:

```go
type statusCollector struct {
    originalRcpts map[string]string
    wrapped       module.StatusCollector
}

func (sc statusCollector) SetStatus(rcptTo string, err error) {
    original, ok := sc.originalRcpts[rcptTo]
    if ok {
        rcptTo = original  // Translate final→original
    }
    sc.wrapped.SetStatus(rcptTo, err)
}
```

So to summarize what your colleague got wrong: the mapping is reverse not forward (final→original, not original→final), it stores only endpoints not intermediate chains, it is built by msgpipeline.go not by the individual modifiers, and it lives in the msgpipeline package not modify or dsn.
