To understand how the choose-fonts kitten works in the kitty terminal, I started by figuring out how to run a kitty instance with isolated settings. Since the kitty repository is already built at `/app`, I can use the binaries directly from `/app/kitty/launcher/`. The key insight is that kitty respects the `KITTY_CONFIG_DIRECTORY` environment variable, which lets me test in complete isolation without affecting any existing kitty configuration.

First, I verified the kitty version:

```bash
/app/kitty/launcher/kitty --version
```

Output:
```
kitty 0.35.2 created by Kovid Goyal
```

To start a single kitty instance with default settings and an isolated config directory, I use:

```bash
export KITTY_CONFIG_DIRECTORY="$(mktemp -d /tmp/kitty-conf-XXXXXX)"
xvfb-run -a /app/kitty/launcher/kitty --single-instance --instance-group "onboard-$$"
```

The `--single-instance` flag ensures only one kitty instance runs in that group, and `--instance-group` with a unique name (using `$$` for the process ID) prevents conflicts with other running kitty instances. The `xvfb-run -a` prefix provides a virtual X11 display for headless environments. I can also start kitty with no config at all using:

```bash
xvfb-run -a /app/kitty/launcher/kitty -c NONE --single-instance --instance-group "onboard-$$"
```

The `-c NONE` option tells kitty to skip loading any configuration file entirely.

Now, to invoke the choose-fonts kitten from inside a running kitty instance, there are two ways. The first is using the `kitty +kitten` prefix:

```bash
/app/kitty/launcher/kitty +kitten choose-fonts
```

The second is calling `kitten` directly:

```bash
/app/kitty/launcher/kitten choose-fonts
```

Both methods work identically. Note that these commands require either being inside a running kitty terminal or having a display available. When I checked the help output by running:

```bash
/app/kitty/launcher/kitten choose-fonts --help
```

I saw the available options:

```
Usage: kitten choose-fonts 

Choose the fonts used in kitty

Options:
  --reload-in [=parent]
    By default, this kitten will signal only the parent kitty
    instance it is running in to reload its config, after making
    changes. Use this option to instead either not reload the
    config at all or in all running kitty instances.
    Choices: parent, all, none

  --help, -h

kitten choose-fonts 0.35.2 created by Kovid Goyal
```

The `--reload-in` option is particularly important - it controls whether kitty reloads the configuration after the font selection is saved. The default is `parent`, which reloads only the parent kitty instance. Other choices are `all` (reload all running kitty instances) or `none` (don't reload at all).

To understand the end-to-end behavior, I traced through the source code. The choose-fonts subcommand is registered in `/app/kittens/choose_fonts/main.go` at lines 74-99. Here's how it's set up:

```go
type Options struct {
	Reload_in string
}

func EntryPoint(root *cli.Command) {
	ans := root.AddSubCommand(&cli.Command{
		Name:             "choose-fonts",
		ShortDescription: "Choose the fonts used in kitty",
		Run: func(cmd *cli.Command, args []string) (rc int, err error) {
			opts := Options{}
			if err = cmd.GetOptionValues(&opts); err != nil {
				return 1, err
			}
			return main(&opts)
		},
	})
	ans.Add(cli.OptionSpec{
		Name:    "--reload-in",
		Dest:    "Reload_in",
		Type:    "choices",
		Choices: "parent, all, none",
		Default: "parent",
		Help: `By default, this kitten will signal only the parent kitty instance it is
running in to reload its config, after making changes. Use this option to
instead either not reload the config at all or in all running kitty instances.`,
	})
	clone := root.AddClone(ans.Group, ans)
	clone.Hidden = false
	clone.Name = "choose_fonts"
}
```

This code shows that the `Options` struct captures the `Reload_in` value, and the `EntryPoint` function registers both `choose-fonts` (with hyphens) and `choose_fonts` (with underscores) as valid command names. The option parsing happens through `cmd.GetOptionValues(&opts)`, which populates the `Options` struct, and then the `main` function is called with those options.

The most interesting part is what happens when you press Enter at the final confirmation step. This is handled in `/app/kittens/choose_fonts/final.go` at lines 78-95:

```go
if event.MatchesPressOrRepeat("enter") {
	event.Handled = true
	patcher := config.Patcher{Write_backup: true}
	path := filepath.Join(utils.ConfigDir(), "kitty.conf")
	updated, err := patcher.Patch(path, "KITTY_FONTS", self.settings.serialized(), "font_family", "bold_font", "italic_font", "bold_italic_font")
	if err != nil {
		return err
	}
	if updated {
		switch self.handler.opts.Reload_in {
		case "parent":
			config.ReloadConfigInKitty(true)
		case "all":
			config.ReloadConfigInKitty(false)
		}
	}
	self.lp.Quit(0)
	return nil
}
```

This code reveals several important details. When Enter is pressed, the kitten creates a `config.Patcher` with `Write_backup: true`, which means it will create a backup of the existing config file. It then determines the config file path using `utils.ConfigDir()` and patches the file with the selected font settings. The `Patch` method uses a marker called `"KITTY_FONTS"` to identify the section to update, and it updates four font settings: `font_family`, `bold_font`, `italic_font`, and `bold_italic_font`. After successfully patching the file, it checks the `Reload_in` option value and either reloads just the parent instance or all instances accordingly.

The config directory resolution is controlled by the `KITTY_CONFIG_DIRECTORY` environment variable, as shown in `/app/tools/utils/paths.go` at lines 88-91:

```go
func ConfigDirForName(name string) (config_dir string) {
	if kcd := os.Getenv("KITTY_CONFIG_DIRECTORY"); kcd != "" {
		return Abspath(Expanduser(kcd))
	}
	// ... otherwise search xdg locations ...
}
```

If `KITTY_CONFIG_DIRECTORY` is set, it takes precedence over the standard XDG config locations. This is exactly what I used to isolate my testing.

Now for the crucial question: does kitty actually remember the font choice across restarts? To verify this, I created an automated end-to-end test that runs the entire choose-fonts flow in a headless environment and then checks persistence. Here's the test script I ran:

```bash
#!/bin/bash
set -euo pipefail
CONF_DIR=$(mktemp -d /tmp/kitty-conf-choose-fonts-XXXXXX)
RUNTIME_DIR=$(mktemp -d /tmp/kitty-runtime-choose-fonts-XXXXXX)
SOCK=/tmp/kitty-choose-fonts-$$.sock
chmod 700 "$RUNTIME_DIR"
export KITTY_CONFIG_DIRECTORY="$CONF_DIR"
export XDG_RUNTIME_DIR="$RUNTIME_DIR"

echo "==== Starting kitty with choose-fonts kitten in headless mode ===="
xvfb-run -a /app/kitty/launcher/kitty --listen-on "unix:$SOCK" -o allow_remote_control=socket-only -c NONE /app/kitty/launcher/kitten choose-fonts --reload-in none &
KPID=$!

echo "==== Waiting for socket to be ready ===="
for i in $(seq 1 200); do
  if [ -S "$SOCK" ]; then 
    echo "Socket ready after $i attempts"
    break
  fi
  sleep 0.05
done

sleep 3

echo "==== Sending Enter keys to navigate through choose-fonts ===="
/app/kitty/launcher/kitten @ --to "unix:$SOCK" send-key --match all enter
sleep 0.7
/app/kitty/launcher/kitten @ --to "unix:$SOCK" send-key --match all enter
sleep 0.7
/app/kitty/launcher/kitten @ --to "unix:$SOCK" send-key --match all enter

wait "$KPID" || true

echo ""
echo "==== KITTY_CONFIG_DIRECTORY: $CONF_DIR ===="
echo "==== kitty.conf (if written) ===="
if [ -f "$CONF_DIR/kitty.conf" ]; then
  cat "$CONF_DIR/kitty.conf"
else
  echo "(no kitty.conf created)"
fi

echo ""
echo "==== Verifying font settings are persisted and readable ===="
KITTY_CONFIG_DIRECTORY="$CONF_DIR" /app/kitty/launcher/kitty +runpy 'from kitty.cli import create_default_opts; o=create_default_opts(); print("font_family:", o.font_family); print("bold_font:", o.bold_font); print("italic_font:", o.italic_font); print("bold_italic_font:", o.bold_italic_font)'

echo ""
echo "==== Temp directories created ===="
echo "CONF_DIR: $CONF_DIR"
echo "RUNTIME_DIR: $RUNTIME_DIR"
echo "SOCK: $SOCK"
```

This script does several things. It creates isolated temporary directories for the config and runtime data, starts kitty in headless mode using `xvfb-run` (which provides a virtual X11 display), enables remote control via a Unix socket, and launches the choose-fonts kitten with `--reload-in none` to prevent automatic reloading. It then waits for the socket to be ready and sends three Enter keypresses using the remote control protocol to navigate through the kitten's interface (first Enter selects a font family, second Enter selects the regular face, third Enter confirms the final selection). After the kitten exits, it checks whether a config file was written and verifies the font settings can be read back.

So to answer the original question: yes, kitty absolutely remembers the font choice across restarts. When you press Enter at the final confirmation step in choose-fonts, the kitten writes the font settings to `kitty.conf` in your config directory (either `$KITTY_CONFIG_DIRECTORY` if set, or the standard XDG config location). The settings are written to a marked section that the patcher can update in the future without disturbing other config settings. When you start kitty again, it reads this config file and applies the font settings automatically. The font selection is not just for the current session - it's a permanent configuration change that persists until you either run choose-fonts again or manually edit the config file.

The complete flow is: subcommand registration in `main.go` → option parsing via `GetOptionValues` → user interaction through the TUI → Enter key press in `final.go` → config file patching via `config.Patcher.Patch()` → optional reload signal via `ReloadConfigInKitty()` → exit. The `--reload-in` option controls whether running kitty instances immediately see the change, but regardless of that option, the config file is always written to disk, ensuring persistence across restarts.
