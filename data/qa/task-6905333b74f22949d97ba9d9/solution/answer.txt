Paperless uses Django Q for background task processing with Redis as the message broker. This architecture allows the web application to remain responsive while heavy operations like OCR processing and document indexing happen asynchronously in the background.

Three services coordinate background work as defined in docker/supervisord.conf. The gunicorn process handles HTTP requests, the document_consumer watches the consumption folder for new files, and the qcluster command starts the Django Q worker processes that execute queued tasks.

Command:
```
cat /app/docker/supervisord.conf | grep -A1 "\[program:"
```

Output:
```
[program:gunicorn]
command=gunicorn -c /usr/src/paperless/gunicorn.conf.py paperless.asgi:application
--
[program:consumer]
command=python3 manage.py document_consumer
--
[program:scheduler]
command=python3 manage.py qcluster
```

The Q_CLUSTER configuration in src/paperless/settings.py controls the task queue behavior. This configuration defines how many worker processes run in parallel, the Redis connection string, and timeout settings for tasks that run for extended periods.

Command:
```
cd /app/src && DJANGO_SETTINGS_MODULE=paperless.settings python3 -c "import django; django.setup(); from django.conf import settings; print(settings.Q_CLUSTER)"
```

Output:
```
{'name': 'paperless', 'catch_up': False, 'recycle': 1, 'retry': 1810, 'timeout': 1800, 'workers': 3, 'redis': 'redis://localhost:6379'}
```

Scheduled tasks are created by database migrations and stored in the django_q_schedule table. These recurring tasks handle maintenance operations like training the document classifier, optimizing the search index, and checking for new emails.

Command:
```
cd /app/src && DJANGO_SETTINGS_MODULE=paperless.settings python3 -c "import django; django.setup(); from django_q.models import Schedule; [print(f'Name: {s.name}, Function: {s.func}, Type: {s.schedule_type}') for s in Schedule.objects.all()]"
```

Output:
```
Name: Train the classifier, Function: documents.tasks.train_classifier, Type: H
Name: Optimize the index, Function: documents.tasks.index_optimize, Type: D
Name: Perform sanity check, Function: documents.tasks.sanity_check, Type: W
Name: Check all e-mail accounts, Function: paperless_mail.tasks.process_mail_accounts, Type: I
```

The schedule types are H (Hourly), D (Daily), W (Weekly), and I (Minutes interval).

Background jobs are triggered using the async_task() function from Django Q. When code calls async_task(), the task is serialized and pushed to the Redis queue, where it waits for an available worker to pick it up. The task_name parameter provides a readable identifier that appears in logs and the admin interface. Here are the main locations where tasks are submitted:

In src/documents/views.py for API uploads:
```
async_task(
    "documents.tasks.consume_file",
    temp_filename,
    override_filename=doc_name,
    task_id=task_id,
    task_name=os.path.basename(doc_name)[:100],
)
```

In src/documents/management/commands/document_consumer.py for folder watching:
```
logger.info(f"Adding {filepath} to the task queue.")
async_task(
    "documents.tasks.consume_file",
    filepath,
    override_tag_ids=tag_ids if tag_ids else None,
    task_name=os.path.basename(filepath)[:100],
)
```

In src/paperless_mail/mail.py for email attachments:
```
async_task(
    "documents.tasks.consume_file",
    path=temp_filename,
    override_filename=pathvalidate.sanitize_filename(att.filename),
    task_name=att.filename[:100],
)
```

In src/documents/bulk_edit.py for bulk operations:
```
async_task("documents.tasks.bulk_update_documents", document_ids=affected_docs)
```

To observe task submission at runtime, I used the Django shell to manually enqueue a task. When async_task() is called, it immediately returns a unique task ID and logs the enqueue event. The queue_size() function shows how many tasks are waiting to be processed.

Command:
```
cd /app/src && DJANGO_SETTINGS_MODULE=paperless.settings python3 -c "import django; django.setup(); from django_q.tasks import async_task, queue_size; task_id = async_task('documents.tasks.sanity_check', task_name='test_sanity_check'); print(f'Task ID: {task_id}'); print(f'Queue size: {queue_size()}')"
```

Output:
```
[Q] INFO Enqueued 1
Task ID: 072e676d7a374a82a71c39c5bf7426dc
Queue size: 1
```

Note: The task ID is a randomly generated UUID that will differ on each execution.

Pending tasks are stored in Redis under the key django_q:paperless:q. This is how the system distinguishes between work that is waiting versus work that is actively being processed. Tasks in the Redis queue are waiting, while tasks being handled by a worker are no longer in the queue.

```
redis-cli KEYS "django_q:*"
```

Output:
```

```

When the worker processes tasks, the transition from queued to active is visible in the logs. Running the qcluster command starts the worker cluster, which spawns multiple worker processes. Each worker logs when it begins processing a specific task. The "processing [task_name]" message indicates active execution, while "Processed [task_name]" indicates completion.

After processing completes, tasks are removed from the Redis queue and their results are persisted to the database. This is where task state ends up being stored permanently. The Task model records execution details including start time, stop time, success status, and the return value.


Failed tasks are recorded in the Failure model, which preserves the error message and traceback for debugging. This allows you to investigate what went wrong after the fact without needing to reproduce the failure.

Command:
```
cd /app/src && DJANGO_SETTINGS_MODULE=paperless.settings python3 -c "import django; django.setup(); from django_q.models import Failure; [print(f'FAILED: {f.name}\n  Function: {f.func}\n  Result: {f.result}') for f in Failure.objects.all()]"
```

Output:
```
FAILED: will_fail
  Function: documents.tasks.nonexistent_function
  Result: 'NoneType' object is not callable : Traceback (most recent call last):
  File "/usr/local/lib/python3.11/site-packages/django_q/cluster.py", line 432, in worker
    res = f(*task["args"], **task["kwargs"])
TypeError: 'NoneType' object is not callable
```

Note: This output appears when there are failed tasks. If no tasks have failed, the command produces no output.

To summarize where task state is stored at each stage of the lifecycle:

Pending or waiting tasks are stored in the Redis queue under the key django_q:paperless:q. Active or running tasks are held by the worker process. Completed tasks are persisted to the database in the django_q_task table via the Task model. Successful tasks can be queried via Success.objects.all(). Failed tasks can be queried via Failure.objects.all(). Scheduled recurring tasks are stored in the django_q_schedule table via the Schedule model.

The actual task functions that get executed by workers are defined in dedicated task modules. These functions contain the business logic for document processing, indexing, and other background operations.

In src/documents/tasks.py: consume_file() handles document ingestion, train_classifier() runs ML training hourly, index_optimize() optimizes the search index daily, sanity_check() performs integrity checks weekly, and bulk_update_documents() re-indexes documents after bulk edits.

In src/paperless_mail/tasks.py: process_mail_accounts() checks configured email accounts every 10 minutes.

Real-time progress updates are sent via WebSocket from src/documents/consumer.py using the _send_progress() method to the StatusConsumer in src/paperless/consumers.py. This allows the frontend UI to display live progress bars during document consumption without polling the server.
