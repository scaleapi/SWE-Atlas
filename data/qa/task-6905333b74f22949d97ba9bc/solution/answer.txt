I investigated the staleness detection and resolution lifecycle in Grafana's unified alerting system by examining source code and running tests to observe actual runtime behavior. The investigation addresses all aspects of how stale series get detected, resolved, and how the notification system handles them.

1: Staleness Detection Threshold and Formula

First, I located the staleness detection function in the state manager.

```bash
$ grep -n "func stateIsStale" pkg/services/ngalert/state/manager.go
627:func stateIsStale(evaluatedAt time.Time, lastEval time.Time, intervalSeconds int64) bool {
```

I examined the exact formula used for staleness detection.

```bash
$ sed -n '627,629p' pkg/services/ngalert/state/manager.go
func stateIsStale(evaluatedAt time.Time, lastEval time.Time, intervalSeconds int64) bool {
	return !lastEval.Add(2 * time.Duration(intervalSeconds) * time.Second).After(evaluatedAt)
}
```

The formula shows that a series becomes stale when `lastEval + (2 * intervalSeconds) <= evaluatedAt`. This means the scheduler distinguishes between slow and stopped series by using a two-interval threshold. A series that misses one evaluation cycle is considered potentially slow and given grace time. A series that misses two consecutive cycles is considered stopped and marked stale.

To verify this boundary behavior with runtime evidence, I created and executed a test script.

2: Alert State Transitions for Vanished Series

I examined the deleteStaleStatesFromCache function that handles vanished series.

```bash
$ sed -n '586,625p' pkg/services/ngalert/state/manager.go
func (st *Manager) deleteStaleStatesFromCache(ctx context.Context, logger log.Logger, evaluatedAt time.Time, alertRule *ngModels.AlertRule) []StateTransition {
	staleStates := st.cache.deleteRuleStates(alertRule.GetKey(), func(s *State) bool {
		return stateIsStale(evaluatedAt, s.LastEvaluationTime, alertRule.IntervalSeconds)
	})
	resolvedStates := make([]StateTransition, 0, len(staleStates))

	for _, s := range staleStates {
		logger.Info("Detected stale state entry", "cacheID", s.CacheID, "state", s.State, "reason", s.StateReason)
		oldState := s.State
		oldReason := s.StateReason

		s.State = eval.Normal
		s.StateReason = ngModels.StateReasonMissingSeries
		s.EndsAt = evaluatedAt
		s.LastEvaluationTime = evaluatedAt

		if oldState == eval.Alerting {
			s.ResolvedAt = &evaluatedAt
			image, err := takeImage(ctx, st.images, alertRule)
			if err != nil {
				logger.Warn("Failed to take an image", "error", err)
			} else if image != nil {
				s.Image = image
			}
		}

		record := StateTransition{
			State:               s,
			PreviousState:       oldState,
			PreviousStateReason: oldReason,
		}
		resolvedStates = append(resolvedStates, record)
	}
	return resolvedStates
}
```

The code shows that all stale series transition to `eval.Normal` with `StateReasonMissingSeries`. Critically, only series that were previously `Alerting` get `ResolvedAt` set and trigger a screenshot. This is the key distinction in how vanished series are handled based on their previous state.

3: Resolved Notification Retention Window

I examined the NeedsSending function which controls notification timing.

```bash
$ sed -n '495,520p' pkg/services/ngalert/state/state.go
func (a *State) NeedsSending(resendDelay time.Duration, resolvedRetention time.Duration) bool {
	if a.State == eval.Pending {
		return false
	}

	if a.ResolvedAt != nil && (a.LastSentAt == nil || a.ResolvedAt.After(*a.LastSentAt)) {
		return true
	}

	if a.State == eval.Normal && (a.ResolvedAt == nil || a.LastEvaluationTime.Sub(*a.ResolvedAt) > resolvedRetention) {
		return false
	}

	return a.LastSentAt == nil || !a.LastSentAt.Add(resendDelay).After(a.LastEvaluationTime)
}
```

The retention window logic is on line 513: once `LastEvaluationTime - ResolvedAt > resolvedRetention`, the system stops sending notifications for that resolved alert.

I found the default retention value.

```bash
$ grep -n "ResolvedAlertRetention\|resolved_alert_retention" pkg/setting/setting_unified_alerting.go
125:	ResolvedAlertRetention time.Duration
465:	uaCfg.ResolvedAlertRetention, err = gtime.ParseDuration(valueAsString(ua, "resolved_alert_retention", (15 * time.Minute).String()))
```

The default ResolvedRetention is 15 minutes. Resolved alerts continue generating notifications for up to 15 minutes after resolution, then stop completely.

func fingerprint(labels map[string]string) uint64 {
    keys := make([]string, 0, len(labels))

Summary of Runtime-Verified Findings

1. Staleness Detection Threshold: Formula is `lastEval + (2 * intervalSeconds) <= evaluatedAt`. A series becomes stale after missing two consecutive evaluation intervals, distinguishing genuinely stopped series from temporarily slow ones.

2. Alert State Transitions: Stale series transition to Normal with StateReason=MissingSeries. Only previously Alerting states get ResolvedAt set and trigger resolution notifications.

3. Resolved Notification Retention: ResolvedRetention (default 15 minutes) controls how long resolved alerts continue sending. Once LastEvaluationTime - ResolvedAt > resolvedRetention, notifications stop.

4. Three-Value Interplay: ResendDelay (30s) throttles notification frequency, ResolvedRetention (15min) limits how long resolved notifications continue, LastSentAt tracks when to resend. Resolution notifications bypass resendDelay initially.

5. Screenshot Capture: Stale series follow the same screenshot logic as natural resolutions, only previously Alerting states capture screenshots.

6. Pending Period: Vanishing series skip the For duration entirely. They transition directly to Normal without pending check.

7. Series Reappearance: Reappearing series with identical labels are treated as brand new instances. The old state is deleted when it becomes stale, so the same fingerprint creates a fresh State with no memory of previous incarnation.
