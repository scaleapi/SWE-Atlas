The definitive log line that signals Grafana is ready to accept connections is the one containing "HTTP Server Listen" from the http.server logger. This appears after all the background services have initialized and tells me exactly where and how the server is exposed. The address field shows [::]:3000, which means the server binds to all IPv6 interfaces (and IPv4 via dual-stack) on port 3000. The protocol field confirms it is using plain HTTP, while subUrl being empty indicates no URL prefix is configured for reverse proxy setups. The socket field being empty confirms it is not running in Unix socket mode.

The startup logs reveal that several background services initialize before the HTTP server announces it is listening. The sqlstore establishes the database connection and creates the default admin user and organization. The migrator runs any pending database schema migrations. The provisioning services handle loading dashboards and alert rules from configuration files. The ngalert.scheduler starts the alert evaluation engine with its configured tick interval. The query_service initializes the data query handling layer. The grafana-apiserver starts the Kubernetes-style API server for managing resources like playlists and dashboards. All of these complete their initialization before the HTTP server log line appears, meaning the backend is substantially operational before the UI becomes accessible.

I then tested the health endpoint to observe what a healthy response looks like.

```
$ curl -s http://localhost:3000/api/health | jq
```

```
Output:
{
  "database": "ok",
  "version": "9.2.0",
  "commit": "NA"
}
```

The health endpoint at /api/health returns a JSON response with three fields. The database field is the critical indicator of system readiness, showing "ok" when Grafana can successfully query its database. This check works by executing a simple SELECT 1 query against the configured database backend. When this query succeeds, the field shows "ok" and the endpoint returns HTTP 200. When the database is unreachable or the query fails, the field shows "failing" and the endpoint returns HTTP 503 Service Unavailable. The version and commit fields provide build information and may be hidden if anonymous version hiding is enabled.

The health check implementation in http_server.go uses a databaseHealthy function that performs the actual database connectivity test:

```go
func (hs *HTTPServer) databaseHealthy(ctx context.Context) bool {
    const cacheKey = "db-healthy"
    if cached, found := hs.CacheService.Get(cacheKey); found {
        return cached.(bool)
    }
    err := hs.SQLStore.WithDbSession(ctx, func(session *db.Session) error {
        _, err := session.Exec("SELECT 1")
        return err
    })
    healthy := err == nil
    return healthy
}
```

I then opened a browser to http://localhost:3000 and observed the login flow. The login page appeared with fields for username and password. I entered the default credentials admin for both username and password.

```
Login attempt:
Username: admin
Password: admin
```

After clicking Log in, Grafana immediately displayed a password change prompt before allowing me to proceed. The form showed a message asking me to update my password with fields for new password and confirm new password. There was also an option to skip this step. This prompt appears because Grafana detects that the password used for login is still the default value "admin" and enforces a security check to encourage users to set a custom password. The frontend code in LoginCtrl.tsx checks if the submitted password equals the literal string "admin", and if so, redirects to the password change view rather than proceeding directly to the main dashboard. This internal state change represents the finalization of the admin account's security posture, transitioning from using insecure default credentials to requiring user acknowledgment or password update before full access is granted.

The background services visible in the startup logs correspond to components registered in the background service registry. These include the sqlstore for database connectivity, the migrator for schema management, the provisioning services for dashboard and alert configuration, the ngalert scheduler for alert evaluation, the query service for handling data queries, and the grafana-apiserver for the resource API. The startup logs show that all these services complete their initialization routines before the HTTP server begins listening, which means by the time a user can access the login page in their browser, the entire Grafana backend including alerting, provisioning, and API services is already active and ready to handle requests.
