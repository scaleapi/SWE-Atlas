The behavior you are seeing with k6 scripts that work during init but fail during VU execution when trying to load modules dynamically is by design. k6 intentionally freezes module resolution after the initialization phase completes, and this is enforced through a locking mechanism in the module resolver combined with explicit checks on the require() function.

The core of this behavior lives in js/modules/resolution.go where the ModuleResolver struct has a boolean field called locked. During the init phase (when __VU equals 0), modules can be resolved and cached normally. Once initialization completes, the bundle calls ModuleResolver.Lock() in js/bundle.go at line 129:

```go
// js/bundle.go line 129
bundle.ModuleResolver.Lock()
```

This sets the locked flag to true, and from that point forward, any attempt to resolve a module that is not already in the cache will fail. The lock implementation is straightforward:

```go
// js/modules/resolution.go
func (mr *ModuleResolver) Lock() {
    mr.locked = true
}
```

When the resolver is locked and a module is requested that was not previously resolved during initialization, the resolve function returns an error. The error message is defined as a constant:

```go
// js/modules/resolution.go
const notPreviouslyResolvedModule = "the module %q was not previously resolved during initialization (__VU==0)"
```

However, in practice, VU code attempting to load modules will hit a different guard first. The require() function itself checks whether code is running in the init context by checking if vu.state is nil. During init, vu.state is nil, so require() works. During VU execution, vu.state is not nil, so require() fails immediately with its own error. This check happens in js/bundle.go:

```go
// js/bundle.go
func (r *requireImpl) require(specifier string) (*sobek.Object, error) {
    if !r.inInitContext() {
        return nil, fmt.Errorf(cantBeUsedOutsideInitContextMsg, "require")
    }
    return r.modSys.Require(specifier)
}
```

The error message for this case is defined in js/initcontext.go:

```go
// js/initcontext.go
const cantBeUsedOutsideInitContextMsg = `the "%s" function is only available in the init stage ` +
    `(i.e. the global scope), see https://grafana.com/docs/k6/latest/using-k6/test-lifecycle/ for more information`
```

I verified this behavior by creating a test script that imports the k6/http module during init, then uses it during VU execution. The script ran successfully with multiple VUs:

```javascript
// test_resolved_module.js
import http from 'k6/http';
import { sleep } from 'k6';

export const options = { vus: 2, iterations: 2 };

export default function () {
    const res = http.get('https://httpbin.test.k6.io/get');
    console.log(`VU ${__VU} iter ${__ITER}: http module works, status=${res.status}`);
    sleep(0.1);
}
```

Running this produces successful output showing both VUs can use the init-resolved http module:

```
time="2026-01-20T23:53:57Z" level=info msg="VU 2 iter 0: http module works, status=200" source=console
time="2026-01-20T23:53:57Z" level=info msg="VU 1 iter 0: http module works, status=200" source=console
iterations: 2 (completed successfully)
```

To demonstrate the failure case, I created a script that attempts to call require() during VU execution:

```javascript
// test_require_during_vu.js
export const options = { vus: 1, iterations: 1 };

export default function () {
    console.log(`VU ${__VU}: Attempting require() during VU execution...`);
    try {
        const crypto = require('k6/crypto');
        console.log(`VU ${__VU}: Unexpectedly succeeded with require()`);
    } catch (e) {
        console.log(`VU ${__VU}: require() failed with: ${e.message}`);
    }
}
```

The output confirms the exact error message:

```
time="2026-01-20T23:54:14Z" level=info msg="VU 1: Attempting require() during VU execution..." source=console
time="2026-01-20T23:54:14Z" level=info msg="VU 1: require() failed with: the \"require\" function is only available in the init stage (i.e. the global scope), see https://grafana.com/docs/k6/latest/using-k6/test-lifecycle/ for more information" source=console
```

For relative specifiers, the resolution depends on which module is currently executing. The ModuleResolver maintains a reverse map that tracks module-to-URL mappings so it can determine the correct base path for relative imports. When a module is imported during init with a relative specifier like ./helper.js, the resolver converts it to an absolute URL and caches it under that full specifier. I tested this with a helper module:

```javascript
// helper.js
export function greet(name) { return `Hello, ${name}!`; }
export const VERSION = '1.0.0';
```

```javascript
// test_relative_specifier.js
import { greet, VERSION } from './helper.js';

export const options = { vus: 1, iterations: 2 };

export default function () {
    const msg = greet(`VU-${__VU}`);
    console.log(`Relative import works: ${msg}, version=${VERSION}`);
}
```

The output shows relative imports resolved during init work correctly during VU execution:

```
time="2026-01-20T23:54:20Z" level=info msg="Relative import works: Hello, VU-1!, version=1.0.0" source=console
time="2026-01-20T23:54:20Z" level=info msg="Relative import works: Hello, VU-1!, version=1.0.0" source=console
iterations: 2 (completed successfully)
```

I also tested with 20 VUs and 100 iterations to verify the behavior is consistent under load pressure. The module caching and locking mechanism performed identically regardless of VU count, confirming that the freezing is not affected by runtime pressure.

Dynamic import() syntax is not supported in k6. Attempting to use it results in the error "dynamic modules not enabled in the host program" which comes from the Sobek JavaScript runtime that k6 uses. This is separate from the require() blocking and represents a deliberate decision not to enable dynamic module loading in the runtime configuration.

The module cache is stored in the ModuleResolver.cache field as a map from specifier string to moduleCacheElement. When Lock() is called, any subsequent resolve() call for a specifier not already in this cache will fail. Modules that were cached during init remain accessible because the cache persists and VU code can instantiate cached modules through the ModuleSystem without needing to re-resolve them.
