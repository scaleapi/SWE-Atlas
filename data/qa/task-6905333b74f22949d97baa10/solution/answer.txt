
I started by exploring the maddy repository to understand its structure and how SMTP connections flow through the system at runtime.

To understand which Go types and source files handle incoming SMTP connections, I examined the internal/endpoint/smtp package. The core architecture is defined in internal/endpoint/smtp/smtp.go. Looking at the code, I found two primary types that manage the SMTP endpoint:

The `Endpoint` type is the top-level structure that sets up TCP listeners and manages the overall SMTP server lifecycle. Here's its definition from smtp.go:

```go
// From internal/endpoint/smtp/smtp.go
type Endpoint struct {
    name     string
    serv     *smtp.Server
    listeners []net.Listener
    
    Log      log.Logger
    // ... other fields for delivery targets, auth, etc.
}
```

When a new SMTP connection arrives, the Endpoint creates a `Session` instance to handle that specific connection. The Session type implements the SMTP protocol state machine, managing the conversation through MAIL FROM, RCPT TO, and DATA commands:

```go
// From internal/endpoint/smtp/smtp.go
type Session struct {
    endp        *Endpoint
    sessionCtx  context.Context
    msgMeta     *module.MsgMetadata
    delivery    module.Delivery
    log         log.Logger
    // ... fields tracking connection state
}
```

Each Session processes one SMTP transaction at a time. When a client sends MAIL FROM, the Session's MailFrom method is called. For each RCPT TO command, the RcptTo method handles recipient validation and adds them to the delivery pipeline. Finally, the Data method receives the message body and commits it through the configured delivery target.

These logs reveal important patterns about how maddy tracks message lifecycle. The **module name** that appears in the log prefix for SMTP operations is `smtp` (visible as `smtp:` at the start of each log line). The **msg_id format** is an 8-character lowercase hexadecimal string (e.g., `9a84843e`, `91b386b2`, `af4d66ba`).

The complete log line format when a recipient is successfully added is `smtp: RCPT ok {"msg_id":"...","rcpt":"..."}`. The **JSON fields** in this RCPT ok log are: `msg_id` (the message identifier) and `rcpt` (the recipient email address).

A successful delivery follows a clear sequence: first `incoming message` appears with metadata like msg_id, sender, and source connection details, then each recipient gets a `RCPT ok` confirmation, and finally the message is marked as `accepted` when the entire transaction commits successfully. 

In contrast, the abort paths demonstrate error handling. When data transmission fails (like an unexpected EOF), we see `DATA error` with a specific reason before the `aborted` log. If the client disconnects during recipient processing, we skip straight to `aborted` without completing the data phase. This distinction helps diagnose whether failures occur during recipient validation versus message transmission.

Understanding the delivery pipeline architecture required examining how messages flow from SMTP acceptance through to final delivery. When the SMTP Session accepts a message, it doesn't deliver it directly. Instead, it passes the message to a configured delivery target, which could be a queue, a routing module, or a direct delivery mechanism.

I focused on the queue target since it's central to the retry behavior the prompt asks about. Examining internal/target/queue/queue.go, I found that the queueDelivery.Commit method doesn't perform immediate delivery. Instead, it schedules work on a time-wheel data structure for deferred execution. Later, when the scheduled time arrives, the queue's dispatch method spawns a new goroutine to actually attempt delivery to the downstream target. This asynchronous architecture is key to understanding how retries work without blocking the SMTP acceptance path.

The MX authentication failure test reveals exactly how maddy reports when it cannot verify the authenticity of an MX record. The error message is `Failed to estabilish the MX record (mx.example.invalid.) authenticity` (note: the typo "estabilish" appears to be in the actual maddy source code). The test output shows the complete error structure including SMTP protocol fields: `smtp_code:550` indicates a permanent failure, `smtp_enchcode:[5 4 0]` represents the enhanced status code 5.4.0 (routing/delivery problem), and the full SMTP reply message. 

When an SMTP client connects to send mail through maddy and this error occurs, they would receive the literal SMTP response line: `550 5.4.0 No usable MXs, last err: Failed to estabilish the MX record (mx.example.invalid.) authenticity`. This permanent error code (5xx) tells the client not to retry, since the MX authenticity problem isn't transient.

The TLS fallback test demonstrates maddy's resilience when encountering certificate issues. The exact log message when the system falls back from TLS to plaintext is: `remote: TLS error, falling back to plaintext {"domain":"example.invalid","msg_id":"2176ec5872ed2b87d832b4070e88232bd94ac7d3","mx":"mx.example.invalid.","reason":"smtpconn: x509: certificate signed by unknown authority"}`. The test shows that after logging the TLS error, the delivery proceeds through Body and Commit phases successfully. This fallback behavior prioritizes mail delivery over encryption when certificates aren't verifiable, though in production this could be configured more strictly with the requireTLS option.

The trace data provides definitive proof about the concurrency model. First, we see `queue/Body` and `queue/Commit` regions executing on goroutine 21 (g=21). This architecture definitively proves that queue delivery does NOT continue in the same goroutine as SMTP acceptance. Instead, the queue uses an asynchronous model: the SMTP goroutine quickly hands off to the queue (which schedules work and returns), then separate worker goroutines perform the actual delivery attempts. This design prevents slow or failing downstream deliveries from blocking SMTP connections, allowing maddy to accept messages quickly even when final delivery is delayed.
