I started digging into how SFTPGo handles external utilities like rsync, and pretty quickly found myself looking at the parseCommandPayload function in sftpd/ssh_cmd.go. What caught my attention was how simple the parsing logic is, it just uses strings.Split on spaces. I wanted to see what this actually means in practice, so I wrote a quick test:

```bash
cat > /tmp/test_parsing.go << 'EOF'
package main

import (
    "fmt"
    "strings"
)

func parseCommandPayload(command string) (string, []string, error) {
    parts := strings.Split(command, " ")
    if len(parts) < 2 {
        return parts[0], []string{}, nil
    }
    return parts[0], parts[1:], nil
}

func getDestPath(args []string) string {
    if len(args) == 0 {
        return ""
    }
    destPath := args[len(args)-1]
    if strings.HasPrefix(destPath, "'") && strings.HasSuffix(destPath, "'") {
        destPath = strings.TrimPrefix(destPath, "'")
        destPath = strings.TrimSuffix(destPath, "'")
    }
    return destPath
}

func main() {
    testCases := []string{
        "rsync --server -e.Lsfx . /uploads",
        "rsync --server --rsync-path=/bin/sh . /uploads",
        "rsync --server . '/uploads/my file.txt'",
    }
    for _, cmd := range testCases {
        name, args, _ := parseCommandPayload(cmd)
        destPath := getDestPath(args)
        fmt.Printf("Input: %s\n", cmd)
        fmt.Printf("  Name: %s, Args: %v, DestPath: %s\n\n", name, args, destPath)
    }
}
EOF
go run /tmp/test_parsing.go
```

Output:
Input: rsync --server -e.Lsfx . /uploads
  Name: rsync, Args: [--server -e.Lsfx . /uploads], DestPath: /uploads

Input: rsync --server --rsync-path=/bin/sh . /uploads
  Name: rsync, Args: [--server --rsync-path=/bin/sh . /uploads], DestPath: /uploads

Input: rsync --server . '/uploads/my file.txt'
  Name: rsync, Args: [--server . '/uploads/my file.txt'], DestPath: file.txt'

That second test case is interesting . the "--rsync-path=/bin/sh" flag just gets passed through as a regular argument. There's no filtering happening here at all. The function splits on spaces and that's it. Whatever arguments the client sends become part of the args slice.

Before diving into what happens downstream, I should clarify where filtering actually does occur. Looking at processSSHCommand in ssh_cmd.go, the command name (like "rsync" or "scp") gets checked against an allowlist using utils.IsStringInSlice(name, enabledSSHCommands). The allowlist is defined in sftpd.go:

```bash
cat > /tmp/test_allowlist.go << 'EOF'
```

So the filtering rule is simple: only the command name gets validated against a whitelist. Commands like "ls", "cat", "rm", or "bash" would be rejected. But here's the critical gap: once a command passes the name check, none of its arguments get filtered. The allowlist only gates which executables can run, not what flags they receive.

This got me wondering about what happens to these arguments downstream. Looking at getSystemCommand in ssh_cmd.go, I can see the flow as the command gets parsed, then the last argument (the destination path) gets validated through ResolvePath. But everything else? It just passes straight through to exec.Command. So I tested what this means for dangerous flags:

```bash
cat > /tmp/test_dangerous_flags.go << 'EOF'
```

The quote handling is also broken in an interesting way. Since it's just splitting on spaces, quoted strings don't stay together.

```bash
cat > /tmp/test_quote_parsing.go << 'EOF'
```

The quotes are treated as literal characters. A path like '/uploads/my file.txt' gets split into two pieces with the quotes still attached. This isn't really an attack vector on its own, but it shows how minimal the parsing is, there's no attempt to handle structured input properly.

What really matters is the order of operations. I wanted to understand exactly when different checks happen in the processing chain:

```bash
cat > /tmp/test_permission_order.go << 'EOF'
package main

import (
    "fmt"
    "path/filepath"
    "strings"
)

    fmt.Println("Step 1: Args from parseCommandPayload")
    fmt.Printf("  args = %v\n", args)
    for _, arg := range args {
        if strings.Contains(arg, "--rsync-path") {
            fmt.Printf("  WARNING: Malicious flag detected: %s\n", arg)
        }
    }
    
    fmt.Println("Step 2-3: ResolvePath on last arg only")
    destPath := args[len(args)-1]
    resolved := filepath.Join(homeDir, filepath.Clean(destPath))
    args[len(args)-1] = resolved
    fmt.Printf("  resolved path = %s\n", resolved)
    
    fmt.Println("Step 4: Permission check")
    fmt.Println("  Checking symlink permissions...")
```

The malicious flag is already in the args array before any permission checking happens. The permission system can't help because it only looks at path permissions, not at what flags are being passed. By the time we get to step 4, the damage is done , the flag is already going to be passed to exec.Command.

I also looked at the symlink protections. The server tries to add --safe-links or --munge-links to prevent symlink-based escapes, but there's a problem with how it checks if the flag is already present:

```bash
cat > /tmp/test_symlink_check.go << 'EOF'
package main

import "fmt"

func IsStringInSlice(obj string, list []string) bool {
    for _, v := range list {
        if v == obj {
            return true
        }
    }
    return false
}

func injectSymlinkFlags(args []string) []string {
    result := make([]string, len(args))
    copy(result, args)
    if !IsStringInSlice("--safe-links", result) {
        result = append([]string{"--safe-links"}, result...)
    }
    return result
}

func main() {
    testCases := [][]string{
        {"--server", ".", "/uploads"},
        {"--server", "--safe-links", ".", "/uploads"},
        {"--server", "--safe-links=0", ".", "/uploads"},
    }

    for _, args := range testCases {
        found := IsStringInSlice("--safe-links", args)
        result := injectSymlinkFlags(args)
        fmt.Printf("Args: %v\n", args)
        fmt.Printf("  Exact match '--safe-links': %v\n", found)
        fmt.Printf("  After injection: %v\n\n", result)
    }
}
EOF
go run /tmp/test_symlink_check.go
```

Output:
Args: [--server . /uploads]
  Exact match '--safe-links': false
  After injection: [--safe-links --server . /uploads]

Args: [--server --safe-links . /uploads]
  Exact match '--safe-links': true
  After injection: [--server --safe-links . /uploads]

Args: [--server --safe-links=0 . /uploads]
  Exact match '--safe-links': false
  After injection: [--safe-links --server --safe-links=0 . /uploads]

See that last case, if you send --safe-links=0, the exact string match for "--safe-links" fails, so the server adds "--safe-links" to the beginning. But now rsync sees both flags, and depending on how rsync handles duplicates, the --safe-links=0 might take precedence and disable the protection entirely.

There's also an issue with the path containment check in isSubDir:

Using strings.HasPrefix means /home/user_attacker passes as a subdirectory of /home/user because it starts with the same prefix. That's a classic path traversal bug.

Putting this all together, the security model has some serious holes. The command name gets checked against an allowlist, and the final path argument goes through ResolvePath, but everything in between , all those flags and intermediate arguments , just passes straight through to exec.Command with no validation. The parseCommandPayload function uses basic string splitting, quotes aren't handled as grouping delimiters, and the symlink protections can be bypassed with flag value tricks. The whole thing adds up to CVE-2025-24366, which lets attackers inject arbitrary rsync flags including --rsync-path for command execution.
