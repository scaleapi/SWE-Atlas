When large amounts of terminal graphics data arrive in kitty faster than the system can process them, the data flow management occurs through several interconnected mechanisms in the child monitor and graphics manager subsystems. The primary buffering decisions happen in child-monitor.c where the IO thread manages reading from child processes and writing responses back.

The IO loop in child-monitor.c uses poll to wait for events on child file descriptors. The vt_parser_has_space_for_input function determines whether the parser buffer can accept more data. When the buffer fills up, the POLLIN flag is not set for that child, causing the system to naturally slow down data intake.


The read_bytes function handles data intake with explicit buffer management. When vt_parser_create_write_buffer returns with zero available space, the function returns true to indicate the child is not dead but simply cannot accept more data yet.

```c
// File: kitty/child-monitor.c, lines 1336-1356
static bool
read_bytes(int fd, Screen *screen) {
    ssize_t len;
    size_t available_buffer_space;

    uint8_t *buf = vt_parser_create_write_buffer(screen->vt_parser, &available_buffer_space);
    if (!available_buffer_space) return true;

    while(true) {
        len = read(fd, buf, available_buffer_space);
        if (len < 0) {
            if (errno == EINTR || errno == EAGAIN) continue;
            if (errno != EIO) perror("Call to read() from child fd failed");
            vt_parser_commit_write(screen->vt_parser, 0);
            return false;
        }
        break;
    }
    vt_parser_commit_write(screen->vt_parser, len);
    return len != 0;
}
```

The graphics subsystem manages its own storage through the GraphicsManager structure defined in graphics.c. When images arrive faster than they can be processed, the storage quota enforcement kicks in. The apply_storage_quota function removes older unreferenced images to make room for new ones.

```c
// File: kitty/graphics.c, lines 289-300
static void
apply_storage_quota(GraphicsManager *self, size_t storage_limit, id_type currently_added_image_internal_id) {
    // First remove unreferenced images, even if they have an id
    remove_images(self, trim_predicate, currently_added_image_internal_id);
    if (self->used_storage < storage_limit) return;

    HASH_SORT(self->images, oldest_img_first);
    while (self->used_storage > storage_limit && self->images) {
        remove_image(self, self->images);
    }
    if (!self->images) self->used_storage = 0;  // sanity check
}
```

The write_to_child function handles actual output with non blocking behavior. When write returns EWOULDBLOCK or EAGAIN, the function breaks out of the loop and waits for the next poll cycle rather than blocking.

```c
// File: kitty/child-monitor.c, lines 1442-1466 (debug macros omitted for clarity)
static void
write_to_child(int fd, Screen *screen) {
    size_t written = 0;
    ssize_t ret = 0;
    screen_mutex(lock, write);
    while (written < screen->write_buf_used) {
        ret = write(fd, screen->write_buf + written, screen->write_buf_used - written);
        if (ret > 0) {
            written += ret;
        }
        else if (ret == 0) {
            // could mean anything, ignore
            break;
        } else {
            if (errno == EINTR) continue;
            if (errno == EWOULDBLOCK || errno == EAGAIN) break;
            perror("Call to write() to child fd failed, discarding data.");
            written = screen->write_buf_used;
        }
    }
```

The system adapts quietly to sustained pressure through the polling mechanism. When more data arrives than can be immediately processed, the poll timeout and event flags naturally throttle intake. The has_pending_wakeups flag defers main loop wake ups to batch process accumulated data.

The rendering cycle also participates in flow control. The render function checks time since last render against repaint_delay and defers rendering if called too soon. This prevents the renderer from consuming CPU cycles needed for data processing.

```c
// File: kitty/child-monitor.c, lines 870-878
static void
render(monotonic_t now, bool input_read) {
    EVDBG("input_read: %d, check_for_active_animated_images: %d", input_read, global_state.check_for_active_animated_images);
    static monotonic_t last_render_at = MONOTONIC_T_MIN;
    monotonic_t time_since_last_render = last_render_at == MONOTONIC_T_MIN ? OPT(repaint_delay) : now - last_render_at;
    if (!input_read && time_since_last_render < OPT(repaint_delay)) {
        set_maximum_wait(OPT(repaint_delay) - time_since_last_render);
        return;
    }
```

The maximum_wait variable coordinates timing across subsystems. Various components call set_maximum_wait to indicate when they need attention, and the main loop uses this to balance responsiveness with efficiency.

Observable runtime behavior under pressure shows graceful degradation rather than hard failures. The terminal remains responsive to user input while background data processing catches up. Graphics operations complete in order but with visible timing delays under heavy load.

The talk thread provides separate backpressure for remote control connections. The prune_peers function removes finished peers when their read is complete and no pending writes remain. The message queue grows dynamically via ensure_space_for allocations but is naturally bounded by the rate at which peers connect and send messages.

Decisions about data flow and backpressure are distributed across these components rather than centralized, with each layer implementing appropriate buffering and throttling for its domain.
