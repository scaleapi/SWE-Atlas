```python
#!/usr/bin/env python3
import socket
import threading

def handle_client(conn, addr):
    print(f"[MOCK] Connection from {addr}", flush=True)
    try:
        conn.send(b"220 mock-server.local ESMTP Mock\r\n")
        while True:
            data = conn.recv(1024)
            if not data:
                break
            cmd = data.decode('utf-8', errors='ignore').strip().upper()
            if cmd.startswith('EHLO') or cmd.startswith('HELO'):
                conn.send(b"250-mock-server.local\r\n250-PIPELINING\r\n250-8BITMIME\r\n250 SIZE 35000000\r\n")
            elif cmd.startswith('MAIL FROM'):
                conn.send(b"250 2.0.0 OK\r\n")
            elif cmd.startswith('RCPT TO'):
                conn.send(b"451 4.7.1 Temporary failure, please retry later\r\n")
            elif cmd.startswith('QUIT'):
                conn.send(b"221 Bye\r\n")
                break
    finally:
        conn.close()

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server.bind(('127.0.0.1', 19996))
server.listen(5)
print("[MOCK] Starting RCPT temp error SMTP server on 127.0.0.1:19996", flush=True)
while True:
    conn, addr = server.accept()
    threading.Thread(target=handle_client, args=(conn, addr), daemon=True).start()
```

then I tested timeout behavior by creating a mock server that accepts connections but never responds to observe how long maddy waits before timing out.

```python
#!/usr/bin/env python3
import socket
import time

def handle_client(conn, addr):
    print(f"[TIMEOUT-MOCK] Accepted connection from {addr} at {time.strftime('%H:%M:%S')}", flush=True)
    # Accept connection but never send any response - just hang
    while True:
        time.sleep(60)

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server.bind(('127.0.0.1', 19998))
server.listen(5)
print("[TIMEOUT-MOCK] Timeout SMTP server listening on 127.0.0.1:19998", flush=True)
while True:
    conn, addr = server.accept()
    threading.Thread(target=handle_client, args=(conn, addr), daemon=True).start()
```

```text
[debug] queue: using message ID = 46978df5-1        {"msg_id":"46978df5"}
```

still no timeout error. the delivery attempt was still hanging after 5+ minutes. this means when a server accepts connections but never responds, maddy doesn't have a read timeout configured and will wait indefinitely. this is because the default net.Dialer in Go has no timeout set, and maddy's smtp_downstream module doesn't configure one. this could cause queue starvation if all parallelism slots get blocked on hanging servers.

to test what happens when multiple messages are queued to different destinations with varying response times, I configured max_parallelism 2 and sent three messages simultaneously - one to a working server (msg_a), one to the hanging server (msg_b), and one to the 451 server (msg_c).

```bash
grep -E "semaphore|delivery attempt|will retry|delivered" /tmp/maddy-queue-test/maddy_mixed.log
```

```text
[debug] queue: waiting on delivery semaphore for msg_a
[debug] queue: delivery semaphore acquired for msg_a
[debug] queue: waiting on delivery semaphore for msg_b  
[debug] queue: delivery semaphore acquired for msg_b
[debug] queue: waiting on delivery semaphore for msg_c
[debug] queue: delivery attempt #1        {"msg_id":"msg_a"}
[debug] queue: delivery attempt #1        {"msg_id":"msg_b"}
queue: delivered        {"msg_id":"msg_a","attempt":1}
[debug] queue: delivery semaphore acquired for msg_c
[debug] queue: delivery attempt #1        {"msg_id":"msg_c"}
queue: will retry        {"attempts_count":1,"msg_id":"msg_c","next_try_delay":"14m59.999s"}
```

this shows that msg_a and msg_b acquired semaphores first (max_parallelism=2), while msg_c had to wait. when msg_a completed quickly, msg_c was able to acquire the semaphore and start its delivery attempt. meanwhile msg_b continued blocking on the hanging server. the retry timing for msg_c is calculated from when its delivery attempt actually started, not from when it was queued. so a blocking message does not directly affect other messages' retry timing, but it can delay when their first attempt begins if all parallelism slots are occupied. this could lead to queue starvation if max_parallelism slots are all blocked on hanging servers.
