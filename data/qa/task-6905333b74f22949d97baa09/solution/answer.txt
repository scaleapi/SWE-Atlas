I started by building TruffleHog from source and exploring its architecture through code analysis and practical experiments:

```
$ go build -o trufflehog .
```

Here's what I discovered about its detection system.

When I examined how detector configurations are loaded, I found they are compiled directly into the binary rather than loaded from external files. Looking at pkg/engine/defaults/defaults.go, there's a buildDetectorList() function starting that explicitly instantiates each detector:

```go
func buildDetectorList() []detectors.Detector {
    return []detectors.Detector{
        &abyssale.Scanner{},
        &abuseipdb.Scanner{},
        &accuweather.Scanner{},
        // ... continues for hundreds more
    }
}
```

I counted 857 Scanner{} instances in this file, meaning there are over 850 detectors compiled into the binary. Each detector is a Go struct implementing the detectors.Detector interface. No external configuration files are loaded for detector definitions at startup – the --config flag is only for custom rules, not for loading default detectors.

Running the binary with --help reveals the available commands and flags:

```
$ ./trufflehog --help
```

The help output shows subcommands for different scan sources: git, github, gitlab, filesystem, s3, gcs, syslog, circleci, docker, travisci, postman, elasticsearch, jenkins, and huggingface. For detector capabilities, the key flags are:

1. --include-detectors: Comma separated list of detector types to include. Protobuf name or IDs may be used, as well as ranges. Defaults to "all".
2. --exclude-detectors: Comma separated list of detector types to exclude. IDs defined here take precedence over the include list.
3. --no-verification: Don't verify the results.
4. --only-verified: Only output verified results.
5. --print-avg-detector-time: Print the average time spent on each detector.

When I ran TruffleHog with verbose logging, the initialization messages showed how detectors get registered:

```
$ ./trufflehog filesystem ./test-repo --json --log-level=4 2>&1 | head -20

engine initialized
setting up aho-corasick core
set up aho-corasick core
starting scanner workers     {"count": 12}
starting detector workers    {"count": 96}
starting verificationOverlap workers  {"count": 12}
starting notifier workers    {"count": 12}
```

The worker counts are calculated as multipliers of the --concurrency flag (which defaults to NumCPU). Looking at pkg/engine/engine.go, I found numWorkers := e.concurrency * e.detectorWorkerMultiplier, where the multiplier defaults to 8. This explains why detector workers = 96 on my 12-core system (12 × 8). The higher multiplier for detector workers accounts for network I/O bound verification operations.

For verification setup, I examined the build dependencies and found HTTP client libraries that confirm network verification. In pkg/common/http.go, there's extensive use of github.com/hashicorp/go-retryablehttp:

```go
import (
    "crypto/tls"
    "crypto/x509"
    "net/http"
    "github.com/hashicorp/go-retryablehttp"
)
```

The HTTP client features include retry mechanisms with configurable retries (default: 3), custom User-Agent headers set to "TruffleHog", a default timeout of 5 seconds, and pinned certificate pools for TLS security.

Verification happens in parallel, not sequentially. In pkg/engine/engine.go, the startDetectorWorkers function spawns multiple goroutines:

```go
func (e *Engine) startDetectorWorkers(ctx context.Context) {
    numWorkers := e.concurrency * e.detectorWorkerMultiplier
    ctx.Logger().V(2).Info("starting detector workers", "count", numWorkers)
    for worker := 0; worker < numWorkers; worker++ {
        e.wgDetectorWorkers.Add(1)
        go func() {
            defer e.wgDetectorWorkers.Done()
            e.detectorWorker(ctx)
        }()
    }
}
```

Each worker processes chunks concurrently through channels, allowing multiple verification requests to happen simultaneously.

For the JSON output schema, I examined pkg/output/json.go. The structure for a finding includes:

```go
v := &struct {
    SourceMetadata        *source_metadatapb.MetaData
    SourceID              sources.SourceID
    SourceType            sourcespb.SourceType
    SourceName            string
    DetectorType          detectorspb.DetectorType
    DetectorName          string
    DetectorDescription   string
    DecoderName           string
    Verified              bool
    VerificationError     string `json:",omitempty"`
    VerificationFromCache bool
    Raw                   string
    RawV2                 string
    Redacted              string
    ExtraData             map[string]string
    StructuredData        *detectorspb.StructuredData
}
```

The verification status is captured by the Verified boolean field plus VerificationError (a string that's omitted if empty). There are no confidence scores – results are binary: verified, unverified, or unknown. Rich metadata about the location is included through SourceMetadata, which contains file paths, line numbers, commit information, timestamps, and author emails depending on the source type.

Regarding repository traversal, I created a test repository with mixed file types to observe scanning behavior. Git history is processed sequentially by commit, but chunks within commits are processed in parallel across workers. In pkg/gitparse/gitparse.go, I found the default limits:

```go
const (
    defaultDateFormat    = "Mon Jan 2 15:04:05 2006 -0700"
    defaultMaxDiffSize   int64 = 2 * 1024 * 1024 * 1024 // 2GB
    defaultMaxCommitSize int64 = 2 * 1024 * 1024 * 1024 // 2GB
)
```

From my test scans with verbose logging (--log-level=5), I observed how TruffleHog decides which files to scan. The logs show explicit file handling messages:

```
$ ./trufflehog git file:///tmp/test_repo --log-level=5 2>&1 | grep -E "handling|skipping|mime"

handling binary file    {"commit": "f462a44", "path": "binary.dat"}
dataErrChan closed, all chunks processed    {"path": "binary.dat", "mime": "application/octet-stream"}
```

Binary files are scanned by default - the logs show "handling binary file" with MIME type detection ("application/octet-stream"). When using --force-skip-binaries, the logs instead show:

```
skipping binary file    {"commit": "f462a44", "path": "/tmp/test_repo"}
```

Files can be excluded using --exclude-paths with regex patterns. Additional flags like --force-skip-binaries and --force-skip-archives are available to skip those file types. Archives are decompressed and scanned recursively, configurable with --archive-max-size and --archive-max-depth.

The detector architecture uses an interface-based design where each detector implements the Detector interface defined in pkg/detectors/detectors.go:

```go
type Detector interface {
    FromData(ctx context.Context, verify bool, data []byte) ([]Result, error)
    Keywords() []string
    Type() detectorspb.DetectorType
    Description() string
}
```

The Keywords() method returns strings used for efficient pre-filtering. For example, the Stripe detector in pkg/detectors/stripe/stripe.go returns ["k_live"] as its keyword. TruffleHog uses the Aho-Corasick algorithm (implemented in pkg/engine/ahocorasick/) to build a trie from all detector keywords at startup. This enables O(n) substring matching across all 850+ detectors simultaneously, providing efficient pre-filtering before individual detectors perform their regex pattern matching.

The detection pipeline flows as follows: sources produce chunks, which pass through decoders, then through the Aho-Corasick filter to identify potentially matching detectors. Only those detectors receive the chunk for full regex pattern matching. If verification is enabled, matching credentials trigger HTTP requests to validate them. Results are deduplicated and dispatched to the configured output format.

Throughout my investigation, I did not modify any source files. All test artifacts including the test repository were cleaned up after the experiments.
