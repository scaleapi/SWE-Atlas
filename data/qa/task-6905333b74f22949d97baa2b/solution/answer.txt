When I read this project's SSH exec handling, the part that matters for the security boundary is simple to state but easy to get wrong in practice: the server does not execute a "shell string", it constructs an argv vector, it normalizes and resolves the destination path before putting it into argv, and then it conditionally injects rsync safety options depending on the user's permissions. The quickest way to settle the "is it shell like" and "do the guardrails reach the real argv" suspicions is to observe what argv an OS process actually receives at runtime under two different permission contexts.

The "shell like execution" suspicion is wrong in this codebase. The system command is created with exec.Command, which passes argv elements directly to the OS process without shell interpretation. This happens in sftpd/ssh_cmd.go in the getSystemCommand function.

```go
cmd := exec.Command(c.command, args...)
uid := c.connection.User.GetUID()
gid := c.connection.User.GetGID()
cmd = wrapCmd(cmd, uid, gid)
command.cmd = cmd
command.realPath = path
return command, nil
```

The "path guardrails are superficial" suspicion is partly wrong. The destination argument that reaches the OS process is not simply whatever the client typed. The last argument is first normalized by getDestPath, which trims quotes, converts to forward slashes, forces a leading slash when needed, and applies path.Clean.

```go
func (c *sshCommand) getDestPath() string {
	if len(c.args) == 0 {
		return ""
	}
	destPath := strings.Trim(c.args[len(c.args)-1], "'")
	destPath = strings.Trim(destPath, "\"")
	destPath = filepath.ToSlash(destPath)
	if !path.IsAbs(destPath) {
		destPath = "/" + destPath
	}
	result := path.Clean(destPath)
	if strings.HasSuffix(destPath, "/") && !strings.HasSuffix(result, "/") {
		result += "/"
	}
	return result
}
```

After normalization, the path is resolved to a filesystem path under the user's home directory using ResolvePath, and then the last argument in argv is replaced with this resolved path.

```go
if len(c.args) > 0 {
	var err error
	sshPath := c.getDestPath()
	path, err = c.connection.fs.ResolvePath(sshPath)
	if err != nil {
		return command, err
	}
	args = args[:len(args)-1]
	args = append(args, path)
}
```

The "safety flags are a patch rather than a policy" suspicion is partly right. For rsync, the code ensures that a safety option exists, but it does not remove conflicts or enforce precedence. If the client supplies conflicting flags up front, the server does not rewrite them into a single unambiguous policy.

```go
if c.command == "rsync" {
	if c.connection.User.HasPerm(dataprovider.PermCreateSymlinks, c.getDestPath()) {
		if !utils.IsStringInSlice("--safe-links", args) {
			args = append([]string{"--safe-links"}, args...)
		}
	} else {
		if !utils.IsStringInSlice("--munge-links", args) {
			args = append([]string{"--munge-links"}, args...)
		}
	}
}
```

The logic checks if the flag is already present before adding it, but if the client supplies both --safe-links and --munge-links, the server does nothing to resolve the contradiction. The argv that reaches rsync can contain both flags.

To observe what actually reaches an OS process at runtime, a probe program can construct the argv using the same code paths and execute a stub that captures its received argv. The probe uses two permission contexts: one with full permissions including create_symlinks, and one without.

```go
func main() {
	workDir := os.Getenv("WORKDIR")
	homeDir := filepath.Join(workDir, "home")
	stubDir := filepath.Join(workDir, "stub")
	
	perms1 := map[string][]string{"/": {"*"}}
	runCase("benign", homeDir, stubDir, workDir, perms1, []string{"--archive", "src/", "dest/"})
	
	perms2 := map[string][]string{"/": {"download", "upload"}}
	runCase("adversarial", homeDir, stubDir, workDir, perms2, []string{"--archive", "--safe-links", "--munge-links", "--", "--fake", "src/", "dest/;echo_PWN"})
}
```

The stub is a shell script that writes its argv to a file, proving what the OS process actually receives.

Runtime output from the benign case shows the permission dependent safety injection. With full permissions that include create_symlinks, the server injects --safe-links at the front of argv. The destination that reaches the process is a resolved filesystem path under the home directory, not the raw client string.

```
CASE benign
BUILT_ARGV
/usr/bin/rsync
--safe-links
--archive
src/
/tmp/tmp.xIXjOZYhJB/home/dest
STUB_CAPTURE
/tmp/tmp.xIXjOZYhJB/stub/rsync
--safe-links
--archive
src/
/tmp/tmp.xIXjOZYhJB/home/dest
```

Runtime output from the adversarial case demonstrates three things at once. The destination contains ;echo_PWN and it reaches the stub as part of a single argv element. This is exactly what you expect from direct exec and exactly what you would not expect if the server were handing a shell a single command string. If this were shell execution, the semicolon would be parsed as a command separator.

```
CASE adversarial
BUILT_ARGV
/usr/bin/rsync
--archive
--safe-links
--munge-links
--
--fake
src/
/tmp/tmp.xIXjOZYhJB/home/dest/;echo_PWN
STUB_CAPTURE
/tmp/tmp.xIXjOZYhJB/stub/rsync
--archive
--safe-links
--munge-links
--
--fake
src/
/tmp/tmp.xIXjOZYhJB/home/dest/;echo_PWN
```

The argv contains both --safe-links and --munge-links, showing that the server does not resolve conflicts into a single policy when the client supplies contradictory flags. The --fake argument survives the -- separator and reaches the process, showing that client supplied options pass through to the real argv.

This evidence proves what argv reaches the OS process and shows a place where "safety flags" are enforced as string presence rather than as an unambiguous policy. It does not prove a privilege boundary break without pinning down rsync semantics under conflicting flags, but the conflicting flags behavior is worth noting as a potential weakness.

The experiment uses a temporary helper file with build tag taskgen that exposes the argv construction logic. The helper and all temporary files are created in /tmp and cleaned up after the experiment. Running git status confirms the repository has no modifications to tracked files.
