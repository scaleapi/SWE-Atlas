To understand how paperless ngx processes documents at runtime I first need to start the application with all required services. The system architecture relies on three main services coordinated through supervisord. The gunicorn web server handles HTTP requests and serves the web interface on port 8000. The document consumer continuously monitors the consume directory for new files and processes them through the ingestion pipeline. The scheduler known as qcluster manages background tasks including classifier training and index optimization. These three services communicate through Redis which acts as a message broker enabling asynchronous task processing and real time updates to thconnected clients through WebSocket channels.

I installed the Python dependencies first by running pip install

pip install -r requirements.txt

The installation completed with 127 packages including django, whoosh for search indexing, scikit learn for the ML classifier, and celery for task scheduling. The scikit learn package is particularly important because it provides the MLPClassifier neural network used to automatically categorize documents based on their content. Whoosh provides full text search capabilities allowing users to find documents by searching their OCR extracted content.

The documents_document table is the central table storing all document metadata including title, content extracted via OCR, checksums for duplicate detection, timestamps, and foreign keys to correspondent and document type records.

The services involved in this single document ingestion span multiple modules working together in a coordinated pipeline. The consumer service defined in src/documents/consumer.py orchestrates the entire process detecting new files and managing the workflow. The parsers module in src/documents/parsers.py determines which parser to use based on the detected MIME type. For PDF documents the paperless_tesseract parser in src/paperless_tesseract/parsers.py handles the actual OCR extraction converting images to searchable text. The classifier module in src/documents/classifier.py loads any trained machine learning model to automatically assign tags correspondents and document types. The signals handlers in src/documents/signals/handlers.py process post consumption events like adding inbox tags and updating the search index. Finally the index module in src/documents/index.py adds the document to the Whoosh full text search index enabling content based searches.

The storage system uses three separate directories each serving a distinct purpose. The originals directory stores the uploaded file exactly as received without any modifications preserving the original document for archival purposes. The archive directory contains a searchable PDF version with embedded OCR text allowing the document to be searched even if the original was a scanned image. The classifier training behavior is controlled by the train_classifier function in src/documents/tasks.py at lines 38 through 63. The training only occurs when specific conditions are met. First there must be at least one Tag DocumentType or Correspondent configured with matching_algorithm set to MATCH_AUTO which has a value of 6. Second the training data must have changed since the last training session which is determined by comparing SHA1 hashes of the document content and labels.

if not Tag.objects.filter matching_algorithm equals Tag.MATCH_AUTO exists
and not DocumentType.objects.filter matching_algorithm equals Tag.MATCH_AUTO exists
and not Correspondent.objects.filter matching_algorithm equals Tag.MATCH_AUTO exists then
    return

The classifier does not retrain automatically on every upload which is an important optimization for performance. It only trains when there are matching models configured with MATCH_AUTO algorithm and when the training data has actually changed since the last training. The data hash comparison in classifier.py at line 163 prevents unnecessary retraining by comparing a SHA1 hash of all document content and labels. If the hash matches the previous training session the function returns False and skips retraining entirely.

if self.data_hash and new_data_hash equals self.data_hash then return False

The database tables that receive new rows during document ingestion form a relational structure. The documents_document table stores the main document record with fields including id as primary key, title extracted from filename or content, content containing the full OCR text, mime_type identifying the file format, checksum for duplicate detection, created and modified timestamps, added timestamp for when the document entered the system, filename and archive_filename for storage paths, and foreign keys to correspondent_id and document_type_id. The documents_log table stores all log entries for each consumption operation with fields for id, group UUID to correlate related log entries, message text, level indicating severity, and created timestamp. The documents_document_tags through table stores the many to many relationship between documents and tags. The django_admin_log table records administrative actions including document creation for audit purposes.

The filename pattern by default uses the generate_filename function from src/documents/file_handling.py at line 128. When PAPERLESS_FILENAME_FORMAT environment variable is not set the system defaults to a seven digit zero padded primary key followed by the file extension. This simple pattern ensures unique filenames and maintains chronological ordering based on document ID.

filename equals doc.pk colon 07 plus counter_str plus filetype_str

This produces filenames like 0000001.pdf and 0000002.pdf. When PAPERLESS_FILENAME_FORMAT is configured the system supports a rich set of placeholders including title, correspondent, document_type, created with various date components like created_year created_month and created_day, added with similar date components, asn for archive serial number, tags dictionary accessor, and tag_list for comma separated tag names. This allows administrators to customize the storage structure based on their organizational needs.

The complete ingestion pipeline sequence observed from the logs reveals a sophisticated multi stage process

1. Consumer detects new file in consume directory through filesystem polling
2. Pre consume script runs if PAPERLESS_PRE_CONSUME_SCRIPT is configured
3. File hash calculated using MD5 to check for duplicates against existing documents
4. MIME type detected using the python magic library for accurate file type identification
5. Parser selected based on MIME type from registered parser classes
6. Document parsed and OCR performed extracting text content from images
7. Text content extracted from parser and stored for full text search
8. Date parsed from filename patterns or extracted from document content
9. Thumbnail generated as PNG preview image for web interface display
10. Classifier loaded and predictions made if a trained model exists
11. Document record created in database with all extracted metadata
12. Files copied to originals archive and thumbnails directories
13. Original file deleted from consume directory after successful processing
14. Post consume script runs if PAPERLESS_POST_CONSUME_SCRIPT is configured
15. Search index updated via Whoosh to enable content based queries
16. WebSocket notification sent to connected clients for real time UI updates

The three core services work together in a coordinated architecture. The consumer watches for files and processes them through the ingestion pipeline transforming raw documents into searchable indexed records. The scheduler runs background training tasks when triggered ensuring the classifier model stays current with new document patterns. Gunicorn serves the API and web interface allowing users to search browse and manage their document archive. Redis acts as the message broker coordinating between these services enabling asynchronous task execution and real time status updates through WebSocket channels.
