The wp-calypso monorepo provides four distinct test commands in package.json: test-client, test-packages, test-server, and test-build-tools. Each command targets different parts of the codebase with separate Jest configurations. To understand their runtime differences, I started by examining their configuration files.

The test-client command uses test/client/jest.config.js which sets rootDir to '../../client' and references @automattic/calypso-jest as its base. Notably, it includes a moduleNameMapper that redirects '^@automattic/calypso-config$' to '<rootDir>/server/config/index.js'. The setupFilesAfterEnv points to test/client/setup-test-framework.js.

The test-packages command uses test/packages/jest.config.js which specifies projects: [ '<rootDir>/packages/*/jest.config.js' ], making it run all package-level configs. Its moduleNameMapper only maps 'react-markdown' without any calypso-config redirection. Individual packages inherit from test/packages/jest-preset.js which sets setupFilesAfterEnv to test/packages/setup.js.

The test-server command uses test/server/jest.config.js with rootDir '../../client/server'. Its moduleNameMapper redirects '@automattic/calypso-config' to 'calypso/server/config', different from test-client's path.

The test-build-tools command uses test/build-tools/jest.config.js with rootDir '../../build-tools'. It inherits its configuration from @automattic/calypso-jest without any moduleNameMapper redirections for calypso-config, resolving dependencies through the custom monorepo resolver instead.

To observe runtime behavior, I selected the help-center package since it depends on @automattic/components, another internal monorepo package. Located at packages/help-center, its package.json shows dependencies including "@automattic/components": "workspace:^". The components package at packages/components has "calypso:src": "src/index.ts" in its package.json, alongside "main": "dist/cjs/index.js" and "module": "dist/esm/index.js".

Created a test file to trace module resolution:

The internal package resolves to SOURCE files at packages/components/src/index.ts, not the built dist/cjs/index.js despite that being the "main" field in package.json.

The resolution mechanism lives in packages/calypso-jest/src/module-resolver.js:

const enhancedResolve = require( 'enhanced-resolve' );

const resolver = enhancedResolve.create.sync( {
  extensions: [ '.json', '.js', '.jsx', '.ts', '.tsx' ],
  mainFields: [ 'calypso:src', 'main' ],
  conditionNames: [ 'calypso:src', 'node', 'require' ],
} );

module.exports = function ( request, options ) {
  return resolver( options.basedir, request ).replace( /\0#/g, '#' );
};

The mainFields array lists 'calypso:src' first, meaning it checks for that package.json field before 'main'. This causes all internal monorepo packages to resolve to their TypeScript source rather than compiled output.

This redirect happens in test/client/jest.config.js:

The moduleNameMapper uses a regex pattern matching exactly '@automattic/calypso-config' and substitutes the server config path. Checking what test-packages does:

No calypso-config mapping exists, so it uses normal resolution through the custom resolver which would load packages/calypso-config/src/index.ts based on the calypso:src field.

The actual files differ significantly. The browser version at packages/calypso-config/src/index.ts starts with:

if ( 'undefined' === typeof window ) {
  throw new Error( 'Trying to initialize the configuration outside of a browser context.' );
}

if ( ! window.configData ) {
  if ( 'development' === process.env.NODE_ENV ) {
    console.error(
      '%cNo configuration was found: ' +
      '%cPlease see ' +
      '%cpackages/calypso-config/README.md ' +
      '%cfor more information.',
      'color: red; font-size: 120%',
      'color: white;',
      'color: #0267ff;',
      'color: white'
    );
  }
  window.configData = {};
}

It expects window.configData and throws in node environments. The server version at client/server/config/index.js:

const configPath = require( 'path' ).resolve( __dirname, '..', '..', '..', 'config' );
const { default: createConfig } = require( '@automattic/create-calypso-config' );
const parser = require( './parser' );

const { serverData, clientData } = parser( configPath, {
  env: process.env.CALYPSO_ENV || process.env.NODE_ENV || 'development',
  enabledFeatures: process.env.ENABLE_FEATURES,
  disabledFeatures: process.env.DISABLE_FEATURES,
} );

module.exports = createConfig( serverData );
module.exports.clientData = clientData;

It reads configuration from the filesystem using parser, working in node environments without window dependency.

The source of these differences lies in setupFilesAfterEnv. For test-client, test/client/setup-test-framework.js contains:

import '@testing-library/jest-dom';

const nodeCrypto = require( 'node:crypto' );
const { ReadableStream, TransformStream } = require( 'node:stream/web' );
const { TextEncoder, TextDecoder } = require( 'util' );
const nock = require( 'nock' );

nock.disableNetConnect();

  if ( ! nock.isActive() ) {
    nock.activate();
  }
} );

afterAll( () => {
  nock.restore();
  nock.cleanAll();
} );

global.TextEncoder = TextEncoder;
global.TextDecoder = TextDecoder;

global.CSS = {
  supports: jest.fn(),
};

global.fetch = jest.fn( () =>
  Promise.resolve( {
    json: () => Promise.resolve(),

jest.mock( 'wpcom-proxy-request', () => ( {
  __esModule: true,

The test-server environment runs in node (no document/window) but has native fetch and TextEncoder from Node.js. The calypso-config resolves to client/server/config/index.js due to its moduleNameMapper.

The test/server/jest.config.js has:

moduleNameMapper: {
  '^@automattic/calypso-config$': 'calypso/server/config',
  '^@automattic/calypso-config/(.*)$': 'calypso/server/config/$1',
},

The pattern 'calypso/server/config' is a module specifier, not a file path. Jest resolves this through node's module resolution from the rootDir '../../client/server', finding it at client/server/config/index.js.

The initialization sequence occurs in this order:

1. Jest starts the test environment (node or jsdom)
2. setupFiles run (if specified)
3. setupFilesAfterEnv run (this is where globals get assigned)
4. Test files are required and module-level code executes
5. describe blocks register but don't run
6. beforeAll hooks execute
7. Individual tests run
8. afterAll hooks execute

This matters because in test-client, globals like fetch exist at stage 4 (module load), while in test-packages they don't exist at any stage. Code that checks for API availability at module scope will behave differently.

Checking how test-client configuration specifies jsdom - examining test/client/jest.config.js:

const path = require( 'path' );
const base = require( '@automattic/calypso-jest' );

module.exports = {
  ...base,
  rootDir: '../../client',
  cacheDirectory: path.join( __dirname, '../../.cache/jest' ),
  testPathIgnorePatterns: [ '<rootDir>/server/' ],

  moduleNameMapper: {
    '^@automattic/calypso-config$': '<rootDir>/server/config/index.js',
    'react-markdown': '<rootDir>/node_modules/react-markdown/react-markdown.min.js',
  },
  transformIgnorePatterns: [
    'node_modules[\\/\\\\](?!.*\\.(?:gif|jpg|jpeg|png|svg|scss|sass|css)$)',

  testEnvironmentOptions: {
    url: 'https://example.com',

It doesn't explicitly set testEnvironment but inherits from base which sets 'node'. However, the setupFilesAfterEnv and testEnvironmentOptions with url suggest jsdom usage. Looking at the actual test run output, tests have access to document and window, confirming jsdom despite the base saying 'node'. This happens because Jest's default environment is 'jsdom' when not specified at the top level, and the base preset's 'node' setting gets overridden implicitly by the test environment detection.

The complete picture shows three critical differences causing tests to pass individually but fail in suites:

First, calypso-config resolution differs. In test-client and test-server, it redirects to client/server/config/index.js which works in node. In test-packages, it resolves to packages/calypso-config/src/index.ts which throws outside browser contexts unless mocked per-package.

Second, global API availability differs. test-client provides fetch, CSS.supports, TextEncoder, TextDecoder, ReadableStream, and TransformStream. test-packages only provides ResizeObserver and matchMedia. test-server relies on native Node.js APIs. Tests depending on these globals fail in test-packages but pass in test-client.

Third, test environment (node vs jsdom) varies per package in test-packages based on individual overrides, while test-client consistently uses jsdom. Packages without testEnvironment overrides run in node with no document or window, failing tests that expect browser DOM APIs.

The module resolution for internal packages remains consistent across all test commands, the custom resolver in packages/calypso-jest/src/module-resolver.js always prioritizes calypso:src field, loading source TypeScript files instead of built distributions. This consistency means internal dependencies don't contribute to test failures - the resolution doesn't differ based on execution context.
