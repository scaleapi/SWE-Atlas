The behaviors observed in k6's ramping executor are not bugs but intentional design decisions that become clearer when tracing through the actual code paths and running the test suite.

The apparent race condition between handlers does not exist because the two handler strategies operate on fundamentally different data structures and execute in a coordinated sequence. The scheduledVUsHandlerStrategy processes raw execution steps and calls start() and gracefulStop() on VU handles, while maxAllowedVUsHandlerStrategy processes graceful execution steps and calls hardStop(). These handlers are created once and passed to iterateSteps(), which coordinates their execution based on TimeOffset values.

```go
handleNewMaxAllowedVUs = runState.maxAllowedVUsHandlerStrategy()
handleNewScheduledVUs  = runState.scheduledVUsHandlerStrategy()
handledGracefulSteps := runState.iterateSteps(ctx, handleNewMaxAllowedVUs, handleNewScheduledVUs)
```

The iterateSteps function processes both step arrays in time order, using a waiter function that sleeps until the correct TimeOffset before invoking each handler. When both handlers have steps at the same timestamp, raw steps take priority. This time-based sequencing means the handlers never actually execute simultaneously despite running in the context of a goroutine.

The VUs that appear stuck in a state between active and stopped are actually in the graceful completion state, which is expected behavior. When gracefulStop() is called on a VU handle, the VU stops accepting new iterations but completes its current iteration. The VU remains reserved until the graceful period expires or the iteration finishes naturally. This prevents data loss from interrupted iterations.

```go
func (vh *vuHandle) gracefulStop() {
    vh.mutex.Lock()
    defer vh.mutex.Unlock()
    switch vh.state {
    case running:
        vh.changeState(toGracefulStop)
    }
    vh.canStartIter = make(chan struct{})
}
```

The VU buffer leak concern is unfounded because the lifecycle is properly managed through the vuHandle struct. The getVU and returnVU closures in runLoopsIfPossible handle acquisition and return:

```go
getVU := func() (lib.InitializedVU, error) {
    pvu, err := rs.executor.executionState.GetPlannedVU(rs.executor.logger, false)
    if err != nil {
        rs.executor.logger.WithError(err).Error("Cannot get a VU from the buffer")
        cancel()
        return pvu, err
    }
    rs.wg.Add(1)
    atomic.AddInt64(rs.activeVUsCount, 1)
    rs.executor.executionState.ModCurrentlyActiveVUsCount(+1)
    return pvu, err
}
returnVU := func(initVU lib.InitializedVU) {
    rs.executor.executionState.ReturnVU(initVU, false)
    atomic.AddInt64(rs.activeVUsCount, -1)
    rs.wg.Done()
    rs.executor.executionState.ModCurrentlyActiveVUsCount(-1)
}
```

Both gracefulStop and hardStop paths lead to returning the VU to the buffer through these closures, ensuring no leaks occur regardless of how the VU terminates.

The ctrl+c behavior where VUs continue running relates to the graceful shutdown sequence. When the test is interrupted, context cancellation propagates through the system. The iterateSteps function detects the cancelled context and exits, but runRemainingGracefulSteps continues processing remaining graceful steps in its goroutine. VUs receive graceful stop signals and complete their current iterations. After the gracefulStop timeout expires, any remaining VUs receive hard stops. The TestRampingVUsGracefulRampDown test verifies this behavior:


The gracefulRampDown setting directly causes VUs to remain reserved longer than stage targets indicate. When configured with a long gracefulRampDown period like 30 seconds, VUs that ramp down are held in reserved state for that duration to allow iteration completion. The difference between raw steps showing 1 VU and graceful steps showing 6 VUs during this period is the graceful reservation working correctly.

All observed behaviors stem from the graceful handling mechanisms working as designed. The apparent discrepancies between handler counts, the stuck VU states, and the extended ramp-down periods are features that prevent data loss and ensure clean iteration completion rather than concurrency bugs requiring synchronization fixes.
