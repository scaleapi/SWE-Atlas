When kitty processes keyboard input during normal interactive use, the input travels through a layered architecture where the GLFW windowing layer captures raw events, the keys.c module encodes them for the terminal protocol, and the child-monitor.c I/O thread delivers the encoded bytes to the running shell. Observable runtime behavior with debug_keyboard enabled reveals this flow explicitly through console output that traces each transformation.

The GLFW layer receives platform-specific key events and converts them into a uniform GLFWkeyevent structure. The on_key_input function in keys.c serves as the primary entry point where input first becomes visible to kitty's core logic. When debug_keyboard is enabled via the --debug-keyboard command line flag, the function immediately prints diagnostic information showing the raw event details.

```c
void
on_key_input(GLFWkeyevent *ev) {
    Window *w = active_window();
    const int action = ev->action, mods = ev->mods;
    const uint32_t key = ev->key, native_key = ev->native_key;
    const char *text = ev->text ? ev->text : "";

    if (OPT(debug_keyboard)) {
        if (!key && !native_key && text[0]) {
            debug("\x1b[33mon_IME_input\x1b[m: text: %s ", text);
        } else {
            debug("\x1b[33mon_key_input\x1b[m: glfw key: 0x%x native_code: 0x%x action: %s %stext: '%s' state: %d ",
                    key, native_key,
                    (action == GLFW_RELEASE ? "RELEASE" : (action == GLFW_PRESS ? "PRESS" : "REPEAT")),
                    format_mods(mods), text, ev->ime_state);
        }
    }
```

Once encoded, the schedule_write_to_child function queues the bytes for delivery to the shell process. This function operates on a per-window write buffer protected by mutex locks, allowing the main thread to continue processing while the I/O thread handles actual delivery.

```c
bool
schedule_write_to_child(unsigned long id, unsigned int num, ...) {
    va_list ap;
    schedule_write_to_child_generic(id, num, va_start, get_next_arg, va_end);
}
```

The child-monitor.c I/O thread runs independently and manages bidirectional communication with all child processes. The io_loop function uses poll to monitor file descriptors for both read and write readiness. When the write buffer contains data, POLLOUT is set for that child's file descriptor.

```c
for (i = 0; i < self->count; i++) {
    screen = children[i].screen;
    children_fds[EXTRA_FDS + i].events = vt_parser_has_space_for_input(screen->vt_parser) ? POLLIN : 0;
    screen_mutex(lock, write);
    children_fds[EXTRA_FDS + i].events |= (screen->write_buf_used ? POLLOUT  : 0);
    screen_mutex(unlock, write);
}
```

The write_to_child function performs non-blocking writes, handling EAGAIN and EWOULDBLOCK by deferring to the next poll cycle rather than blocking the I/O thread.

The shell process receives the input bytes through its standard input. When the shell echoes the character back, the output flows through the reverse path. The I/O thread reads from the child's file descriptor into the VT parser buffer.

The main thread periodically calls parse_input which invokes do_parse for each active screen. This function runs the parser on accumulated data and checks if input was actually processed. The input_delay option controls how frequently the main loop wakes up to process accumulated input.

The rendering cycle in child-monitor.c checks the render timing against repaint_delay. The render function determines if enough time has passed and whether input has been received that warrants an immediate update.

The architecture separates concerns between three execution contexts: the main thread handles rendering and UI, the I/O thread handles child process communication, and GLFW callbacks handle platform input events. Communication between these contexts uses mutex-protected buffers and wakeup mechanisms.

Summary of input flow through components as observed at runtime:

```
1. GLFW: Platform key event → GLFWkeyevent structure
   - key=0x61 (ord('a')), action=PRESS(1), mods=0
   
2. keys.c: on_key_input → shortcut check → encode_key_for_tty
   - encode_key_for_tty(key=97) returns 'a'
   
3. child-monitor.c: schedule_write_to_child → write buffer
   - Bytes queued: [0x61]
   
4. I/O thread: write_to_child → child stdin
   - write(fd, "a", 1)
   
5. Shell: processes input → echoes to stdout
   - Shell writes 'a' back
   
6. I/O thread: read_bytes → VT parser buffer
   - Reads 'a' from child fd
   
7. Main thread: parse_input → do_parse
   - Parser dispatches: ('draw', 'a')
   
8. screen.c: screen_draw_text → Update cell buffers
   - Line 0: 'a', Cursor x=1
   
9. Main thread: render → send_cell_data_to_gpu → OpenGL draw

10. Display: Frame buffer swap
```

The debug output consistently shows this flow for every keypress, confirming the architectural separation between input handling, child communication, parsing, and rendering stages.
