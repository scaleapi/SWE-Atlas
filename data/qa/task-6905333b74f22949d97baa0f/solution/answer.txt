then, I examined the engine.go file to understand deduplication. the key deduplication logic is in the `notifierWorker` function at line 1189:

```go
// engine.go lines 1216-1221
key := fmt.Sprintf("%s%s%s%+v", result.DetectorType.String(), result.Raw, result.RawV2, result.SourceMetadata)
if val, ok := e.dedupeCache.Get(key); ok && (val != result.DecoderType ||
    result.SourceType == sourcespb.SourceType_SOURCE_TYPE_POSTMAN) {
    continue
}
e.dedupeCache.Add(key, result.DecoderType)
```

the comment in the code explains the intent as follows

```go
// Dedupe results by comparing the detector type, raw result, and source metadata.
// We want to avoid duplicate results with different decoder types, but we also
// want to include duplicate results with the same decoder type.
```

then, I examined the code flow to determine whether deduplication happens before or after overlap detection. looking at the scannerWorker function at line 777, I found the routing logic:

```go
// engine.go lines 795-816
matchingDetectors := e.AhoCorasickCore.FindDetectorMatches(decoded.Chunk.Data)
if len(matchingDetectors) > 1 && !e.verificationOverlap {
    wgVerificationOverlap.Add(1)
    e.verificationOverlapChunksChan <- verificationOverlapChunk{
        chunk:                       *decoded.Chunk,
        detectors:                   matchingDetectors,
        decoder:                     decoded.DecoderType,
        verificationOverlapWgDoneFn: wgVerificationOverlap.Done,
    }
    continue
}

for _, detector := range matchingDetectors {
    decoded.Chunk.Verify = e.shouldVerifyChunk(sourceVerify, detector, e.detectorVerificationOverrides)
    wgDetect.Add(1)
    e.detectableChunksChan <- detectableChunk{
        chunk:    *decoded.Chunk,
        detector: detector,
        decoder:  decoded.DecoderType,
        wgDoneFn: wgDetect.Done,
    }
}
```

so here's what happens when a chunk gets scanned: the scannerWorker first checks how many detectors matched that chunk. if more than one detector matched and the --allow-verification-overlap flag isn't set, the chunk gets sent to verificationOverlapWorker for overlap checking. otherwise, if only one detector matched or if --allow-verification-overlap is enabled, the chunk goes straight to detectorWorker.

the verificationOverlapWorker at line 924 looks for cases where different detectors found similar secrets in the same chunk. when it finds such duplicates between detectors, it sets the errOverlap error on those results before sending them to the results channel. findings that aren't duplicates get passed along to detectableChunksChan for normal processing.

after the detectorWorker at line 1036 processes each chunk, it calls processResult which sends the findings into the results channel. finally, the notifierWorker at line 1189 pulls from that channel and applies deduplication using the dedupeCache.

the key insight here is that overlap detection happens in verificationOverlapWorker, which runs before the results even reach notifierWorker where deduplication happens. so overlap detection comes first, then deduplication. the verificationOverlapWorker uses the `likelyDuplicate` function at line 887 to detect when DIFFERENT detectors find similar secrets.

```go
// engine.go lines 887-922
func likelyDuplicate(ctx context.Context, val chunkSecretKey, dupes map[chunkSecretKey]struct{}) bool {
    const similarityThreshold = 0.9
    // ...
    // If the detector type is the same, we don't need to compare the strings.
    // These are not duplicates, and should be verified.
    if val.detectorKey.Type() == dupeKey.detectorKey.Type() {
        continue
    }
    // ...
}
```

when a likely duplicate is found between different detectors, the `errOverlap` error is set at line 988:

```go
// engine.go lines 39-42
var errOverlap = errors.New(
    "More than one detector has found this result. For your safety, verification has been disabled." +
        "You can override this behavior by using the --allow-verification-overlap flag.",
)

// engine.go line 988
res.SetVerificationError(errOverlap)
```

since my test file only contains a slack webhook URL which is detected by a single detector (SlackWebhook), the condition `len(matchingDetectors) > 1` was never true in the scannerWorker, so chunks went directly to detectorWorker and overlap detection was never triggered.

the reason the same logical secret sometimes produces one result and sometimes produces multiple is due to how the deduplication cache works. the deduplication key is built from DetectorType, Raw, RawV2, and SourceMetadata. when results from different decoders arrive at the notifier worker, the first one gets cached, and subsequent ones with different decoder types are filtered out. however, since worker threads process results concurrently, the order of arrival varies between runs, causing different results to "win" the race to the cache.
