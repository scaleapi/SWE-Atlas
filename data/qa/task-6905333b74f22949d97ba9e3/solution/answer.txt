I started by investigating kitty's remote control system to understand how commands flow from `kitten @ ls` to the running kitty instance and back. 
The system uses two communication mechanisms: Unix sockets for external connections and TTY-based DCS escape sequences for shell integration.

When you specify a `--to` address or have `KITTY_LISTEN_ON` set, it uses Unix domain sockets (or TCP). When neither is available and you're running inside a kitty window, it uses the controlling TTY directly by writing DCS (Device Control String) escape sequences that kitty intercepts.

I examined the client code in `kitty/remote_control.py` and found two I/O classes:

```python
# From kitty/remote_control.py (lines 317-358)
class SocketIO:
    def __init__(self, to: str):
        self.family, self.address = parse_address_spec(to)[:2]

    def __enter__(self) -> None:
        import socket
        self.socket = socket.socket(self.family)
        self.socket.setblocking(True)
        self.socket.connect(self.address)
```

And for TTY-based communication:

```python
# From kitty/remote_control.py (lines 361-366)
class RCIO(TTYIO):
    def simple_recv(self, timeout: float) -> bytes:
        ans: List[bytes] = []
        read_command_response(self.tty_fd, timeout, ans)
        return b''.join(ans)
```

The decision between socket and TTY is made in `do_io()`:

The function automatically appends `-{kitty_pid}` to relative unix socket paths from config files:

```python
# From kitty/main.py (lines 325-343)
def expand_listen_on(listen_on: str, from_config_file: bool) -> str:
    if from_config_file and listen_on == 'none':
        return ''
    listen_on = expandvars(listen_on)
    if '{kitty_pid}' not in listen_on and from_config_file and listen_on.startswith('unix:'):
        listen_on += '-{kitty_pid}'
    listen_on = listen_on.replace('{kitty_pid}', str(os.getpid()))
    if listen_on.startswith('unix:'):
        path = listen_on[len('unix:'):]
        if not path.startswith('@'):
            if path.startswith('~'):
                listen_on = f'unix:{os.path.expanduser(path)}'
            elif not os.path.isabs(path):
                import tempfile
                listen_on = f'unix:{os.path.join(tempfile.gettempdir(), path)}'
```

So for a kitty instance with PID 12345 configured with `listen_on unix:kitty`, the actual socket path would be `/tmp/kitty-12345`.


The protocol uses DCS (Device Control String) escape sequences. I used kitty's actual encoding function to capture the real bytes:

```python
# capture_protocol.py
import sys, json
sys.path.insert(0, '/app')
from kitty.remote_control import encode_send

ls_command = {
    "cmd": "ls",
    "version": [0, 35, 2],
    "no_response": False,
    "payload": {}
}

encoded = encode_send(ls_command)
print(f"Hex: {encoded.hex()}")
print(f"Repr: {encoded!r}")
```

Running this:

```
$ python3 capture_protocol.py
```

Output:
```
Hex: 1b50406b697474792d636d647b22636d64223a20226c73222c202276657273696f6e223a205b302c2033352c20325d2c20226e6f5f726573706f6e7365223a2066616c73652c20227061796c6f6164223a207b7d7d1b5c

Repr: b'\x1bP@kitty-cmd{"cmd": "ls", "version": [0, 35, 2], "no_response": false, "payload": {}}\x1b\\'
```

Breaking this down:
- `\x1bP` (hex 1b50) - ESC P, the DCS introducer
- `@kitty-cmd` - The marker identifying this as a kitty remote control command
- `{"cmd": "ls", ...}` - JSON payload
- `\x1b\\` (hex 1b5c) - ESC \, the ST (String Terminator)

The encoding function from the source:

```python
# From kitty/remote_control.py (lines 308-310)
def encode_send(send: Any) -> bytes:
    es = ('@kitty-cmd' + json.dumps(send)).encode('ascii')
    return b'\x1bP' + es + b'\x1b\\'
```

Responses use the same format:

```python
# From kitty/remote_control.py (lines 52-53)
def encode_response_for_peer(response: Any) -> bytes:
    return b'\x1bP@kitty-cmd' + json.dumps(response).encode('utf-8') + b'\x1b\\'
```


When kitty receives a remote command, the processing flows through these functions:

1. `boss.py: _handle_remote_command()` - Entry point (line 590)
2. `remote_control.py: parse_cmd()` - Extracts and validates JSON (line 56)
3. `remote_control.py: handle_cmd()` - Dispatches to specific command (line 213)
4. `rc/base.py: command_for_name()` - Retrieves command handler (line 222 in handle_cmd)
5. `rc/ls.py: LS.response_from_kitty()` - Generates ls response (line 48)

The dispatch logic:

```python
# From kitty/remote_control.py (lines 213-222)
def handle_cmd(
    boss: BossType, window: Optional[WindowType], cmd: Dict[str, Any], peer_id: int, self_window: Optional[WindowType]

    no_response = cmd.get('no_response', False)
    if tuple(v)[:2] > version[:2]:
        if no_response:
            return None
        return {'ok': False, 'error': 'The kitty client you are using...'}
    c = command_for_name(cmd['cmd'])  # Gets the LS handler
```

For successful commands, no log output is produced - kitty silently processes the command and returns the response. 
For unsuccessful ones, it triggers log_error() calls, logging the errors.

Shell integration enables remote control without explicit socket configuration when running inside a kitty window. This works through environment variables that kitty sets for child processes:

```python
# From kitty/child.py (lines 244-249)

if self.add_listen_on_env_var and boss.listening_on:

```

```python
# From kitty/tabs.py (line 491)
fenv['KITTY_WINDOW_ID'] = str(next_window_id())
```

When `KITTY_LISTEN_ON` is set, kitten uses the socket path. When it's not set but the process is running in a kitty window, kitten falls back to TTY-based communication - it writes the DCS escape sequence directly to the controlling terminal:

```python
# From kitty/remote_control.py (lines 383-385)
io: Union[SocketIO, RCIO] = SocketIO(to) if to else RCIO()
with io:
    io.send(send_data)
```

The `RCIO` class (inheriting from `TTYIO`) opens the controlling TTY via `open_tty()` and writes the escape sequence directly. Kitty's terminal parser intercepts sequences starting with `\x1bP@kitty-cmd` and routes them to the remote control handler rather than displaying them.


This allows the shell to detect whether it's running inside kitty and adjust behavior accordingly (like setting the terminal title or reporting the current directory).


SUMMARY

The remote control system uses:
1. Unix sockets when `--listen-on` is configured (path includes PID like `/tmp/kitty-12345`)
2. TTY escape sequences when running inside a kitty window without explicit socket
3. DCS format: `\x1bP@kitty-cmd{JSON}\x1b\\` for both request and response
4. Command dispatch through `command_for_name()` to handlers in `kitty/rc/`
5. Environment variables `KITTY_LISTEN_ON`, `KITTY_PID`, `KITTY_WINDOW_ID` for shell integration

No source files were modified during this investigation. All the temporary test files were cleaned up.
