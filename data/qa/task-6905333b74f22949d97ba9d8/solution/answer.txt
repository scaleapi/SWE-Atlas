The test_load_classifier_cached failure reveals a potential source of non-determinism. The test uses `pytest.mark.skip` decorator which only works with pytest, not Django's test runner which uses unittest. When running with `python manage.py test`, the skip is ignored and the test executes and fails. This inconsistency causes different results depending on how tests are invoked.

I examined classifier reuse versus retrain behavior by creating an investigation script and running it within the Django test framework. The classifier uses SHA1 hashing to determine whether retraining is necessary.

```python
classifier = DocumentClassifier()
tasks.train_classifier()
```

```
First train_classifier() completed in 0.7522s

Model file mtime: 1766725764.4933274
Second train_classifier() (no data change) completed in 0.0041s
Model file mtime: 1766725764.4933274
Model file unchanged (hash match): True
```

The data hashing mechanism in `classifier.py` calculates a SHA1 hash of all training documents content combined with their correspondent, document type, and tag labels. When the hash matches the stored hash, training is skipped entirely. This takes 0.0041 seconds compared to 0.7522 seconds for actual training. The decision logic is in the train method.

```python
m = hashlib.sha1()
for doc in Document.objects.order_by("pk").exclude(tags__is_inbox_tag=True):
    preprocessed_content = preprocess_content(doc.content)
    m.update(preprocessed_content.encode("utf-8"))
new_data_hash = m.digest()
if self.data_hash and new_data_hash == self.data_hash:
    return False
```

I verified whether shared state leaks between test cases by checking classifier instance isolation. Each call to `DocumentClassifier()` creates a new independent instance. The investigation showed that setting `c1.data_hash` does not affect `c2.data_hash` because instances do not share state.

The `load_classifier()` function also creates fresh instances each time, confirmed by comparing object IDs from successive calls. However, the test isolation relies on `DirectoriesMixin` which creates temporary directories for each test and overrides `MODEL_FILE` via settings. If a test modifies `settings.MODEL_FILE` directly without the mixin, subsequent tests could see stale state.

```python
def setUp(self) -> None:
    self.dirs = setup_directories()
    super(DirectoriesMixin, self).setUp()

def tearDown(self) -> None:
    super(DirectoriesMixin, self).tearDown()
    remove_dirs(self.dirs)
```

The confidence threshold for predictions is not configurable. The classifier uses sklearn `MLPClassifier.predict()` which returns the most likely class without probability thresholds.

```python
def predict_correspondent(self, content):
    if self.correspondent_classifier:
        X = self.data_vectorizer.transform([preprocess_content(content)])
        correspondent_id = self.correspondent_classifier.predict(X)
        if correspondent_id != -1:
            return correspondent_id
        else:
            return None
```

There is no confidence filtering. The prediction is accepted if it returns any valid ID other than -1. This means low-confidence predictions can still be assigned, which could cause non-deterministic behavior if training data is minimal.

For asynchronous task behavior, I examined whether background tasks continue after test assertions complete. The system uses django_q for background task scheduling with redis as the message broker.

```python
Q_CLUSTER = {
    "name": "paperless",
    "catch_up": False,
    "recycle": 1,
    "retry": PAPERLESS_WORKER_RETRY,
    "timeout": PAPERLESS_WORKER_TIMEOUT,

```

The `TASK_WORKERS` variable is computed dynamically based on available CPU cores via the `default_task_workers()` function.

However, during tests, `async_task` calls are typically mocked to prevent actual background execution. The test suite shows this pattern extensively.

```python
@mock.patch("documents.views.async_task")
def test_upload_with_title(self, async_task):
    ...
    async_task.assert_called_once()
```

The document consumption signal handlers execute synchronously, not asynchronously. I verified this by connecting a test handler and measuring timing.

```
Signal sent and completed in 0.1179s
Handler received signal: True
Thread: MainThread
Elapsed time includes handler work (~0.1s): True
This confirms signals execute synchronously
```

The signals are connected in the `apps.py` file ready method.

```python
document_consumption_finished.connect(add_inbox_tags)
document_consumption_finished.connect(set_correspondent)
document_consumption_finished.connect(set_document_type)
document_consumption_finished.connect(set_tags)
document_consumption_finished.connect(set_log_entry)
document_consumption_finished.connect(add_to_index)
```

The `add_to_index` handler uses whoosh `AsyncWriter` for search indexing. The `AsyncWriter` class is a `threading.Thread` subclass that buffers operations and commits in a separate thread if the index is locked.

```python
class AsyncWriter(threading.Thread, IndexWriter):
    """Convenience wrapper for a writer object that might fail due to locking.
    This object will attempt once to obtain the underlying writer, and if it's
    successful, will simply pass method calls on to it. If this object can't
    obtain a writer immediately, it will buffer delete, add, and update method
    calls in memory until you call commit(). At that point, this object will
    start running in a separate thread, trying to obtain the writer over and
    over, and once it obtains it, replay all the buffered method calls."""
```

This is a potential source of non-determinism. If the index is locked during one test, the `AsyncWriter` may buffer operations and commit them in a background thread. This commit could complete after the test assertions but before tearDown completes, or even during the next test. The test isolation through `DirectoriesMixin` creates separate index directories for each test, which should prevent cross-test contamination for the whoosh index specifically.

The runtime execution flow for scheduling these tasks goes through the `apps.py` ready method which connects signal handlers. When `document_consumption_finished.send()` is called in `consumer.py`, Django iterates through all connected handlers synchronously on the main thread. Each handler completes before the next one starts. The only asynchronous operation is the potential whoosh `AsyncWriter` commit if index locking occurs.

The `consume_file()` function in `tasks.py` handles barcode splitting before document consumption. Effect on training data during test runs: When barcode splitting is enabled, a single input PDF can result in multiple Document records. Each split document is consumed separately with its own OCR-extracted content. The classifier's training data includes all split documents individually. For example, if a 6-page PDF with 2 separator barcodes is processed, 3 documents are created instead of 1, potentially with different content distributions. This affects the effective training data during test runs if tests create PDFs with barcodes, as the number of training documents will be larger than the number of input files. The hash-based training cache uses all document content, so the training hash will differ based on whether documents were split or consumed as a single file.

In summary, non-deterministic test failures can arise from several sources. The `pytest.mark.skip` decorator on test_load_classifier_cached does not work with Django's test runner, causing unexpected test failures. The shared in-memory SQLite database with `cache=shared` could allow state leakage if tests run concurrently. The absence of confidence thresholds means low-quality predictions are still assigned. The whoosh `AsyncWriter` can commit in background threads under contention. Test isolation through `DirectoriesMixin` and `override_settings` prevents most shared state issues, but tests that bypass these patterns may see unexpected classifier or index state from previous tests.

The machine learning classifier training pipeline exhibits deterministic behavior when the hash-based caching works correctly, but the timing of when training occurs relative to document insertion matters significantly. If a test creates documents and immediately queries the classifier without calling `train_classifier()` first, the classifier will use whatever model was previously trained, which may not include the newly created documents. This ordering dependency is particularly problematic when tests run in parallel or when test execution order changes between runs, as the effective training dataset can vary depending on which documents exist at the moment training is invoked.

The OCR pipeline introduces additional complexity because external subprocesses like tesseract and ghostscript have their own failure modes that are not controlled by the test framework. Environment differences such as installed library versions, available fonts, and system memory can cause OCR to succeed on one machine but fail on another with the same input file. The test output showed ghostscript failing with a rangecheck error, which typically indicates a configuration or compatibility issue rather than a problem with the test logic itself. These environmental dependencies make it difficult to achieve consistent test results across different development and continuous integration environments.

The interaction between synchronous signal handlers and the whoosh `AsyncWriter` creates a subtle race condition window. While the six signal handlers connected to `document_consumption_finished` all execute synchronously on the main thread, the final handler `add_to_index` delegates to `AsyncWriter` which may spawn a background thread for the actual index commit. If a test completes its assertions and begins teardown while this background thread is still running, the index operations could interfere with directory cleanup or leak into subsequent tests. The `DirectoriesMixin` mitigates this by creating isolated index directories, but any test that shares an index directory or does not properly wait for background operations to complete could experience intermittent failures that are difficult to reproduce and diagnose.
