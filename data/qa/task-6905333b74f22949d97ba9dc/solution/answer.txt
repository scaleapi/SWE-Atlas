I've completed comprehensive testing of paperless-ngx's OCR processing behavior with multi-page PDFs by reproducing the full document consumption flow in the development environment. I did not modify any repository source files. Instead, I set up the necessary services and used the actual API endpoint to upload a test document, then traced the processing through logs and database queries.

First, I ensured the development environment was properly configured. The system required three components to be running:

- Redis (verified with `redis-cli ping`)
- Django development server on port 8000
- Django-Q worker cluster (`python manage.py qcluster`)

I also created a superuser account for API authentication using `python manage.py createsuperuser` and set the password via Django shell. I configured `PAPERLESS_OCR_OUTPUT_TYPE=pdf` in the environment to avoid Ghostscript PDF/A conversion issues during testing.

To verify the upload behavior and processing flow, I created a multi-page PDF with text on three pages using ReportLab. The document contained recognizable text like "Test Document for OCR Processing", "Invoice Number: INV-2025-001", and "Amount: $1,234.56" across the pages.

I then uploaded this PDF through the actual production endpoint:

```
POST /api/documents/post_document/
```

Using curl with token authentication and multipart form data.

The HTTP response I received immediately after upload submission was:

```
HTTP/1.1 200 OK
Content-Type: application/json
X-Api-Version: 2
X-Version: 1.7.0
Content-Length: 4

"OK"
```

This shows the upload returns status `200 OK` with a simple JSON string `"OK"` as the message. The response is immediate and asynchronous - the actual processing happens in the background worker queue.

The log `Consuming test_multipage.pdf` confirms that the consumer task picked up the uploaded file and started a consumption job

The log `Detected mime type: application/pdf` confirms that the input has been identified as a PDF and a PDF-capable parser will be selected.

The log `Parser: RasterisedDocumentParser` confirms that the Paperless selects a parser that can run OCRmyPDF and also tries to extract text.

The log `Parsing test_multipage.pdf...` confirms that the parsing of the uploaded multi-page document  has started.

The log `Extracted text from PDF file ...` confirms that the paperless tried to extract any existing text from the PDF (via a text extractor).

The log `Calling OCRmyPDF with args: {..}` shows the OCRmyPDF invocation parameters and it reveals how the system processes documents. The parameters show parallel processing with 4 jobs, English language, image preprocessing enabled (clean, deskew, rotation), and a sidecar text file for extracted content. The `skip_text` parameter indicates the PDF already has text, so OCR focuses on enhancement rather than initial extraction.

The log `Generating thumbnail for test_multipage.pdf...` indicates that after producing the archive, the thumbnail gets generated to display it on UI.

The log `Document 2025-12-09 test_multipage consumption finished` confirms that the document has gone through the entire pipeline and has been uploaded successfully.
