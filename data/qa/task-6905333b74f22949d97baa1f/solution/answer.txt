Now, for understanding how k6 tracks metrics during a test run, I traced through the codebase starting from the metrics registration system.

The central component is the `Registry` struct in `metrics/registry.go`. This maintains a thread-safe map of all registered metrics:

```go
// From metrics/registry.go (lines 12-17)
type Registry struct {
    metrics map[string]*Metric
    l       sync.RWMutex
    rootTagSet *atlas.Node
}
```

At test startup, k6 registers built-in metrics through `RegisterBuiltinMetrics` in `metrics/builtin.go`. This function creates the core iteration tracking metrics:

```go
// From metrics/builtin.go (lines 78-84)
func RegisterBuiltinMetrics(registry *Registry) *BuiltinMetrics {
    return &BuiltinMetrics{
        VUs:               registry.MustNewMetric(VUsName, Gauge),
        VUsMax:            registry.MustNewMetric(VUsMaxName, Gauge),
        Iterations:        registry.MustNewMetric(IterationsName, Counter),
        IterationDuration: registry.MustNewMetric(IterationDurationName, Trend, Time),
        DroppedIterations: registry.MustNewMetric(DroppedIterationsName, Counter),
        // ... HTTP, WebSocket, gRPC metrics follow
    }
}
```

The `Iterations` metric is registered as a Counter type, meaning it simply accumulates values. The `IterationDuration` is a Trend, which tracks statistical distributions like min, max, average, and percentiles.

---

Each measurement is represented as a `Sample` in `metrics/sample.go`:

```go
// From metrics/sample.go (lines 14-33)
type TimeSeries struct {
    Metric *Metric
    Tags   *TagSet
}

type Sample struct {
    TimeSeries
    Time     time.Time
    Value    float64
    Metadata map[string]string
}
```

A Sample captures the metric being measured, the timestamp, the value, and optional metadata. Samples are grouped into containers for efficient batch processing.

The actual iteration execution and metric emission happen in `js/runner.go`. When a Virtual User (VU) runs an iteration, the `runFn` method handles timing and sample emission:

```go
// From js/runner.go (lines 817-877)
func (u *VU) runFn(ctx context.Context, isDefault bool, fn sobek.Callable, cancel func(), args ...sobek.Value,
) (v sobek.Value, isFullIteration bool, t time.Duration, err error) {
    // ... setup code ...
    
    startTime := time.Now()
    
    err = u.moduleVUImpl.eventLoop.Start(func() (err error) {
        v, err = fn(sobek.Undefined(), args...)  // Run the JS test script
        return err
    })
    
    // ... context handling ...
    endTime := time.Now()
    
    builtinMetrics := u.Runner.preInitState.BuiltinMetrics
    ctm := u.state.Tags.GetCurrentValues()
    
    if isFullIteration && isDefault {
        u.state.Samples <- iterationSamples(startTime, endTime, ctm, builtinMetrics)
    }
    
    return v, isFullIteration, endTime.Sub(startTime), err
}
```

The key observation here is that after the JavaScript function completes, k6 measures the elapsed time and pushes iteration samples to the `Samples` channel. The `iterationSamples` helper creates two samples:

```go
// From js/runner.go (lines 879-902)
func iterationSamples(startTime, endTime time.Time, ctm metrics.TagsAndMeta, 
    builtinMetrics *metrics.BuiltinMetrics) metrics.Samples {
    return metrics.Samples([]metrics.Sample{
        {
            TimeSeries: metrics.TimeSeries{
                Metric: builtinMetrics.IterationDuration,
                Tags:   ctm.Tags,
            },
            Time:  endTime,
            Value: metrics.D(endTime.Sub(startTime)),  // Duration in milliseconds
        },
        {
            TimeSeries: metrics.TimeSeries{
                Metric: builtinMetrics.Iterations,
                Tags:   ctm.Tags,
            },
            Time:  endTime,
            Value: 1,  // Increment counter by 1
        },
    })
}
```

This function produces one sample for the iteration duration (how long it took) and one for the iteration count (always 1, to increment the counter).

Samples flow from VUs through this channel to the output subsystem.

The `output.Manager` in `output/manager.go` coordinates distribution to configured outputs. It spawns a goroutine that reads samples every 50 milliseconds:

```go
// From output/manager.go (lines 12, 56-75)
const sendBatchToOutputsRate = 50 * time.Millisecond

go func() {
    defer wg.Done()
    ticker := time.NewTicker(sendBatchToOutputsRate)
    defer ticker.Stop()
    
    buffer := make([]metrics.SampleContainer, 0, cap(samplesChan))
    for {
        select {
        case sampleContainer, ok := <-samplesChan:
            if !ok {
                sendToOutputs(buffer)
                return
            }
            buffer = append(buffer, sampleContainer)
        case <-ticker.C:
            sendToOutputs(buffer)
            buffer = make([]metrics.SampleContainer, 0, cap(buffer))
        }
    }
}()
```

This batching approach balances throughput with latency - samples are collected for up to 50ms before being dispatched to all configured outputs (JSON, InfluxDB, cloud, etc.).

For internal aggregation, the `MetricsEngine` in `metrics/engine/engine.go` uses an `OutputIngester` (defined in `metrics/engine/ingester.go`) that feeds samples into metric sinks:

```go
// From metrics/engine/ingester.go (lines 62-99)
func (oi *OutputIngester) flushMetrics() {
    sampleContainers := oi.GetBufferedSamples()
    if len(sampleContainers) == 0 {
        return
    }
    
    oi.metricsEngine.MetricsLock.Lock()
    defer oi.metricsEngine.MetricsLock.Unlock()
    
    for _, sampleContainer := range sampleContainers {
        samples := sampleContainer.GetSamples()
        for _, sample := range samples {
            m := sample.Metric
            oi.metricsEngine.markObserved(m)
            m.Sink.Add(sample)  // Add to the metric's sink
            // ... submetric handling ...
        }
    }
}
```

Each metric type has its own Sink implementation in `metrics/sink.go`. For the Iterations counter:

```go
// From metrics/sink.go (lines 47-58)
type CounterSink struct {
    Value float64
    First time.Time
}

func (c *CounterSink) Add(s Sample) {
    c.Value += s.Value  // Simply sum all values
    if c.First.IsZero() {
        c.First = s.Time
    }
}
```

For IterationDuration, the TrendSink maintains a distribution:

```go
// From metrics/sink.go (lines 104-133)
type TrendSink struct {
    values   []float64
    sorted   bool
    count    uint64
    min, max float64
    sum      float64
}

func (t *TrendSink) Add(s Sample) {
    if t.count == 0 {
        t.max, t.min = s.Value, s.Value
    } else {
        if s.Value > t.max { t.max = s.Value }
        if s.Value < t.min { t.min = s.Value }
    }
    t.values = append(t.values, s.Value)
    t.count++
    t.sum += s.Value
}
```

The `ExecutionState` in `lib/execution.go` tracks iteration counts using atomic counters:

```go
// From lib/execution.go (lines 292-309)
func (es *ExecutionState) AddFullIterations(count uint64) uint64 {
    return atomic.AddUint64(es.fullIterationsCount, count)
}

func (es *ExecutionState) AddInterruptedIterations(count uint64) uint64 {
    return atomic.AddUint64(es.interruptedIterationsCount, count)
}
```

The `getIterationRunner` helper in `lib/executor/helpers.go` wraps VU execution and updates these counters:

```go
// From lib/executor/helpers.go (lines 104-141)
func getIterationRunner(executionState *lib.ExecutionState, logger *logrus.Entry,
) func(context.Context, lib.ActiveVU) bool {
    return func(ctx context.Context, vu lib.ActiveVU) bool {
        err := vu.RunOnce()
        
        select {
        case <-ctx.Done():
            executionState.AddInterruptedIterations(1)
            return false
        default:
            if err != nil {
                if handleInterrupt(ctx, err) {
                    executionState.AddInterruptedIterations(1)
                    return false
                }
            }
            executionState.AddFullIterations(1)
            return true
        }
    }
}
```

Complete Metric Flow Summary:

To trace the collection of the `iterations` metric from start to finish:

1. The executor calls `getIterationRunner` which invokes `vu.RunOnce()`
2. `RunOnce()` calls `runFn()` which executes the JavaScript test function
3. After execution, `runFn()` calls `iterationSamples()` creating a Sample with `Value: 1` for the Iterations metric
4. This Sample is pushed to `u.state.Samples` channel
5. The `output.Manager` goroutine reads samples every 50ms and distributes them to outputs
6. The `OutputIngester` feeds samples to the `MetricsEngine`
7. The `CounterSink.Add()` method sums the value into its running total
8. At test end, the aggregated `CounterSink.Value` provides the final iteration count shown in the summary

The test suite confirms this architecture is functioning correctly, with 52 of 82 packages passing their tests.
