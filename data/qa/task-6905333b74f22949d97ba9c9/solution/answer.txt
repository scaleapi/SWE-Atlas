The matching logic for sr/sr1 in Scapy involves a two-stage process that occurs in the _process_packet method of sendrecv.py. When a packet arrives, the system first computes a hash using hashret() to find potential matches in a dictionary of sent packets, then validates with answers() to confirm the match. The issue with tunneled packets stems from how hashret() handles IP-in-IP tunneling.

Looking at the _process_packet function in sendrecv.py reveals the core matching mechanism:

```python
def _process_packet(self, r):
    if r is None:
        return
    ok = False
    h = r.hashret()
    if h in self.hsent:
        hlst = self.hsent[h]
        for i, sentpkt in enumerate(hlst):
            if r.answers(sentpkt):
                self.ans.append(QueryAnswer(sentpkt, r))
```

The hashret() call determines which bucket of sent packets to search, and the answers() call validates whether the received packet truly responds to a sent packet. Both must succeed for a match, but the critical insight is that if two different sent packets produce the same hash, either could incorrectly match an incoming response.

The IP layer's hashret() implementation in scapy/layers/inet.py contains special handling for tunneled packets:

```python
def hashret(self):
    if ((self.proto == socket.IPPROTO_ICMP) and
        (isinstance(self.payload, ICMP)) and
            (self.payload.type in [3, 4, 5, 11, 12])):
        return self.payload.payload.hashret()
    if not conf.checkIPinIP and self.proto in [4, 41]:
        return self.payload.hashret()
    if self.dst == "224.0.0.251":
        return struct.pack("B", self.proto) + self.payload.hashret()
    if conf.checkIPsrc and conf.checkIPaddr:
        return (strxor(inet_pton(socket.AF_INET, self.src),
                       inet_pton(socket.AF_INET, self.dst)) +
                struct.pack("B", self.proto) + self.payload.hashret())
    return struct.pack("B", self.proto) + self.payload.hashret()
```

The condition `if not conf.checkIPinIP and self.proto in [4, 41]` is where the false matching originates. When conf.checkIPinIP is False and the protocol field indicates IP-in-IP (proto 4) or IPv6-in-IP (proto 41), the method returns self.payload.hashret() directly, completely bypassing the outer IP layer's addressing information. This means two packets destined for different tunnel endpoints will produce identical hashes if their inner payloads match.

It's important to note why conf.checkIPsrc and conf.checkIPaddr don't solve this problem. These settings only affect the final branch of hashret() which computes an XOR of source and destination addresses. However, when proto is 4 or 41 and checkIPinIP is False, the code returns early before ever reaching the checkIPsrc/checkIPaddr logic. The tunnel-specific bypass takes precedence, making those other configuration options irrelevant for IP-in-IP packets.

Verifying the conf.checkIPinIP setting shows it defaults to True in config.py:

```
>>> from scapy.all import *
>>> conf.checkIPinIP
True
```

However, if this setting gets changed to False, the tunnel-aware hash computation is disabled. Creating test packets demonstrates the collision. Two requests to different tunnel endpoints with identical inner packets:

```
>>> tunnel1_req = IP(src='1.1.1.1', dst='2.2.2.2', proto=4)/IP(src='10.0.0.1', dst='10.0.0.2')/ICMP(type=8, id=1234, seq=1)
>>> tunnel2_req = IP(src='1.1.1.1', dst='3.3.3.3', proto=4)/IP(src='10.0.0.1', dst='10.0.0.2')/ICMP(type=8, id=1234, seq=1)
>>> tunnel2_resp = IP(src='3.3.3.3', dst='1.1.1.1', proto=4)/IP(src='10.0.0.2', dst='10.0.0.1')/ICMP(type=0, id=1234, seq=1)
```

With conf.checkIPinIP set to False, computing the hashes shows the collision:

```
>>> conf.checkIPinIP = False
>>> tunnel1_req.hashret().hex()
'0000000301d2040100'
>>> tunnel2_req.hashret().hex()
'0000000301d2040100'
>>> tunnel2_resp.hashret().hex()
'0000000301d2040100'
>>> tunnel1_req.hashret() == tunnel2_req.hashret()
True
```

All three packets produce hash 0000000301d2040100 despite tunnel1_req targeting 2.2.2.2 and tunnel2_req targeting 3.3.3.3. The outer IP addresses are completely ignored in the hash computation. Using unique ICMP sequence numbers does not prevent this collision because the hash is computed from the inner payload structure, not sequence-specific fields when the outer layer is bypassed.

The answers() method compounds this issue. Testing the cross-matching behavior confirms Scapy considers the response from tunnel 3.3.3.3 to be a valid answer to the request sent to tunnel 2.2.2.2:

```
>>> tunnel2_resp.answers(tunnel1_req)
1
>>> tunnel2_resp.answers(tunnel2_req)
1
```

Both return 1, meaning Scapy validates either probe as a match. In the sr/sr1 matching loop, if tunnel1_req was stored first in the hsent dictionary and tunnel2_resp arrives, it will incorrectly match tunnel1_req because both the hash lookup succeeds and answers() validates.

This explains the discrepancy with external tools. Tcpdump and Wireshark examine actual packet bytes on the wire and correctly see that responses from 3.3.3.3 belong to requests sent to 3.3.3.3. Scapy's internal matching relies on the computed hash which intentionally discards outer addressing information when checkIPinIP is False.

The IP.answers() method in inet.py also has corresponding logic:

```python
def answers(self, other):
    if not conf.checkIPinIP:
        if self.proto in [4, 41]:
            return self.payload.answers(other)
        if isinstance(other, IP) and other.proto in [4, 41]:
            return self.answers(other.payload)
```

This parallel handling means both hash computation and answer validation skip the outer layer, creating consistent but incorrect behavior. TCP flag validation and Raw payload handling are unrelated to this issue since the problem occurs at the IP layer before those checks are reached.

Setting conf.checkIPinIP to True resolves the false matching. With this setting enabled, the hash computation includes the outer IP addresses:

```
>>> conf.checkIPinIP = True
>>> tunnel1_req.hashret().hex()
'03030303040000000301d2040100'
>>> tunnel2_req.hashret().hex()
'02020202040000000301d2040100'
>>> tunnel1_req.hashret() == tunnel2_req.hashret()
False
```

The hashes now differ because the outer IP XOR (1.1.1.1 XOR 2.2.2.2 vs 1.1.1.1 XOR 3.3.3.3) produces 03030303 and 02020202 respectively at the start of each hash. With unique hashes, responses can only match their correct originating requests.

Verifying the answers() behavior with checkIPinIP enabled confirms proper matching:

```
>>> tunnel2_resp.answers(tunnel1_req)
0
>>> tunnel2_resp.answers(tunnel2_req)
1
```

Now tunnel2_resp correctly rejects tunnel1_req (returns 0) and only matches tunnel2_req (returns 1).

The observable behavior under the problematic configuration manifests as responses being paired with the wrong requests in sr/sr1 results. Since the matching loop iterates through sent packets in storage order and returns the first match, the pairing depends on which request was stored first and which response arrives first, creating apparently random mismatches from the user's perspective despite deterministic underlying behavior.
