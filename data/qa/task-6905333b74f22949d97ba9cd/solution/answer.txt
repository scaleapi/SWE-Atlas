I reproduced the TCP overlap/retransmission scenario end-to-end: wrote pcaps to disk, then read them back offline through TCPSession/HTTP reassembly. I ran two variants-one where the overlap carries identical bytes and one where it carries different bytes-to see exactly what Scapy reconstructs at runtime. Then I traced the behavior back to the specific code path in this checkout.

I used the local Scapy checkout mounted at /app.


Creating the test pcaps with overlapping segments

First, create two pcap files - one where the overlap carries identical bytes (v1) and one where the overlap carries different bytes (v2):

```
cat > /tmp/create_pcaps.py << 'EOF'
import sys
sys.path.insert(0, '/app')
from scapy.all import Ether, IP, TCP, Raw, wrpcap, conf
conf.verb = 0

src_ip, dst_ip = "10.0.0.1", "10.0.0.2"
sport, dport = 54321, 80
http_headers = b"POST / HTTP/1.1\r\nHost: xxx\r\nContent-Length: 16\r\n\r\n"  # 50 bytes

def create_pcap(seg2_body, filename):
    seg1 = http_headers + b"AAAABBBBCCCC"  # 50 + 12 = 62 bytes
    seq = 1000
    packets = [
        Ether()/IP(src=src_ip, dst=dst_ip)/TCP(sport=sport, dport=dport, seq=seq-1, flags='S'),
        Ether()/IP(src=dst_ip, dst=src_ip)/TCP(sport=dport, dport=sport, seq=5000, ack=seq, flags='SA'),
        Ether()/IP(src=src_ip, dst=dst_ip)/TCP(sport=sport, dport=dport, seq=seq, ack=5001, flags='A'),
        Ether()/IP(src=src_ip, dst=dst_ip)/TCP(sport=sport, dport=dport, seq=seq, ack=5001, flags='PA')/Raw(load=seg1),
        Ether()/IP(src=src_ip, dst=dst_ip)/TCP(sport=sport, dport=dport, seq=1058, ack=5001, flags='PA')/Raw(load=seg2_body),
        Ether()/IP(src=src_ip, dst=dst_ip)/TCP(sport=sport, dport=dport, seq=1066, ack=5001, flags='FA'),
    ]
    wrpcap(filename, packets)
    print(f"Created {filename}")

create_pcap(b"CCCCDDDD", "/tmp/v1.pcap")  # identical overlap
create_pcap(b"XXXXYYYY", "/tmp/v2.pcap")  # different overlap
EOF
python3 /tmp/create_pcaps.py
```

Output:

```
Created /tmp/v1.pcap
Created /tmp/v2.pcap
```

The segment layout:
* Seg1 body[0:12] = b"AAAABBBBCCCC" (seq=1000, 62 bytes total with HTTP headers)
* Seg2 body[8:16] = overlap region + new bytes (seq=1058, 8 bytes)
* Overlap region: bytes 8–11 (body[8:12])


Verifying packet processing order

```
cat > /tmp/show_order.py << 'EOF'
import sys
sys.path.insert(0, '/app')
from scapy.all import rdpcap, TCP, Raw, conf
conf.verb = 0

for pcap in ["/tmp/v1.pcap", "/tmp/v2.pcap"]:
    print(f"{pcap}:")
    for pkt in rdpcap(pcap):
        if TCP in pkt and Raw in pkt:
            print(f"  seq={pkt[TCP].seq}, payload_len={len(bytes(pkt[TCP].payload))}")
EOF
python3 /tmp/show_order.py
```

Output:

```
/tmp/v1.pcap:
  seq=1000, payload_len=62
  seq=1058, payload_len=8
/tmp/v2.pcap:
  seq=1000, payload_len=62
  seq=1058, payload_len=8
```

Scapy processes packets in pcap order: Pkt1 (seq=1000) first, then Pkt2 (seq=1058). The "last processed" segment wins in overlapping regions.

Reassembled byte stream for Variant 1 (identical overlap)

In this variant, both segments carry b"CCCC" in the overlap region (body[8:12]).

```
cat > /tmp/reassemble_v1.py << 'EOF'
import sys
sys.path.insert(0, '/app')
from scapy.all import sniff, conf
from scapy.sessions import TCPSession
from scapy.layers.http import HTTP
conf.verb = 0

for pkt in sniff(offline="/tmp/v1.pcap", session=TCPSession):
    if HTTP in pkt:
        body = bytes(pkt[HTTP])
        body = body[body.find(b"\r\n\r\n")+4:]
        print(f"Stream: {body!r}")
        print(f"Length: {len(body)} bytes")
        print(f"Hex: {body.hex()}")
EOF
python3 /tmp/reassemble_v1.py
```

Output:

```
Stream: b'AAAABBBBCCCCDDDD'
Length: 16 bytes
Hex: 41414141424242424343434344444444
```


Reassembled byte stream for Variant 2 (different overlap)

In this variant, Seg1 has b"CCCC" and Seg2 has b"XXXX" in the overlap region (body[8:12]).

```
cat > /tmp/reassemble_v2.py << 'EOF'
import sys
sys.path.insert(0, '/app')
from scapy.all import sniff, conf
from scapy.sessions import TCPSession
from scapy.layers.http import HTTP
conf.verb = 0

for pkt in sniff(offline="/tmp/v2.pcap", session=TCPSession):
    if HTTP in pkt:
        body = bytes(pkt[HTTP])
        body = body[body.find(b"\r\n\r\n")+4:]
        print(f"Stream: {body!r}")
        print(f"Length: {len(body)} bytes")
        print(f"Hex: {body.hex()}")
EOF
python3 /tmp/reassemble_v2.py
```

Output:

```
Stream: b'AAAABBBBXXXXYYYY'
Length: 16 bytes
Hex: 41414141424242425858585859595959
```


Which bytes win in the overlap region

Variant 2 proves the overlap policy:

* bytes 0–7 (body[0:8]): b"AAAABBBB" - only Seg1 wrote here
* bytes 8–11 (body[8:12]): **b"XXXX"** - Seg1 attempted b"CCCC", Seg2 attempted b"XXXX", **Seg2 wins**
* bytes 12–15 (body[12:16]): b"YYYY" - only Seg2 wrote here

The segment processed later overwrites previously written bytes in the overlap range. This is a "last write wins" policy.


Where this overwrite behavior is implemented

The overwrite happens in the TCP reassembly buffer which writes incoming data unconditionally.

**File:** `/app/scapy/sessions.py`  
**Method:** `StringBuffer.append(self, data, seq)`

The critical line is an unconditional slice assignment:

```
memoryview(self.content)[seq:seq + data_len] = data
```

There is no "preserve old bytes" guard. When Seg2's bytes map to the same offsets as Seg1's, the later `append` call overwrites what was previously stored. This matches the runtime observation: body[8:12] ends up as b"XXXX", not b"CCCC".


How the offline run reaches that code path

1. `sniff(offline="/tmp/v2.pcap", session=TCPSession)` reads the pcap
2. `TCPSession` processes packets sequentially and computes relative offsets
3. The session appends segment payload bytes via `StringBuffer.append(...)`
4. The HTTP layer receives the reassembled bytestream and exposes it as `bytes(pkt[HTTP])`

Conclusion

For a 4-byte overlap (body[8:12]), the reconstructed stream contains bytes from the segment processed later. Variant 2 produces b"AAAABBBBXXXXYYYY" - body[8:12] is b"XXXX" even though Seg1 provided b"CCCC" there. This matches `/app/scapy/sessions.py` where `StringBuffer.append` performs an unconditional memoryview slice write that overwrites existing bytes.
