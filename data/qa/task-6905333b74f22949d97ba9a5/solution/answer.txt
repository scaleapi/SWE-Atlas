I started investigating the intermittent validation failures during custom alias creation by first examining the codebase to understand how signed suffixes are verified, then running the server and making targeted API calls to observe the exact responses and log entries.

Looking at the custom alias creation endpoint, I found the v3 API handler in app/api/views/new_custom_alias.py starting at line 115. The function new_custom_alias_v3 handles POST requests to /api/v3/alias/custom/new and is decorated with both a Flask-Limiter rate limit and a parallel execution lock. The signed suffix verification happens at lines 184 through 191:

```python
try:
    alias_suffix = check_suffix_signature(signed_suffix)
    if not alias_suffix:
        LOG.w("Alias creation time expired for %s", user)
        return jsonify(error="Alias creation time is expired, please retry"), 412
except Exception:
    LOG.w("Alias suffix is tampered, user %s", user)
    return jsonify(error="Tampered suffix"), 400
```

The actual signature verification logic lives in app/alias_suffix.py at lines 37 through 42:

```python
def check_suffix_signature(signed_suffix: str) -> Optional[str]:
    try:
        return signer.unsign(signed_suffix, max_age=600).decode()
    except itsdangerous.BadSignature:
        return None
```

This immediately explained why tampered and expired suffixes produce the same behavior. The itsdangerous library raises BadSignature for both cases, whether the signature is invalid or the timestamp exceeds 600 seconds. Since check_suffix_signature catches BadSignature and returns None, the API endpoint sees None for both expired and tampered signatures, resulting in the same 412 response with the expired message. The except Exception block in the endpoint only catches exceptions that are not BadSignature, which would be unusual edge cases.

The quota check is performed by the User.can_create_new_alias method in app/models.py at lines 867 through 884:

```python
def can_create_new_alias(self) -> bool:
    """
    Whether user can create a new alias. User can't create a new alias if
    - has more than 15 aliases in the free plan, *even in the free trial*
    """
    if not self.is_active():
        return False

    if self.disabled:
        return False

    if self.lifetime_or_active_subscription():
        return True
    else:
        return (
            Alias.filter_by(user_id=self.id).count()
            < self.max_alias_for_free_account()
        )
```

This method first checks if the user account is active and not disabled. If the user has a lifetime subscription or active paid subscription, they are allowed unlimited aliases. Otherwise, the method counts existing aliases and compares against the free account maximum. When this check passes successfully, no log entry is emitted. The debug log only appears when the check fails and creation is denied.

For successful alias creation attempts, there is an additional rate-limiting layer inside the Alias.create method at lines 1627 through 1641 in app/models.py:

```python
@classmethod
def create(cls, **kw):
    commit = kw.pop("commit", False)
    flush = kw.pop("flush", False)

    new_alias = cls(**kw)
    user = User.get(new_alias.user_id)
    if user.is_premium():
        limits = config.ALIAS_CREATE_RATE_LIMIT_PAID
    else:
        limits = config.ALIAS_CREATE_RATE_LIMIT_FREE
    # limits is array of (hits,days)
    for limit in limits:
        key = f"alias_create_{limit[1]}d:{user.id}"
        rate_limiter.check_bucket_limit(key, limit[0], limit[1])
```

This bucket-based rate limiter requires Redis to be configured. The limits for free users default to 10 aliases per 900 seconds and 50 per 3600 seconds, while paid users get 50 per 900 seconds and 200 per 3600 seconds, as configured at lines 554 through 559 in app/config.py. When Redis is not available, the check_bucket_limit function in app/rate_limiter.py simply returns early at line 29 without enforcing any limit. When Redis is configured and the limit is exceeded, the function logs at INFO level at line 33:

```
Rate limit hit for alias_create_900d:1 (bucket id 1702500000) -> 11/10
```

And then raises a TooManyRequests exception at line 40.

The execution path for alias creation flows through several layers. Incoming requests first hit the Flask-Limiter decorator enforcing the ALIAS_LIMIT rate, returning 429 if exceeded. The require_api_auth decorator then validates the API key and loads the user context, followed by the parallel_limiter.lock preventing concurrent creation for the same user. Inside the handler, can_create_new_alias checks if the user is active, not disabled, and either has a subscription or hasn't exceeded the free quota, returning 400 if denied. The check_suffix_signature function verifies the signed suffix, returning None for any BadSignature which triggers 412. The verify_prefix_suffix function validates domain access, returning 400 on mismatch. If all pass, Alias.create performs bucket rate limiting when Redis is available, then inserts the record, returning 201 on success or 409 if the alias exists.
