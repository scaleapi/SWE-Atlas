I started by setting up the environment and launching the server:

```
$ cp example.env .env
$ source venv/bin/activate
$ python server.py
```

The console output during startup was:

```
>>> URL: http://localhost:7777
MAX_NB_EMAIL_FREE_PLAN is not set, use 5 as default value
Paddle param not set
WARNING: Use a temp directory for GNUPGHOME /tmp/orszwtalwnnbrhphfulr
Upload files to local dir
>>> init logging <<<
 * Serving Flask app "server" (lazy loading)
 * Environment: production
   WARNING: This is a development server. Do not use it in a production deployment.
   Use a production WSGI server instead.
 * Debug mode: on
```

The first message >>> URL: http://localhost:7777 comes from app/config.py when it loads the URL environment variable. The >>> init logging <<< message from app/log.py confirms logging is ready. Flask then prints its startup banner showing debug mode is on.

The server listens on port 7777 as configured in local_main() via app.run(debug=True, port=7777).

Testing the health endpoint:

```
$ curl -s http://localhost:7777/health
success
```

The health check returns plain text "success" with HTTP 200.

For request handling, incoming requests first enter the Flask app and pass through before_request hooks in server.py. These hooks set g.start_time for timing and handle referral code logic from the session.

For user identity, Flask-Login's user_loader decorator with load_user(alternative_id) resolves session-based identity. For API requests, authorize_request() in app/api/base.py checks the Authentication header for an API key. The authentication context is carried via Flask-Login's current_user proxy for web sessions and g.user for API calls, both accessible throughout request handlers.

Testing an unauthenticated request to root:

```
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost:7777/
302
```

Unauthenticated users get redirected to the login page. The index route checks current_user.is_authenticated and redirects accordingly.

Testing with an invalid API key:

```
$ curl -s -H "Authentication: badkey" http://localhost:7777/api/user_info
{"error": "Wrong api key"}
```

The API returns JSON with "Wrong api key" error from authorize_request() which validates the Authentication header.

Checking for background processes after server startup:

```
$ ps aux | grep -E "python.*(cron|job_runner|event_listener)" | grep -v grep
(no output)
```

No background processes started automatically. The cron.py, job_runner.py, and event_listener.py files exist as separate entrypoints not imported by server.py. Only the main Flask process runs when you start the server, handling HTTP requests with nothing else running alongside it.
