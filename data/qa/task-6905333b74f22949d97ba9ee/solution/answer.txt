The diff kitten begins by walking through directories using the walk function in collect.go, which recursively traverses the file tree while applying pattern-based filters to exclude unwanted files. This filtering happens through the allowed function that checks each filename against glob patterns. The test confirms that patterns like "*~" for backup files and "#*#" for autosave files successfully exclude those entries from comparison.

```go
func walk(base string, patterns []string, names *utils.Set[string], pmap, path_name_map map[string]string) error {
    return filepath.WalkDir(base, func(path string, d fs.DirEntry, err error) error {
        is_allowed := allowed(path, patterns...)
        if !is_allowed {
            if d.IsDir() {
                return fs.SkipDir
            }
            return nil
        }
        // ... store name mappings
    })
}
```

The Collection struct in collect.go serves as the central coordinator that tracks all comparison results. It maintains four separate categories through dedicated data structures: a changes map for modified files, a renames map for files that moved, an adds set for new files, and a removes set for deleted files. Each file gets assigned a type through the type_map that labels it as "diff", "rename", "add", or "removal".

```go
type Collection struct {
    changes, renames, type_map map[string]string
    adds, removes              *utils.Set[string]
    all_paths                  []string
    paths_to_highlight         *utils.Set[string]
    added_count, removed_count int
}
```

The collection process begins by calling collect_files which walks both directories simultaneously and builds sets of filenames. Files appearing in both directories get compared by content, files only in the left become removal candidates, and files only in the right become addition candidates. The clever part is rename detection, which uses MD5 content hashing to identify files that changed names but kept their content.

```
=== Content Hashing for Rename Detection ===
old_name.txt MD5: 55f23e7184df105103d4a3baebb1eaa4
new_name.txt MD5: 55f23e7184df105103d4a3baebb1eaa4
Hashes match (rename detected): true
```

When the hashes match and the file content is identical, the collection registers this as a rename rather than separate add and removal operations. The hash_for_path function computes and caches these hashes.

```go
func hash_for_path(path string) (string, error) {
    return hash_cache.GetOrCreate(path, func(path string) (string, error) {
        ans, err := data_for_path(path)
        if err != nil {
            return "", err
        }
        hash := md5.Sum(utils.UnsafeStringToBytes(ans))
        return utils.UnsafeBytesToString(hash[:]), err
    })
}
```

The caching system underpins the entire diff kitten's efficiency. Seven separate LRU caches with a capacity of 4096 entries each store different types of computed data.

```
=== LRU Cache Configuration ===
Cache size for all caches: 4096 entries
Caches initialized:
  - size_cache: stores file sizes
  - mimetypes_cache: stores detected MIME types
  - data_cache: stores raw file content
  - is_text_cache: stores text detection results
  - lines_cache: stores split lines
  - highlighted_lines_cache: stores syntax highlighted lines
  - hash_cache: stores MD5 hashes for rename detection
```

The LRU implementation in tools/utils/cache.go uses a doubly-linked list to track access order combined with a map for O(1) lookups. When the cache exceeds max_size, it evicts the least recently used entry.

```go
type LRUCache[K comparable, V any] struct {
    data     map[K]V
    lock     sync.RWMutex
    max_size int
    lru      *list.List
}
```

Text versus binary detection happens through the is_path_text function which first checks if the file is an image by its MIME type, then validates whether the content is valid UTF-8. Binary files like images get special handling where they display as changed rather than attempting line-by-line diff.

```
=== File Type Detection ===
text.txt is valid UTF-8: true
image.png is valid UTF-8: false (binary)
Text files get line-by-line diff, binary files show as changed
```

The heart of the diff kitten is its anchored diff algorithm in diff.go. Unlike traditional diff algorithms that run in O(n²) worst case, this implementation uses the Thomas G. Szymanski approach that guarantees O(n log n) time. The key insight is to focus on unique lines that appear exactly once in both files as anchoring points.

```
=== Anchored Diff Algorithm Demonstration ===

Old file: 8 lines
New file: 9 lines

Unique line analysis (appears once in both files):
  Line 1: "package main"
  Line 2: "import \"fmt\""
  Line 3: "func main() {"
  Line 4: "\tfmt.Println(\"hello\")"
  Line 5: "\tfmt.Println(\"world\")"
  Line 6: "}"
Total unique anchors: 6
```

The algorithm starts by counting occurrences of each line in both files using negative numbers as markers. A line appearing once in the old file gets -1, and once in the new file gets -4. Only lines with a total of -5 (appearing exactly once in both) become unique anchors.

```go
m := make(map[string]int)
for _, s := range x {
    if c := m[s]; c > -2 {
        m[s] = c - 1
    }
}
for _, s := range y {
    if c := m[s]; c > -8 {
        m[s] = c - 4
    }
}
// Unique lines have m[s] == -5
```

The tgs function then finds the longest common subsequence of these unique line indexes using binary search, which is where the O(n log n) complexity comes from. The algorithm produces matching pairs that serve as anchors for the diff.

```
TGS matching pairs (line indexes): 8 pairs
  Sentinel start: {0, 0}
  Anchor match: old[0] == new[0]
  Anchor match: old[2] == new[2]
  Anchor match: old[4] == new[4]
  Anchor match: old[5] == new[5]
  Anchor match: old[6] == new[7]
  Anchor match: old[7] == new[8]
  Sentinel end: {8, 9}
```

The Diff function then expands these anchor points by including surrounding identical lines and generates unified diff output with context lines.

```diff
diff old.go new.go
--- old.go
+++ new.go
@@ -4,5 +4,6 @@
 
 func main() {
 	fmt.Println("hello")
+	fmt.Println("beautiful")
 	fmt.Println("world")
 }
```

Syntax highlighting runs in parallel across multiple files through the highlight_all function in highlight.go. It uses an images.Context that spawns goroutines equal to the number of CPU cores by default.

```go
func highlight_all(paths []string) {
    ctx := images.Context{}
    ctx.Parallel(0, len(paths), func(nums <-chan int) {
        for i := range nums {
            path := paths[i]
            raw, err := highlight_file(path)
            if err == nil {
                highlighted_lines_cache.Set(path, text_to_lines(raw))
            }
        }
    })
}
```
Thread safety is guaranteed because the LRUCache implementation uses a sync.RWMutex (defined in tools/utils/cache.go) that protects all read and write operations. When multiple goroutines call highlighted_lines_cache.Set() concurrently with different file paths, the mutex ensures atomic updates to the shared cache without data races.

The Parallel function creates a buffered channel with all file indexes, then spawns worker goroutines that consume from this channel until it closes. Each worker calls highlight_file which uses the Chroma library to tokenize the source code and apply syntax coloring through ANSI escape sequences.

```go
func (self *Context) Parallel(start, stop int, fn func(<-chan int)) {
    procs := runtime.NumCPU()
    c := make(chan int, count)
    for i := start; i < stop; i++ {
        c <- i
    }
    close(c)
    var wg sync.WaitGroup
    for i := 0; i < procs; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            fn(c)
        }()
    }
    wg.Wait()
}
```

The complete flow from input to output works as follows. When you run the diff kitten with two directory arguments, the main function in main.go validates they are both directories, then calls create_collection. This function walks both directories with pattern filtering, compares common files by content, uses hash comparison to detect renames among the added and removed candidates, and categorizes everything into the Collection struct. Files that need highlighting get queued to paths_to_highlight, and highlight_all processes them in parallel. Finally, the UI handler renders the results using the LogicalLine and ScreenLine structures from render.go, displaying changes side by side with syntax coloring and line margins.

The design achieves efficiency through several mechanisms working together. The LRU caches prevent redundant file reads and computations. The anchored diff algorithm provides predictable O(n log n) performance instead of potentially quadratic behavior. Parallel highlighting utilizes all available CPU cores. And the careful separation of text versus binary handling ensures appropriate treatment for each file type.
